1、同步计数：CountDownLatch

同步计数器：让一个线程等待（阻塞/等待wait.await(),其他是线程各自执行完毕自己的任务之后，在唤醒等待线程。

原理：

  1. 基于AQS的 共享模式，用state 表示计数状态，
  2. await 等价获取锁 ，只要state 大于0，线程就每次都获取不到锁
  3. countDown相当于释放state ，释放到0 时 会唤醒队首的线程
  4. 由于是共享模式，所有的的线程都会被依次唤醒

2、同步屏障：CyclicBarrier

CountDownLatch的劣势：无法实现重用。每次使用都需要重新创建对象。

CyclicBarrier的优势：可以实现重用

定义：循环栅栏，用来进行线程协作，等待线程满足某个计数。

原理

  1. 每个线程进入之后都要通过 ReentrantLock 加锁，
  2. 之后会根据栅栏的打破状态来进行判断，
    • 如果没打破，就进入 锁的等待队列，
    • 打破了就唤醒所有的等待队列（醒来还需竞争锁）的线程。
  3. 复用的原理，内部结构定义 一个 parties 属性记录初始值，每次初始化时候，会把 count 的值重新设置。

同步屏障：建立一个屏障，拦截数量自定义，所有线程碰到屏障都会被阻挡等待，等屏障拦截达到一定数量，就会唤醒所有线程

3、限流信号量：Semaphore

原理

  1. 基于AQS的共享模式，用state记录允许的线程数量
  2. 每次进入一个线程，都会判断剩余信号量 state
    • 大于 0，就一直尝试CAS 替换 让state 不断减1
    • 小于0，进入同步队列
  3. 每个线程执行完，信号量加1，顺便唤醒节点

5、Fork/Join框架

思想：把大任务拆分成很多小任务，并行执行任务，再把结果合并起来

  1. Fork/Join 是 JDK 1.7 加入的新的线程池实现，它体现的是一种分治思想，适用于能够进行任务拆分的 cpu 密集型运算
  2. 所谓的任务拆分，是将一个大任务拆分为算法上相同的小任务，直至不能拆分可以直接求解。跟递归相关的一些计算，如归并排序、斐波那契数列、都可以用分治思想进行求解
  3. Fork/Join 在分治的基础上加入了多线程，可以把每个任务的分解和合并交给不同的线程来完成，进一步提升了运算效率
  4. Fork/Join 默认会创建与 cpu 核心数大小相同的线程池
