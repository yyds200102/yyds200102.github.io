1、Java内存模型简介

JMM 即 Java Memory Model，它定义了主存（共享内存）、工作内存（线程私有）抽象概念，底层对应着 CPU 寄存器、缓存、硬件内存、 CPU 指令优化等。

Java内存模型，实际上是不存在的，只是一种约定，一种概念

1.1、并发编程模型的两个问题

  • 线程间的通信：共享内存（Java的方式）和发生消息
  • 线程间的同步：控制不同线程间操作发生顺序的机制，共享内存并发模型里面，同步是显示执行的。程序员必须显式的确定某个方法或某段代码需要在线程直接互斥执行。

同步和异步：线程同步和异步的区别主要在于发送请求后是否需要等待返回，对于线程同步来说，发送请求后需要等待返回，等待返回后才能继续发送下一个请求，而线程异步是不需要等待返回的，在发送一个请求后随时可发送下一个请求。

1.2、Java内存模型结构

共享问题

  • 共享：Java堆中的实例域，静态域，数组都是共享的
  • 非共享：局部变量、方法参数、异常处理器参数不会存在线程之间的共享，所以不会有内存可见性问题，也不会受内存模型的影响

如图所示：

  • Java线程的通信由Java内存模型JMM控制，JMM决定一个线程对共享变量的写入何时对另一个线程可见。抽象的来看，JMM定义了主内存和本地内存之间的关系：线程之间的共享变量存储在主内存，每个线程有自己的本地内存，其中存储了主内存的变量副本。
  • 线程A想要给线程B发送消息如 x = 1，那么线程A必须把 x 先刷新回主存，然后线程B必须从主存更新数据。即必须通过主存来完成，因为只有主存是共享的。

1.3、指令重排

概念：执行程序时，为了提高性能，编译器和处理器常常会对指令做重排序。

  • 编译器优化排序：不改变单线程程序语义
  • 指令级并行的重排序：指令并行技术，将多条数据进行重叠执行。如果不存在数据依赖，处理器就可能进行重排序
  • 内存系统的重排序：使用缓存和读写缓冲区

示例：代码 —-> 1编译器级别—-> 2指令级别 —> 3内存系统级别 —> 最终执行的指令序列

1属于编译器重排序，2、3属于处理器重排序，重排序可能导致多线程出现内存可见性问题。

  • 编译器重排序：JMM会禁止特定类型的重排
  • 处理器重排序：JMM通过插入特定内存屏障禁止某些重排

总结：JMM属于语言级别的内存模型确保在不同的编译器和不同的处理器平台上，通过禁止特定类型的编译器重排序和处理器重排序，达到一致的内存可见性保证

写缓冲区导致的顺序不一致

结果可能： x = y = 0

原因：

  • 处理器层面：线程A、B分别把 a = 1，b = 2 读取并置入写缓存区，接着从主存获取数据，计算 得到 x = 0 ， y = 0，最后把x 、y 刷新回主存
  • 内存层面：认为先执行了a = 1 和 b = 2 ，结果应该 x = 2， y = 1

所以造成了执行顺序不一致，因此现代处理器通常都禁止对 写 - 读操作进行重排

内存屏障

为了保证内存可见性，Java编译器通过插入内存屏障禁止重排序。

1.4、happens-before规则

概念：JDK5开始，Java使用新的JSR-133内存模型，使用happens-before概念来阐述操作之间的内存可见性，如果一个操作的执行结果对另一个操作可见，那么两个操作必须有happens-before关系。

目的：确保了内存可见性

规则：

  • 程序顺序规则：一个线程的每个操作，happens-before与该线程的任意后续操作
  • 监视器锁规则：对一个锁的解锁，happens-before于随后对这个锁的加锁
  • volatile变量规则：对于一个volatile域的写，happens-before于任意后续对这个域的读
  • 传递性：A happens-before B，且B happens-before C，则A happens-before C
  • start()规则：如果线程A执行启动线程B（调用start方法），那么线程A的启动操作happens-before于线程B的任意操作
  • join()规则：如果线程A执行ThreadB.join()方法并成功返回，那么线程B的任意操作happens-before于线程A从Thread.join()操作成功返回。

总结：A happens-before B，JMM并不要求A一定要在B之前执行，但是JMM会保证实际的执行结果 一定与 A、B顺序执行的结果一致。

2、重排序

概念：重排序是为了优化程序性能而对指令序列进行重新排序的手段。

2.1、数据依赖性

概念：两个数据访问一个变量，其中一个为写操作，此时这个两个数据就具有数据依赖性。

名称 代码 说明
写-读 a = 1; b = a ; 写入一个变量，再读该变量
写-写 a = 1; a = 2 ; 写入一个变量，再写该变量
读-写 a = b; b= 1; 读取一个变量，再写该变量

很显然，只要发生重排序，程序的执行结果就会发生改变。

注意：编译器和处理器会避免破坏数据依赖性的重排序。

2.2、as-if-serial

概念：无论如何排序，单线程下的执行结果都不能改变

意义：编译器，处理器都遵循 as-if-serial 规则，这使得我们在单线程环境下不需要考虑重排序，不需要考虑内存可见性问题。

在不改变程序执行结果的前提，提高并发度

2.3、对多线程的影响

多线程情况下，重排序可以影响结果：假设希望依次执行write、reader

  • 1、2重排序：若执行顺序 2 —> 3 —->4 —->1 就与期待的结果相违背
  • 3、4重排序：由于3和4存在依赖关系，所以步骤4不会真正的执行，而是把计算结果提前计算并且记录下来，当步骤3为真时，才把结果写入变量i

3、顺序一致性

3.1、数据竞争与顺序一致性

JMM保证如果程序正常同步了（同步原语的正确使用），那么程序将具有顺序一致性，即程序的执行结果与在顺序一致性内存模型中的执行结果相同。

3.2、顺序一致性内存模型

顺序一致性内存模型是一个被理想化的理论（真实不存在）参考模型。提供了极强的内存可见性保证。

特点：

  • 一个线程中的所有操作按照程序的顺序来执行
  • 每个操作都是原子执行并且对所有线程可见。

结构：拥有单一的全局内存，同一时刻只有一个线程可以操作内存，多线程并发执行时，开关装置能够让多个线程串行执行，需要注意的是，这只存在理论中，JMM不可能实现这一点。

3.3、同步程序的顺序一致性效果

同步程序也存在指令重排：

  • 顺序一致性内存模型：所有操作完全按照顺序执行
  • JMM：临界区的代码允许指令重排，但是范围只能在临界区之内。（临界区：指线程A获得锁到A释放锁的这一段区域），这样即提高了效率，有没有改变程序的执行结果。

总结：JMM的宗旨就是在不改变（正确同步）程序执行结果的前提下，尽可能的为编译器和处理器的优化打开方便之门，因为临介区本身就是单线程执行的，而JMM规范已经保证了单线程执行不会有可见性问题。

3.4、未同步程序的顺序一致性效果

​ 未同步或者未正确同步的多线程程序，JMM只提供最小安全性，JMM保证线程读取到的值不会无中生有，只会是默认值或者之前某个线程写入的值。为了实现最小安全性，JVM在分配内存时，会先对内存空间清零，然后分配内存。

​ 至于顺序一致性，JMM不能保证

4、volatile的内存语义

4.1、volatile的特性

一个volatile变量的操作，效果等于使用锁来同步这个对变量的操作，可以理解为volatile就是给一个变量加锁，所以具有以下特性。

  • 可见性：对于一个volatile变量的读，总是可以看到其他线程对该变量的写
  • 原子性：对于volatile变量的单次操作读、写具有原子性。（volatile++是多次操作，所以不具有）

4.2、volatile写-读的内存语义

写：写一个volatile的变量，JMM会把该线程的本地内存的共享变量值强制刷新回主存。

读：读一个volatile的变量，JMM会把该线程的本地内存的共享变量置为无效，从主存获取值。

可见性：写一个volatile的变量 总是 happens-before 与 读一个volatile的变量

可见性保证：为了实现volatile的内存语义，JMM会对编译器和处理器重排序做出限制，具体在下面解释。

4.3、volatile写-读建立的happens-before规则

概念：对于一个volatile域的写，happens-before于任意后续对这个域的读；也就是说，只要程序上我们volatile变量的写在读之前，那么无论底层怎么优化，JMM都会保证实际结果与我们代码一致。

从JSR-133也就是JDK1.5开始，volatile变量的读-写可以实现线程间的通信。

从内存语义来说，volatile的写-读与锁的释放-获取有一样的内存效果，即volatile的写与锁的释放有一样的内存语义，volatile的读 与锁的获取有一样的内存语义

4.4、volatile内存语义的实现

JMM为了实现volatile的内存语义，会限制编译器和处理器的重排序

下面第一次，右第二次 普通读、写 volatile读 volatile写
普通读、写 NO
volatile读 NO NO NO
volatile写 NO NO
  • volatile读操作之后，无论是什么操作，都禁止重排序
  • volatile写操作之后，只允许和普通读写操作进行重排序。
  • 不管前面操作是什么，只要接下来有volatile写，就不能重排序

做法：编译器生成字节码的时候，会在指令序列插入内存屏障来静止特定类型的处理器重排序，JMM采用了保守的策略，保证效果在任意平台有效。

  • 每个volatile写操作的前面插入一个StoreStore字段
  • 每个volatile写操作的后面插入一个StoreLoad字段
  • 每个volatile读操作的后面插入一个LoadLoad字段
  • 每个volatile读操作的后面插入一个LoadStore字段

5、锁的内存语义

概述：final变量的赋值操作都必须在定义时或者构造器中进行初始化赋值，并发现 final 变量的赋值也会通过 putfield 指令来完成，同样在这条指令之后也会加入写屏障，保证在其它线程读到它的值时不会出现为 0 的情况。

5.1、锁的释放-获取建立的happens-before规则

概念：对一个锁的解锁，happens-before于随后对这个锁的加锁。

根据happens-before 规则，2应该happens-before 在5之前，而A释放之后，B获取了同一个锁，所以线程A释放锁之前所有可见的共享变量，在线程B获取之后，立刻对B可见，符合happens-before 规则。

5.2、锁的释放和获取的内存语义

释放锁：JMM会把该线程对应的本地内存中的共享变量刷新到主内存中。

获取锁：JMM会把该线程对应的本地内存置为无效，必须从主存重新读取代码。

6、final域的内存语义

6.1、final域的重排规则

  1. 初始化final域之后之后才能被引用 1 —–> 2
  2. 先读取引用，才能读取数据 3 —–> 4

写final域的重排规则

规则：JMM会下达指令，让编译器会在final域的写之后，构造函数 return 之前，插入一个StoreStore屏障，禁止处理器把final域的写重排序到构造函数之外。即 一定把 final 修饰的对象限定到构造函数之内执行。

没有这个规则：obj就可能是不完整的对象。

读final域的重排规则

规则：编译器会在读final域前面插入一个LoadLoad屏障，防止值未初始化就被读取。

如果没有这个规则：object读取为null

6.2、final域为引用类型

新增规则：在构造函数内对一个final引用类型的初始化，与对该引用类型的赋值之间，不能重排序。即上述 步骤1和步骤2之间不能重排，同时步骤2和步骤3也不能重排。

简要理解：final引用类型的初始化，会限制必须先初始化在赋值。

即：必须先初始化在赋值，必须赋值之后才能获取到对象。

如果没有这个规则：

  • 可能先执行3、在执行2，导致最终值与期望的不符
  • 也可能先执行2再执行1，导致未初始化就赋值

总结

JMM会保证final修饰的变量 被赋值，引用了对象且引用的对象被赋值之后 才会让让该变量能被外部所调用。

6.3、final引用对象溢出问题

前面的规则：确保了在final域的引用变量arr被其他线程可见之前，arr已经成功初始化并且赋值了。

但需要一个前提：final引用对象不能在构造函数内溢出，即构造函数返回前，被构造的对象不能被其他线程所看见，因为final还没有初始化。

JSR-133对final语义的增强

​ 旧的内存模型中，线程可能看到final域的值是变化的，JSR-133中确保了：只要对象是正常构建的（被构造对象的引用在构造函数中没有溢出），不需要使用同步机制，任意线程只能看到构造函数构造完之后的final域的变量。

6.4、扩展：final优化读取速度

编译成字节码的时候，发现 i 直接就是10

原因：JVM把 i 复制一份到 方法栈上，如果数值比较大，也会复制到引用该常量的类的常量池中

7、happens-before详解

7.1、JMM设计

JMM把happens-before要求禁止的重排序分为了下面两类

  • 会改变程序执行结果的重排序，（JMM禁止）
  • 不会改变程序执行结果的重排序（JMM允许）

原则：只要不改变程序运行结果，编译器和处理器怎么优化都行。

7.2、happens-before的定义

  • 如果一个操作happens-before另一个操作，那么第一个操作的执行结果将对第二个操作可见，而且第一个操作的执行顺序在第二个操作之前。（JMM对程序员的承诺）
  • 两个操作之间存在happens-before关系，但是并不一定要完全遵循happens-before关系指定的顺序来执行，只要不改变运行结果，JMM就允许。（JMM对编译器和处理器重排序的约束）

as-if-serial语义保证单线程的执行顺序不会被改变，happens-before关系保证正确同步的多线程程序的执行结果不改变

as-if-serial语义制造了表象：程序按照顺序执行的

happens-before关系制造了表象：正确同步的多线程程序按照happens-before指定顺序执行

7.2、happens-before的规则

  1. 程序顺序规则：一个线程的每个操作，happens-before与该线程的任意后续操作
  2. 监视器锁规则：对一个锁的解锁，happens-before于随后这个锁的加锁
  3. volatile变量规则：对于一个volatile域的写，happens-before于任意后续对这个域的读
  4. 传递性：A happens-before B，且B happens-before C，则A happens-before C
  5. start()规则：如果线程A执行启动线程B（调用start方法），那么线程A的启动操作happens-before于线程B的任意操作
  6. join()规则：如果线程A执行ThreadB.join()方法并成功返回，那么线程B的任意操作happens-before于线程A从Thread.join()操作成功返回。

8、懒汉式单例模式

8.1、双重检查锁定：double-checked locking

有缺陷的懒汉式单例模式：

在单线程中上述的指令重排是被允许的，因为在单线程中不会改变结果，符合as-if-serial规则

但是多线程情况下就会出现问题。

基于volatile的解决方案

原本 可能的 1—>3—>2，由于声明为volatile，所以指令重排被禁止了，

8.2、基于类初始化的解决方案

  • 类的静态属性只会在第一次加载类的时候初始化，可以避免提前被初始化浪费资源
  • 类的初始化期间，JVM会去获取一个锁（每一个类和接口有一个唯一的初始化锁LC），禁止多个同步线程对一个类的初始化，这里等于是JVM帮助我们保证了线程的安全性。

本质：允许指令重排，但是不允许其他线程“看到” 未初始化完成的对象，即其他线程要么获取不到对象，要么只能获取完整的对象。

类初始化分析

类被初始化的几种情况：

  • T是一个类，一个T类型的实例被创建
  • T是一个类，且T中声明的静态方法被调用
  • T声明的静态字段被赋值
  • T声明的静态字段被使用，且这个字段不是常量字段
  • T是一个顶级类，且一个断言语言嵌套在T内部被执行。

过程：

  1. 线程A获取到了初始化锁，看到state == noInitialization 表明线程还未初始化，线程A就开始初始化，并设置 state = initializing ，
  2. 线程B获取到了初始化锁，但是 state = initializing ，于是就释放锁，进入初始化锁的condition开始等待。
  3. 线程A初始化完成，设置 state = initialized ，唤醒所有在condition等待的线程，并且释放锁，到此A的初始化过程完成。
  4. B获取到初始化锁，发现 state = initialized ，就释放初始化锁，到此B的初始化过程完成。

根据JMM的规范：A释放初始化锁在B的获取之前，二者是同一把锁。所以存在happens-before规则：线程A的初始化的写入操作对线程B可见

注意：condition和state只要求JVM实现类似的功能。

对比

  • 双重检查：不仅是静态字段，还可以是实例字段
  • 类初始化：只能是静态字段

9、总结

9.1、处理器的内存模型

顺序一致性内存模型是一个理论概念，JMM和处理器内存模型设计时以此为参照，但会适当的放松条件，因为完全参照会导致效率很低。

  • 放松 写 - 读操作的顺序，产生了TSO内存模型（Total Store Ordering）
  • 继续放松 写 -写顺序，产生了PSO内存模型（Partial Store Order）
  • 继续放松读-写 和 读 - 读 顺序，产生了RMO内存模型（Relaxed Memory Order）

以上都是建立在没有数据依赖的情况下进行的，保证as-if-serial规则

可以注意到：都放松了写-读操作顺序，因为都用了写缓存区。这也导致当前处理器可以比其他处理器先看到临时保存在自己缓存区里的写。

  • 由于JMM的约束性比处理器大，所以JMM会使用内存屏障来禁止某些重排序。
  • JMM屏蔽了不同处理器的差异，让程序的运行结果在不同的处理器平台一致

越追求性能的内存模型，限制会越少。

9.2、JMM内存可见性保证

  • 单线程程序：没有内存可见性问题
  • 正确同步的多线程程序：通过限制处理器和编译器的重排序保证内存可见性。
  • 未正确同步的多线程程序：JMM提供最小安全性保障，线程不会读到无中生有的值，对象初始化了，才能被线程使用，但是不能保证值是否是正确的（如线程A初始化一半时就被线程B读取）

9.3、JSR-133

  • 对volatile的增强：限制和普通变量的重排序
  • 对final的增强：保证final引用没有溢出的情况下，具有初始化安全性。
