Java并发(一):简介

发表于2021-07-04|更新于2021-07-04|Java
|字数总计:3.8k|阅读时长:12分钟|阅读量:

1、上下文切换

多线程原理：即使是单核处理器也支持多线程执行代码，可以通过CPU给每个线程分配CPU时间片来实现这个机制，由于时间片时间很短，所以感觉是多个线程同时执行的。

上下文切换：CPU通过时间片分配执行多线程，当前任务执行一个时间片后湖切换到下一个任务，但是切换前会保存上一个任务的状态，下一次继续执行当前任务时会从保存的状态处继续执行任务。任务从当前任务保存到下一个任务加载的过程就是上下文的切换。

注意：单核CPU执行多线程任务一定会触发线程上下文切换，多核CPU执行多线程任务时，若CPU能满足所有线程的需要，就会并行执行任务而没有线程上下文切换(但是一般情况不可能)，当CPU不够所有线程需要时，也会发生线程上下文切换。

1.1、多线程一定更快么

不一定，因为上下文切换需要消耗时间

使用工具测出，1秒中发生1000多次上下文切换。

1.2、减少上下文切换

  • 无锁并发编程：多线程竞争时，会引起上下文切换，可以使用一些办法避免使用锁，如不同的线程 按照Hash算法处理不同的区域，一个数据只能有一个线程处理。
  • CAS算法：
  • 尽量少的创建线程
  • 协程，在单线程里面实现多任务的调度

1.3、死锁

死锁演示：

  1. 线程t1 执行 占有对象A，线程t2 执行 占有对象B
  2. 线程t1 接着执行需要对象B ，但是无法获取，于是线程 t1 等待
  3. 线程t2 接着执行需要对象A ，但是无法获取，于是线程 t2 等待
  4. 线程 t1 和 t2 将会无限的等待下去。

实际开发：实际开发中不会写出这么愚蠢的代码，但是有可能出现以下情形

  • t1 拿到资源A（锁）之后，因为异常情况而没有释放
  • t1 拿到资源A （锁）之后，释放的时候抛出异常，没有释放成功

避免死锁：

  • 避免一个线程获取多个锁
  • 避免一个线程在锁内占用多个资源
  • 尝试使用定时锁，使用lock.trylock(timeout) 来替代内部锁机制
  • 对于数据库锁，加锁和释放必须在一个数据库连接中，否则会出现解锁失效的问题。

1.4、资源限制的挑战

资源限制：程序的执行速度受限于软件或者硬件资源，如网站带宽2M/s，即使再多的线程，速度也不会变快。

问题：并发编程快的原因：代码串行变成并发执行，但是如果将某段串行的代码并发执行，那么由于资源受限，反而会执行的更慢，因为增加了上下文切换和资源调度的时间，

解决：考虑集群并行执行程序，单机的资源优先，就让程序在多机上运行，如不同的任务通过算法选择一台机器执行程序

资源受限的情况下并发：根据不同的资源限制调整程序的并发度，比如下载文件程序依赖两个资源：带宽和硬盘读写。有数据库操作时，涉及数据库连接数，如果SQL执行的快，而线程的数量比数据库连接数大很多时，某些线程会堵塞，等待数据库连接。

2、volatile原理

概念：用来修饰成员变量和静态成员变量（放在主存中的变量），保证所有线程看到某个变量的值都是一致的，保证了共享变量的”可见性”，同时不会引起线程上下文的切换。

实现原理：每次操作完数据，JVM会发送一条Lock前缀的指令

  1. Lock前缀的指令会引起处理器缓存写回到主存
  2. 一个处理器缓存写回到主存，会导致其他处理器的缓存失效。

效果：olatile的修饰的变量具有可见性

  1. 写一个volatile的变量，JMM会把该线程的本地内存的共享变量值强制刷新回主存。
  2. 读一个volatile的变量，JMM会把该线程的本地内存的共享变量置为无效，从主存获取值。

概念2： volatile 关键字的效果增强：禁止某些情况下的重排序

原理：加入内存屏障

3、synchronized

过去都称为重量级锁，但是在JDK1.6之后优化了（引入轻量级锁和偏向锁），有些情况下并不是那么重了。

锁的对象：当一个线程访问同步代码块时必须获得锁，退出或者抛出异常必须释放锁。

  • 普通同步方法：锁的是实例对象
  • 静态同步方法：锁的是当前类的Class对象
  • 同步方法块：锁的是synchorized括号里配置的对象。

同步代码块实现原理：基于进入和退出Monitor对象

  • monitorenter指令：在编译后插入到同步代码块的开始位置
  • monitorexit指令：插入到同步代码块的结束处和异常处
  • JVM保证每个monitorenter都必须有一个monitorexit与之匹配。
  • 任何一个对象都有一个monitor与之关联，一个线程执行了monitorenter指令后，就会尝试获取monitor，如果成功获取到，那么当前对象就处于锁定状态，执行了monitorexit就会释放monitor。

同步方法实现原理：使用ACC_SYNCHRONIZED 标识，该标识指明了该⽅法是⼀个同步⽅法。

二者本质都是对对象监视器 monitor 的获取

Java对象头：synchorized的锁信息是存在Java对象头里，数组类型3字宽，其他2字宽。Mark Word存储锁信息，来标记是哪个线程获得了该锁，下面是Mark Word存的数据。

  • 偏向锁：存放线程ID
  • 轻量级锁：指向栈中锁记录的指针
  • 重量级锁：指向互斥量的指针

锁的优化：Java1.6对synchorized进行了优化

锁的升级：JavaSE1.6中，锁有四种状态，级别从低到高依次为 无锁—> 偏向锁 —> 轻量级锁 —> 重量级锁，锁可以升级但是不能降级。

偏向锁

缘由：HotSpot作者发现，大多数情况下，锁很少存在竞争情况，大多数情况下，锁都是一个线程多次释放和获得，所以为了让线程获得锁的代价更低，引入了偏向锁

方案：

  • 无锁情况获取：直接获得锁，并在对象头和栈帧的锁记录锁偏向的线程ID
  • 有锁情况获取：任意线程获取锁时，会比较是否为偏向锁是否有本线程的ID，如果有表示已经占有了锁；如果没有，则看当前是否还是偏向锁，若不是则使用CAS竞争锁；若是则会尝试让偏向锁指向自己的线程ID。
  • 锁的释放：当有其他线程竞争锁时，系统会暂停持有锁的线程的执行，然后判断该线程是否存活，若不存活，则把对象头设置为无锁状态；若存活，则该线程的栈会执行去遍历锁记录，选择是否把锁偏向其他的线程、恢复到无锁、或者进行锁的升级。

关闭锁：

总结：偏向锁就是线程A持有锁，不正常释放，直到线程B来获取锁时，才让线程A释放锁

  • 加锁：让对象头记录本线程的ID
  • 释放锁：把持有锁线程ID替换成新的线程的ID

轻量级锁

  • 加锁：线程执行同步代码块之前，JVM会在当前线程的栈帧中创建用于存储锁记录的空间。并把对象头中的Mark Word复制到锁记录中（称为Diasplaced Mark Word）。然后尝试把对象头的Mark Word替换成指向锁记录的指针，如果成功，当前线程获得锁，如果失败，表示其他线程竞争锁，当前线程尝试使用自旋获取锁。
  • 解锁：使用原子的CAS操作把对象头的Diasplaced Mark Word，替换回原对象头，如果成功，表示没有竞争发生。如果失败，就会膨胀成重量级锁。

升级为重量级锁后，就可以避免无用的自旋浪费CPU资源。重量级锁情况下：加锁后，所有线程尝试获取锁时都会被堵塞，当锁释放后，线程会唤醒堵塞的线程

总结：使用CAS获取锁，如果释放时竞争激烈，就把锁升级成重量级锁。

  • 加锁：先把对象头的信息备份，然后让对象头的信息更换为当前线程的信息
  • 解锁：把备份还原

对比

锁 优点 缺点 场景
偏向锁 加锁解锁不需额外消耗 若竞争激烈，会带来额外的锁撤销过程的消耗 单线程访问同步代码
轻量级锁 线程不堵塞，响应快，减少线程上下文切换 若始终获取不到CPU，会一直自旋，浪费资源 追求响应时间，同步代码块执行速度快
重量级锁 不自旋，不消耗CPU 线程堵塞，响应时间满， 追求吞吐量，同步代码块执行时间长

总结：竞争越激烈，用越重的锁。

4、原子操作

术语定义：

  • 缓存行（Cache line）：缓存的最小操作单位
  • 比较与交换（CAS）：每次操作前都把新获得的值与旧值比较，确保中间没有被其他线程修改过
  • CPU流水线：CPU中由56个不同功能的电路单元组成一条指令处理流水线，然后将一条X86指令分解成56步后，再由这些电路单元分别执行，这样实现在一个CPU时钟周期内完成一条指令。
  • 内存顺序冲突：多个CPU同时修改同一个缓存行的不同部分而引起其中一个CPU的操作无效。当出现该冲突时，CPU必须情况流水线。

5、CAS操作引起的三大问题

ABA问题

描述：CAS判断操作，是通过值有没有变化，但是可能出现 A—> B — >A 的情况，CAS认为没有变化，但是实际上已经发生了改变

解决：使用JDK1.5开始提供的AtomicStampedReference原子引用类型，把数据包装到原子引用类型里面，通过原子引用类型操作变量。

循环开销大

描述：CAS可能会一直自旋，可以CPU带来很大的执行开销。

只能保证一个共享变量的原子操作

描述：对一个共享变量操作时，可以进行CAS，但是对于多个变量进行处理就不行。

解决：

  • 使用锁
  • 使用JDK1.5中的AtomicReference引用类型类，把多个变量封装到一个引用类型里面

6、JMM三大特性总结

  • 原子性：保证指令不会受到线程上下文切换的影响
  • 可见性 ：保证指令不会受 cpu 缓存的影响
  • 有序性 ：保证指令不会受 cpu 指令并行优化的影响

synchronized ：保证原子性和可见性

volatile：保证了可见性，指令重排

二者都能保证有序性：

  • synchronized ：通过同步机制，保证代码块之间的有序进行
  • volatile：通过禁止部分指令重排，保证代码执行顺序

7、CPU⾼速缓存

目的：解决CPU处理速度和内存处理速度不对等的问题。

扩展：内存缓存的是硬盘数据⽤于解决硬盘访问速度过慢的问题。

工作流程：先复制⼀份数据到 CPU Cache 中，当 CPU 需要用到的时候就可以直接从 CPU Cache 中读取数据，当运算完成后，再将运算得到的数据写回 Main Memory 中。但是，这样存在 内存缓存不⼀致性的问题！比如执行⼀个 i++操作的话，如果两个线程同时执行的话，假设两个线程从 CPUCache 中读取的 i=1，两个线程做了 1++运算完之后再写回 Main Memory 之后 i=2，⽽正确结果应该是 i=3。

CPU 为了解决内存缓存不⼀致性问题可以通过制定缓存⼀致协议或者其他⼿段来解决。
