1、锁介绍

1.1、各种锁的概念

1.2、ReentrantLock和Synchronize区别

  1. Synchronized是内置关键字，Lock是类

  2. Synchronized无法判断获取锁的状态，Lock可以

  3. Synchronized可以自动释放锁(一个线程执行完)，Lock必须手动释放(死锁的可能)

  4. Synchronized是可重入锁，不能中断，非公平；Lock可重入，可设置公平度

  5. Synchronized适合锁少的代码同步，Lock适合锁大量的同步代码

2、队列同步器AQS

队列同步器AbstractQueuedSynchronizer：用于构建锁或其他同步组件的基础框架，

  1. 使用了一个int成员变量（state）表示同步状态，状态变量用来控制加锁解锁，队列用来放置等待的线程。
  2. 内置的 FIFO 队列来完成资源获取线程的排队工作。
  3. 条件变量Condition来实现等待、唤醒机制，支持多个条件变量，

队列同步器是基于模板方法模式的：使用者可以通过重写指定的方法，同步器调用这些重写的方法实现不同的效果。

同步器的主要实现方式是继承，子类通过继承同步器实现抽象方法管理同步状态。同步器提供的三个方法可以保证状态的改变是线程安全的。同步器是实现锁（其他任意同步组件）的关键，在锁的实现中聚合同步器，从而实现锁的语义。

  • 锁是面向使用者的，定义了使用者和锁交互的接口，隐藏了细节
  • 同步器面向的是锁的实现者，简化了锁的实现方式，屏蔽了同步状态管理、线程的排队、等待与唤醒等底层操作。

2.2、队列同步器分析：同步队列

同步器依赖内部的同步队列：一个FIFO双向队列，完成同步状态的管理。

  • 当前线程获得同步状态失败时，同步器会将当前线程以及等待状态等信息构造成一个节点，加入队列，同时堵塞当前线程（被UNSAFE.park()操作挂起 ）
  • 同步状态释放时，会把下一个节点的线程唤醒，使其再次尝试获取同步状态。

同步器获取同步状态的关键：内部属性state

  • 根据state 的值来判断当前锁是否被占用
  • 申请锁、释放锁的操作其实就是 对 state 属性的 CAS修改

节点的属性

  • 同步构造器拥有头结点和尾结点，没有成功获取同步状态的线程会变成节点加入队列尾部
  • 同步队列遵循FIFO，头结点是获取同步状态成功的节点，首节点的线程结束后会释放同步状态，唤醒后继节点，成功获得同步状态的节点会把自己设置为头结点，由于只有一个线程能够获取到同步状态，所以设置头结点不需要CAS

2.3、队列同步器分析：独占式同步状态

​ 调用同步器的模板方法 acquire(int arg) 可以获取同步状态(相当于获得锁)，该方法对中断不敏感，也就是说获取同步失败时进入同步队列时，如果进行中断不会移出同步队列

addWriter方法：添加到同步队列尾部

acquireQueued方法：节点进入同步队列之后，会判断是否应该堵塞，若不堵塞就自旋。堵塞之后的唤醒依靠前驱节点的的出队或者堵塞线程被中断。

release方法：释放同步状态，唤醒后面的节点

总结

  1. 先尝试CAS获取一次锁
  2. 如果获取失败，就把自己封装成一个节点放入队列（这里进行同步队列的懒加载）
  3. 封装之后会先判断自己的前驱节点是不是头节点，如果是，那么就会再次尝试CAS获取锁，成功则自己设为头节点。若 CAS获取失败，就会阻塞：正常唤醒，interrupted会被不断重置为false，防止进入自我中断。
  4. 被唤醒之后重复执行步骤3。

特点：同时能够有多个线程获取到同步状态

doAcquireShared：先添加到同步队列，然后不断自旋，前驱节点为头结点则尝试获取同步状态，获取到同步状态时会立即唤醒下一个节点。

总结

独占：一次只能有一个线程获取到同步状态，获取失败的线程会在自旋中判断堵塞，只有获得同步状态的线程释放后才会唤醒下一节点。

共享：一次可以有多个线程获取到同步状态，获取失败的线程会在自旋中判断堵塞，获得同步状态的线程释放后唤醒线程。另外注意一点，就是如果当前占有锁的是 共享节点，就会不断的唤醒后面的节点，直到遇到写节点。

  • 每次释放锁或唤醒后面的节点
  • 每次获取锁也会唤醒后面的节点，保证 共享节点 能够不断的被唤醒。

2.7、总结

AQS核⼼思想：，如果被请求的共享资源空闲，则将当前请求资源的线程设置为有效的⼯作线程，并且将共享资源设置为锁定状态。如果被请求的共享资源被占⽤，那么就需要⼀套线程阻塞等待以及被唤醒时锁分配的机制，这个机制AQS 是⽤ CLH 队列锁实现的，即将暂时获取不到锁的线程加⼊到队列中。

AQS的应用：

  • Semaphore **(**信号量)-允许多个线程同时访问：
  • CountDownLatch（倒计时器）：让线程阻塞，倒计时结束才被唤醒
  • CyclicBarrier (循环栅栏)：设置屏障

3.3、ReentrantLock锁：公平锁概念

原理：ReentrantLock提供了两个同步器，实现公平锁和非公平锁，默认是非公平锁！

  • 公平锁：如果同步队列有等待的节点，新节点就直接加入堵塞队列。符合先进先出
  • 非公平锁：先尝试获取锁，获取失败再加入堵塞队列

Lock lock =new ReentrantLock(true); 这种方式创建的就是公平锁，默认非公平锁

3.2、ReentrantLock锁：可重入概念

  • 不可重入：线程只能获取一次锁。如果重复获取锁会容易造成死锁。 sun.lock
  • 可重入：线程可以多次获取同一把锁。避免死锁

原理：

  • AQS内部 维护了一个 变量 state 等于0 表示无锁，**等于1表示获取到锁，大于1 就是可重入锁**。
  • 通过判断当前线程是否已经持有锁来判定能不能继续获得锁。

注意：获得了几次锁，就要释放几次，只有state为0，才表明不再持有该锁，成功释放

4、 读写锁

4.1、读写锁：简单使用

ReentrantReadWriteLock作用：弥补独占锁的不足，公司的场景：读多写少

当读操作远远高于写操作时，这时候使用读写锁让读-读可以并发，提高性能。

特性：

  • 公平与非公平

  • 支持锁降级：写锁可以降级成读锁，不支持锁升级（持有读锁去获取写锁）

  • 支持同一锁的重入，或者锁升级的重入

  • 支持可中断

  • 写锁支持condition，读锁不支持

  • API方面

API：

  • 读写互斥
  • 写写互斥
  • 读读不互斥

获取释放锁

  • 存在写锁时，不能继续获得锁
  • 存在读锁时，可以继续获得读锁。
  • 获取锁的方式：CAS更新标志位
  • 释放锁的方式：读锁要保证线程安全，写锁普通释放

锁降级：已经持有写锁，先获取读锁，然后释放写锁，

目的：保证数据的可见性，若线程A直接释放锁，此时另一个线程B获取锁并修改数据，线程A无法感知到数据的变化。

重入时降级支持：即持有写锁的情况下去获取读锁

读写锁实现一致性缓存：更新时，是先清缓存还是先更新数据库？

假设查询线程 A 查询数据时恰好缓存数据由于时间到期失效，或是第一次查询：这种情况的出现几率非常小

读写锁改进：

  1. 更新： 加写锁 ——-》 查询数据库、清缓存 ——-》 释放写锁
  2. 查询： 加读锁 ——-》 从缓存查 ——-》 释放读锁 ——-》 缓存为空加写锁 —–》 从数据库查 —–》 释放写锁

总结：更新加写锁，读缓存加写锁，读数据库更新缓存加读锁。总之就是需要更新的一律加上写锁。

5、等待通知队列

newCondition等待通知队列概念：

  • 替代object里面的wait和notify() 在多线程场景下更加灵活使用等待通知功能。
  • Lock+Condition 替代：synchronized+Object
  • 通过Lock 的newCondition方法获取

注意：Lock也可以使用 wait和notify()

API：

实现原理

1、分析如何创建的

2、构造：ConditionObject是队列同步器的一个内部类，每个Condition都包含一个队列（等待队列）。

3、等待队列：一个FIFO的队列（双向链表），每个节点都包含一个线程引用，该线程就是在Condition对象上等待的线程，如果调用了await方法，调用的线程就会释放锁，构造成节点加入等待队列进入等待状态。

节点引用更新没有使用CAS：调用await必定获得了锁，不会有线程安全问题。

注意：同步队列和静态队列都属于同步器的静态内部类

4、等待：调用await() 方法即可让一个线程进入等待，相当于一个处于同步队列的线程移动到等待队列中，并且把 标记 waitStatus 设置为 -2，表示处于等待，等待队列通过中断被异常唤醒，会抛出异常。

5、唤醒（打断、超时）：唤醒也是先进先出的，唤醒是把等待队列移动到同步队列，并且使用LockSupport唤醒，注意唤醒后没获取到锁，可能短暂自旋后继续休眠，所以唤醒不一定能立即获得锁。若获取到了锁，就会从wait方法返回，继续执行未完成的任务。

注意：

  1. 调用唤醒和等待的线程必须获得lock锁
  2. await 调用的是 LockSupport 方法进行阻塞的，该方法不释放锁，释放锁的过程 有await的其他步骤完成。

LockSupport和wait

  • wait需要获得锁之后才能使用，而LockSupport 不需要
  • wait 要么随机唤醒，要么唤醒全部，而 LockSupport 是精确唤醒某个线程
  • wait 和notify 不能互换 顺序，而LockSupport 可以

LockSupport详解

每个线程都有自己的一个 Parker 对象，由三部分组成 _counter， _cond和 _mutex

打个比喻线程就像一个旅人，Parker 就像他随身携带的背包，条件变量 _ cond就好比背包中的帐篷。_counter 就好比背包中的备用干粮（0 为耗尽，1 为充足）

  1. 调用park，就是查看标志位，如果为1，就置为0，如果为0，就一直等待。
  2. 调用upark，就是把标志位设置为1，注意是不能叠加的
