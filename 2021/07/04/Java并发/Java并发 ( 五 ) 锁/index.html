1、锁介绍

1.1、各种锁的概念

1.2、ReentrantLock和Synchronize区别

  1. Synchronized是内置关键字，Lock是类

  2. Synchronized无法判断获取锁得状态，Lock可以

  3. Synchronized可以自动释放锁(一个线程执行完)，Lock必须手动释放(死锁的可能)

  4. Synchronized线程1(获得锁，堵塞)线程2(一直等)，Lock会导入到虚拟的阻塞队列进行自旋等待

  5. Synchronized是可重入锁，不能中断，非公平；Lock可重入，可设置公平度

  6. Synchronized适合锁少的代码同步，Lock适合锁大量的同步代码

2、队列同步器AQS

队列同步器AbstractQueuedSynchronizer：用于构建锁或其他同步组件的基础框架，使用了一个int成员变量表示同步状态，内置的 FIFO 队列来完成资源获取线程的排队工作。

​ 同步器的主要实现方式是继承，子类通过继承同步器实现抽象方法管理同步状态。同步器提供的三个方法可以保证状态的改变是线程安全的

​ 同步器是实现锁（其他任意同步组件）的关键，在锁的实现中聚合同步器，从而实现锁的语义。

  • 锁是面向使用者的，定义了使用者和锁交互的接口，隐藏了细节
  • 同步器面向的是锁的实现者，简化了锁的实现方式，屏蔽了同步状态管理、线程的排队、等待与唤醒等底层操作。

2.2、队列同步器分析：同步队列

同步器依赖内部的同步队列：一个FIFO双向队列，完成同步状态的管理。

  • 当前线程获得同步状态失败时，同步器会将当前线程以及等待状态等信息构造成一个节点，并把其加入队列，同时堵塞当前线程，
  • 同步状态释放时，会把首节点中的线程唤醒，使其再次尝试获取同步状态。

节点的属性

  • 同步构造器拥有头结点和尾结点，没有成功获取同步状态的线程会变成节点加入队列尾部
  • 当一个线程获得了同步状态（锁），其他线程就没办法获取到同步状态，所以就没法转换成节点加入同步队列，所以提供了 compareAndSetTail() 设置尾节点。
  • 同步队列遵循FIFO，头结点是获取同步状态成功的节点，首节点的线程结束后会释放同步状态，唤醒后继节点，成功获得同步状态的节点会把自己设置为头结点，由于只有一个线程能够获取到同步状态，所以设置头结点不需要CAS

2.3、队列同步器分析：独占式同步状态

​ 调用同步器的模板方法 acquire(int arg) 可以获取同步状态(相当于获得锁)，该方法对中断不敏感，也就是说获取同步失败时进入同步队列时，如果进行中断不会移出同步队列

addWriter方法：添加到同步队列尾部

acquireQueued方法：节点进入同步队列之后，会判断是否应该堵塞，若不堵塞就一直自旋。堵塞之后的唤醒依靠前驱节点的的出队或者堵塞线程被中断。

release方法：释放同步状态，唤醒后面的节点

总结

​ 获取同步状态时，队列同步器会维护一个队列，获取同步失败的会加入队列进行自旋（自旋会判断是否需要堵塞），当被唤醒（前驱节点释放同步状态或者本线程中断）且前驱节点为头结点时，就会尝试获取同步状态，获取成功后当前线程会设置成队首。

2.4、队列同步器分析：共享式同步状态

特点：同时能够有多个线程获取到同步状态

acquireShared方法分析：尝试获取同步状态，失败则执行doAcquireShared

doAcquireShared：先添加到同步队列，然后不断自旋，前驱节点为头结点则尝试获取同步状态，获取到同步状态时会立即唤醒下一个节点。

releaseShared：释放同步状态

doReleaseShared方法

注意：

  1. 想要获取同步状态，只有一条路：重写的acquireShared() 方法返回 大于等于0
  2. 与独占式相比：释放的情况下可能有多个线程同时释放，所以通过循环和CAS保证线程安全。

总结

独占：一次只能有一个线程获取到同步状态，获取失败的线程会在自旋中判断堵塞，只有获得同步状态的线程释放后才会唤醒下一节点。

共享：一次可以有多个线程获取到同步状态，获取失败的线程会在自旋中重复获取和判断堵塞，一旦获取到同步状态或者释放同步状态

2.5、队列同步器分析：独占式超时

tryAcquireNanos分析：如果中断会抛出异常，直接返回。否则尝试获取同步状态，获取失败后进行超时获取。

doAcquireNanos方法分析：先添加到同步队列，然后不断自旋：前驱节点为头结点时尝试获取同步，获取失败会判断超时，未超时则等待，超时则抛出异常返回。

2.6、队列同步器分析：堵塞条件判断

队列的状态

shouldParkAfterFailedAcquire（前驱节点，当前节点）方法详解

  • 把node的有效前驱（有效是指node不是CANCELLED的）找到，并且将有效前驱的状态设置为SIGNAL，之后便返回true代表马上可以阻塞了。

2.7、总结

AQS核⼼思想：，如果被请求的共享资源空闲，则将当前请求资源的线程设置为有效的⼯作线程，并且将共享资源设置为锁定状态。如果被请求的共享资源被占⽤，那么就需要⼀套线程阻塞等待以及被唤醒时锁分配的机制，这个机制AQS 是⽤ CLH 队列锁实现的，即将暂时获取不到锁的线程加⼊到队列中。

AQS的应用：

  • Semaphore **(**信号量)-允许多个线程同时访问：
  • CountDownLatch（倒计时器）：让线程阻塞，倒计时结束才被唤醒
  • CyclicBarrier (循环栅栏)：设置屏障

3.2、ReentrantLock锁：可重入概念

  • 不可重入：线程只能获取一次锁。如果重复获取锁会容易造成死锁。 sun.lock
  • 可重入：线程可以多次获取同一把锁。避免死锁

原理：

  • AQS内部 维护了一个 变量 state 等于0 表示无锁，**等于1表示获取到锁，大于1 就是可重入锁**。
  • 通过判断当前线程是否已经持有锁来判定能不能继续获得锁。

注意：获得了几次锁，就要释放几次，只有state为0，才表明不再持有该锁，成功释放

3.3、ReentrantLock锁：公平锁概念

原理：

  • 公平锁：如果同步队列有等待的节点，新节点就直接加入堵塞队列。符合先进先出
  • 非公平锁：先尝试获取锁，获取失败再加入堵塞队列

使用非公平锁

  • Lock lock =new ReentrantLock(true); 这种方式创建的就是公平锁，默认非公平锁

3.4、ReentrantLock锁：lockInterruptibly()等待中断

中断：类似 sleep 等代码 都需要捕获异常，设置中断后 如果执行到需要捕获异常的代码时，就会被异常捕获，然后跳转到捕获代码之后的代码执行。

lockInterruptibly()： 需要捕获异常，所以当设置中断时，就会被捕获抛出异常。

4、 读写锁

ReentrantReadWriteLock作用：弥补独占锁的不足，公司的场景：读多写少

特性：

  • 公平与非公平

  • 支持锁降级：写锁可以降级成读锁

  • 支持可重入

  • 支持可中断

  • 支持condition

  • API方面

API：

  • 读写互斥
  • 写写互斥
  • 读读不互斥

原理：state是一个int 变量 ，总共32位，所以拿出前16位 作为读。后16位表示写。

判断读写的状态：通过位运算

  • state & 0x0000FFFF 把高16位抹去表示写状态
  • state >>> 16 位右移动16位即抹去右边16位表示读状态

获取释放锁

  • 存在写锁时，不能继续获得锁（）
  • 存在读锁时，可以继续获得读锁。
  • 获取锁的方式：CAS更新标志位
  • 释放锁的方式：读锁要保证线程安全，写锁普通释放

锁降级：已经持有写锁，先获取读锁，然后释放写锁，

目的：保证数据的可见性，若线程A直接释放锁，此时另一个线程B获取锁并修改数据，线程A无法感知到数据的变化。

5、等待通知队列

newCondition等待通知队列概念：

  • 替代object里面的wait和notify() 在多线程场景下更加灵活使用等待通知功能。
  • Lock+Condition 替代：synchronized+Object
  • 通过Lock 的newCondition方法获取

API：

实现原理

1、分析如何创建的

2、构造：ConditionObject是队列同步器的一个内部类，每个Condition都包含一个队列（等待队列）。

3、等待队列：一个FIFO的队列，每个节点都包含一个线程引用，该线程就是在Condition对象上等待的线程，如果调用了await方法，调用的线程就会释放锁，构造成节点加入等待队列进入等待状态。

节点引用更新没有使用CAS：调用await必定获得了锁，不会有线程安全问题。

注意：同步队列和静态队列都属于同步器的静态内部类

4、等待：调用await() 方法即可让一个线程进入等待，相当于一个处于同步队列的线程移动到等待队列中，等待队列通过中断被异常唤醒，会抛出异常。

5、唤醒：唤醒也是先进先出的，唤醒是把等待队列移动的同步队列，并且使用LockSupport唤醒，注意唤醒后没获取到锁，可能短暂自旋后继续休眠，所以唤醒不一定能立即获得锁。若获取到了锁，就会从wait方法返回，继续执行未完成的任务。
