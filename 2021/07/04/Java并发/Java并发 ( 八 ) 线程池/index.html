1、线程池概述

1.1、概念

不建议用： new Thread().start();

  • 浪费服务器的资源，创建和销毁 浪费资源

  • 不能立马就销毁对象。大量的对象，GC

  • 线程的生命周期比较长，不好管理线程的状态，中断….

  • 不支持周期，调度的功能。线程大容易竞争资源

线程池：维护一批运行的线程，不断的从任务队列中取出线程来运行（调用run方法）。

优点：

  • 降低资源的消耗，不用创建那么多线程
  • 提高响应速度，不需要创建线程，而是直接被已存在的线程调用 run 方法
  • 把线程集中起来便于管理，同时增加FutureTask Future 的多种API

2、Executor框架简介

2.1、简介

Executor框架的结构

  • 任务：即线程，需要实现Runnable或者callable接口
  • 任务的执行：包括任务执行机制的核心接口 Executor ，以及继承自 Executor 接口的 ExecutorService 接口。ThreadPoolExecutor 和 ScheduledThreadPoolExecutor 这两个关键类实现了 ExecutorService 接口。
  • 异步计算的结果：Future 接口以及 Future 接口的实现类 FutureTask 类都可以代表异步计算的结果。

任务执行的过程

线程池直接执行线程对象

  1. 新建线程池对象
  2. 创建线程对象 Runnable 或者 Callable
  3. 线程池对象直接 执行该线程，Callable返回 Future对象

线程池通过 FutureTask 执行线程对象

  1. 新建线程池对象
  2. 创建线程对象 Runnable 或者 Callable，然后通过 FutureTask 进行包装
  3. 线程池对象 执行 FutureTask 对象，线程池无返回值，但是可以通过 FutureTask 对象获取结果。

2.2、 Executor 对象的主要成员

  • Executor接口：Executor框架的基础，实现任务的提交和执行分离
  • ExecutorService：一个线程池接口，继承了Executor接口，一般用作 具体的线程池 向上转型成该类对象。
  • ThreadPoolExecutor：线程池的核心实现类，用于创建线程池，通常使用工厂类 Executors 来创建（也可以自定义）。继承了AbstractExecutorService类，AbstractExecutorService 继承了 ExecutorService接口。
  • ScheduledThreadPoolExecutor：线程池的一个实现类，用于创建线程池，能给延时执行线程。
  • Future接口：线程池执行Callable 任务后，返回的结果为 Future类型

3、线程池详解

3.1、简介

  • ThreadPoolExecutor 是 线程池的核心实现类，提供了 3种线程池的实现
  • ScheduledThreadPoolExecutor 继承了 ThreadPoolExecutor，主要用于给定延迟之后运行任务或者定期执行任务。

3.2、线程池的状态

ThreadPoolExecutor 使用 int 的高 3 位来表示线程池状态，低 29 位表示线程数量

这些信息存储在一个原子变量 ctl 中，目的是将线程池状态与线程个数合二为一，这样就可以用一次 cas 原子操作
进行赋值

3.3、线程池构造方法

  1. corePoolSize 核心线程数目 (最多保留的线程数)
  2. maximumPoolSize 最大线程数目(核心线程数加上救急线程数)
  3. keepAliveTime 救急线程的生存时间(核心线程没有生存时间这个东西，核心线程会一直运行)
  4. unit 时间单位 - 针对救急线程
  5. workQueue 阻塞队列：核心线程用完时进入阻塞队列
  6. threadFactory 线程工厂 - 可以为线程创建时起个好名字
  7. handler 拒绝策略

线程添加策略

  1. 线程池中刚开始没有线程，当一个任务提交给线程池后，线程池会创建一个新线程来执行任务。
  2. 线程池的核心线程满了之后，新任务就会加入阻塞队列，线程池的线程会不断从队列获取线程执行
  3. 阻塞队列也满了之后，如果再有新任务进来，就会创建应急线程执行任务。
  4. 如果应急线程也满了，就会执行拒绝策略。
  5. 任务提交高峰过去之后，如果应急线程如果一段时间没有接收到任务，线程就会被关闭。

拒绝策略

ThreadPoolExecutor 提供的四大拒绝策略

  1. Dubbo 的实现，在抛出 RejectedExecutionException 异常之前会记录日志，并 dump 线程栈信息，方
    便定位问题
  2. Netty 的实现，是创建一个新线程来执行任务
  3. ActiveMQ 的实现，带超时等待（60s）尝试放入队列，类似我们之前自定义的拒绝策略
  4. PinPoint 的实现，它使用了一个拒绝策略链，会逐一尝试策略链中每种拒绝策略

根据构造器创建线程池

3.4、JDK内置线程池

  • SingleThreadExecutor：单个线程池

  • FixedThreadPool：自己控制线程数量

  • CachedThreadPool：根据需要自动决定容量的线程池

  • WorkStealingPool：JDK8新增的

  • ScheduledThreadPool：可以周期性和延迟执行线程

  • SingleThreadScheduledExecutor

newFixedThreadPool

  1. 核心线程数 == 最大线程数（没有救急线程被创建），因此也无需超时时间
  2. 阻塞队列是无界的，可以放任意数量的任务
  3. 适用于任务量已知，相对耗时的任务

newCachedThreadPool

  1. 核心线程数是 0， 最大线程数是 Integer.MAX_VALUE，说明全部是救急线程，救急线程的空闲生存时间是 60s，可以无限制的创建。
  2. 队列采用了 SynchronousQueue 实现特点是，它没有容量，没有线程来取是放不进去的（一手交钱、一手交货）
  3. 整个线程池表现为线程数会根据任务量不断增长，没有上限，使用不当会造成大量创建线程，当任务执行完毕，空闲 1分钟后释放线
    程。 适合任务数比较密集，但每个任务执行时间较短的情况

newSingleThreadExecutor

使用场景

  • 希望多个任务排队执行。线程数固定为 1，任务数多于 1 时，会放入无界队列排队。任务执行完毕，这唯一的线程也不会被释放。
  • 和自己创建单线程执行任务的区别：自己创建一个单线程串行执行任务，如果任务执行失败而终止那么没有任何补救措施，而线程池还会新建一个线程，保证池的正常工作

和Executors.newFixedThreadPool(1) 初始时为1时的区别

  • FinalizableDelegatedExecutorService 应用的是装饰器模式，只对外暴露了 ExecutorService 接口，因此不能调用 ThreadPoolExecutor 中特有的方法
  • Executors.newFixedThreadPool(1) 初始时为1，以后还可以修改，对外暴露的是 ThreadPoolExecutor 对象，可以强转后调用 setCorePoolSize 等方法进行修改

ScheduledThreadPool

Executors 返回线程池对象的弊端如下：

  • FixedThreadPool 和 SingleThreadExecutor ： 允许请求的队列长度为 Integer.MAX_VALUE,可能堆积大量的请求，从而导致 OOM。
  • CachedThreadPool 和 ScheduledThreadPool ： 允许创建的线程数量为 Integer.MAX_VALUE ，可能会创建大量线程，从而导致 OOM。

所以：最好使用有界队列

3.7、异步模式之工作线程

定义：让有限的工作线程（Worker Thread）来轮流异步处理无限多的任务。也可以将其归类为分工模式，它的典型实现就是线程池，也体现了经典设计模式中的享元模式。

注意：不同任务类型应该使用不同的线程池，这样能够避免饥饿，并能提升效率

案例：如果一个餐馆的工人既要招呼客人（任务类型A），又要到后厨做菜（任务类型B）显然效率不咋地，分成服务员（线程池A）与厨师（线程池B）更为合理

创建多大的线程池合适

  1. CPU 密集型运算
    通常采用 cpu 核数 + 1 能够实现最优的 CPU 利用率，+1 是保证当线程由于页缺失故障（操作系统）或其它原因导致暂停时，额外的这个线程就能顶上去，保证 CPU 时钟周期不被浪费
  2. I/O 密集型运算
    CPU 不总是处于繁忙状态，例如，当你执行业务计算时，这时候会使用 CPU 资源，但当你执行 I/O 操作时、远程RPC 调用时，包括进行数据库操作时，这时候 CPU 就闲下来了，你可以利用多线程提高它的利用率。
    1. 经验公式如下：线程数 = 核数 * 期望 CPU 利用率 * 总时间(CPU计算时间+等待时间) / CPU 计算时间
      例如 4 核 CPU 计算时间是 50% ，其它等待时间是 50%，期望 cpu 被 100% 利用，套用公式
      4 * 100% * 100% / 50% = 8
      例如 4 核 CPU 计算时间是 10% ，其它等待时间是 90%，期望 cpu 被 100% 利用，套用公式
      4 * 100% * 100% / 10% = 40

4、FutureTask详解

概念：FutureTask继承了 Future类，实现类 Runnable 接口，有多种状态

作用：

  1. 作为 线程池执行的返回值，通过 get 方法获取结果
  2. 把线程作为参数填入 FutureTask 对象中，这样 ExecutorService 对象 submit 的就不是线程而是 FutureTask 对象，
  3. 通过· cancel 取消任务。
  4. 作为一个线程单独执行。
