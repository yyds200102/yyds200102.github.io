1、线程上下文切换

多线程原理：即使是单核处理器也支持多线程执行代码，可以通过CPU给每个线程分配CPU时间片来实现这个机制，由于时间片时间很短，所以感觉是多个线程同时执行的。

上下文切换：CPU通过时间片分配执行多线程，当前任务执行一个时间片后会切换到下一个任务，但是切换前会保存上一个任务的状态，下一次继续执行当前任务时会从保存的状态处继续执行任务。任务从当前任务保存到下一个任务加载的过程就是上下文的切换。

注意：单核CPU执行多线程任务一定会触发线程上下文切换，因为一个CPU⽽⼀个 CPU 核心在任意时刻只能被⼀个线程使用，多核CPU执行多线程任务时，若CPU能满足所有线程的需要，就会并行执行任务而没有线程上下文切换(但是一般情况不可能)，当CPU不够所有线程需要时，也会发生线程上下文切换。

1.1、多线程一定更快么

不一定，因为上下文切换需要消耗时间

使用工具测出，1秒中发生1000多次上下文切换。

1.2、减少上下文切换

  • 无锁并发编程：CAS + volatile 多线程竞争时，会引起上下文切换，可以使用一些办法避免使用锁，如不同的线程 按照Hash算法处理不同的区域，一个数据只能有一个线程处理。
  • 尽量少的创建线程
  • 协程，在单线程里面实现多任务的调度

1.3、死锁

死锁演示：

  1. 线程t1 执行 占有对象A，线程t2 执行 占有对象B
  2. 线程t1 接着执行需要对象B ，但是无法获取，于是线程 t1 等待
  3. 线程t2 接着执行需要对象A ，但是无法获取，于是线程 t2 等待
  4. 线程 t1 和 t2 将会无限的等待下去。

实际开发：实际开发中不会写出这么愚蠢的代码，但是有可能出现以下情形

  • t1 拿到资源A（锁）之后，因为异常情况而没有释放
  • t1 拿到资源A （锁）之后，释放的时候抛出异常，没有释放成功

避免死锁：

  • 避免一个线程获取多个锁
  • 避免一个线程在锁内占用多个资源
  • 尝试使用定时锁，使用lock.trylock(timeout) 来替代内部锁机制
  • 对于数据库锁，加锁和释放必须在一个数据库连接中，否则会出现解锁失效的问题。

1.4、资源限制的挑战

资源限制：程序的执行速度受限于软件或者硬件资源，如网站带宽2M/s，即使再多的线程，速度也不会变快。

问题：代码串行变成并发执行，但是如果将某段串行的代码并发执行，那么由于资源受限，反而会执行的更慢，因为增加了上下文切换和资源调度的时间，

解决：考虑集群并行执行程序，单机的资源有限，就让程序在多机上运行，如不同的任务通过算法选择一台机器执行程序

资源受限的情况下并发：根据不同的资源限制调整程序的并发度，比如下载文件程序依赖两个资源：带宽和硬盘读写。有数据库操作时，涉及数据库连接数，如果SQL执行的快，而线程的数量比数据库连接数大很多时，某些线程会堵塞，等待数据库连接。

2、可见性

2.1、退不出的循环

每个线程都有自己的工作内存，频繁地从主存中读取run的值，jit即时编译器会将run的值缓存至自己工作内存中的高速缓存中，减少对主存中run的访问以提高效率，当主线程更新run的值时，t线程并不会更新里面的值。

2.2、volatile可见性保证

概念：用来修饰成员变量和静态成员变量（放在主存中的变量），保证所有线程看到某个变量的值都是一致的，保证了共享变量的”可见性”，同时不会引起线程上下文的切换。

注意：volatile不能保证原子性。

实现原理：每次操作完数据，JVM会发送一条Lock前缀的指令

  1. Lock前缀的指令会引起处理器缓存写回到主存
  2. 一个处理器缓存写回到主存，会导致其他处理器的缓存失效。

效果：volatile的修饰的变量具有可见性

  1. 写一个volatile的变量，JMM会把该线程的本地内存的共享变量值强制刷新回主存。
  2. 读一个volatile的变量，JMM会把该线程的本地内存的共享变量置为无效，从主存获取值。

犹豫模式

定义：Balking （犹豫）模式用在一个线程发现另一个线程或本线程已经做了某一件相同的事，那么本线程就无需再做了，直接结束返回。有点类似于单例。

2.3、synchronized保证可见性

原理：和volatile基本一致，可以保证代码块内部变量的可见性

  1. volatile的每次读，就相当于synchronized的加锁，会强制从主存读取共享变量
  2. volatile的每次写，就相当于synchronized的解锁，会强制把共享变量刷新会主存

2.4、总结

  1. synchronized 是属于重量级操作，性能相对更低。
  2. volatile的是轻量级操作

3、原子性

3.1、原子性的定义

原子性：一个操作具有原子操作，即每个操作直接不可分割，那么我们称它具有原子性

4、有序性

4.1、重排序优化

鱼罐头加工优化

工人借助机器，把每条鱼的步骤执行分离开，保证每个机器都能被充分利用。

CPU指令优化：指令流水技术

概念：现代处理器会设计为一个时钟周期完成一条执行时间最长的CPU指令。为什么这么做呢？指令可以划分为一个更小的阶段，例如每条指令都可以分为： 取指令 — 指令译码 — 执行指令 —– 内存访问 —- 数据写回 5个步骤

重排序：CPU利用重排序的思想，把不同指令的不同部分组合起来一块执行，提高效率。

现代处理器支持多级指令流水线，例如同时支持 取指令 -- 指令译码 -- 执行指令 -- 内存访问 -- 数据写回 的处理器，就可以称之为 五级指令流水线，这时CPU可以在一个时钟周期里面，同时运行五条指令的不同阶段（相当于一条执行时间最长的浮渣指令），通过这个技术，虽然不能缩短单条指令的执行时间，但是提高了指令的吞吐率。

重排序后果

导致指令的顺序被打乱了，所以部分指令重排序会被静止，JMM就保证了单线程条件下，重排序不会影响结果

4.2、重排序带来的问题

结果可能的情况

  • 情况1：线程1 先执行，这时 ready = false，所以进入 else 分支结果为 1
  • 情况2：线程2 先执行 num = 2，但没来得及执行 ready = true，线程1 执行，还是进入 else 分支，结果为1
  • 情况3：线程2 执行到 ready = true，线程1 执行，这回进入 if 分支，结果为 4（因为 num 已经执行过了）
  • 情况4：:线程2 执行 ready = true，切换到线程1，进入 if 分支，相加为 0，再切回线程2 执行 num = 2。

这种现象叫做指令重排，是 JIT 编译器在运行时的一些优化，这个现象需要通过大量测试才能复现，可以使用jcstress工具进行测试。

总结：重排序导致某些条件判断与我们的预期不符合，导致程序进入错误的分支。

4.3、volatile解决重排序

volatile 的底层实现原理是内存屏障，Memory Barrier（Memory Fence），会禁止部分重排序

  1. 对 volatile 变量的写指令后会加入写屏障
  2. 对 volatile 变量的读指令前会加入读屏障

不能解决指令交错：非原子性的原因

  1. 写屏障仅仅是保证之后的读能够读到最新的结果，但不能保证其它线程的读跑到它前面去
  2. 而有序性的保证也只是保证了本线程内相关代码不被重排序

4.4、synchronized解决重排序

synchronized解决重排序的问题很简单，加上synchronized之后，这两个方法之间就相当于串行执行，相当于单线程执行。而JMM保证了单线程执行下的程序正确性。

5、CAS深入

5.1、无锁并发

银行问题

接口里面的demo方法会创建100个线程，并且执行 扣 10元的操作。现在希望完善存钱和取钱的方法达到线程安全

接口实现：使用CAS方式

5.2、CAS为什么能够无锁并发

  1. compareAndSet，保证修改的过程中，accounts没有被其他的线程动过
  2. volatile：AtomicInteger类里面维护的变量value，使用volatile修饰的，CAS 必须借助 volatile 才能读取到共享变量的最新值来实现【比较并交换】的效果。

5.3、ABA问题

描述：CAS判断操作，是通过值有没有变化，但是可能出现 A—> B — >A 的情况，CAS认为没有变化，但是实际上已经发生了改变

解决：使用JDK1.5开始提供的AtomicStampedReference原子引用类型，把数据包装到原子引用类型里面，通过原子引用类型操作变量。

循环开销大

描述：CAS可能会一直自旋，可以CPU带来很大的执行开销。

只能保证一个共享变量的原子操作

描述：对一个共享变量操作时，可以进行CAS，但是对于多个变量进行处理就不行。

解决：

  • 使用锁
  • 使用JDK1.5中的AtomicReference引用类型类，把多个变量封装到一个引用类型里面

5.4、CAS和锁的踪迹

  1. CAS 是基于乐观锁的思想：最乐观的估计，不怕别的线程来修改共享变量，就算改了也没关系，我吃亏点再重试呗。
  2. synchronized 是基于悲观锁的思想：最悲观的估计，得防着其它线程来修改共享变量，我上了锁你们都别想改，我改完了解开锁，你们才有机会。
  3. CAS 体现的是无锁并发、无阻塞并发，请仔细体会这两句话的意思
    1. 因为没有使用 synchronized，所以线程不会陷入阻塞，这是效率提升的因素之一
    2. 但如果竞争激烈(写操作多)，可以想到重试必然频繁发生，反而效率会受影响

6、synchronized优化

过去都称为重量级锁，但是在JDK1.6之后优化了（引入轻量级锁和偏向锁），有些情况下并不是那么重了。

锁的对象：当一个线程访问同步代码块时必须获得锁，退出或者抛出异常必须释放锁。

  • 普通同步方法：锁的是实例对象
  • 静态同步方法：锁的是当前类的Class对象
  • 同步方法块：锁的是synchorized括号里配置的对象。

同步实现原理

同步代码块实现原理：基于进入和退出Monitor对象

Monitor对象：对象监听器，每个对象都有一个monitor与之关联。锁住了一个对象，就是获得对象相关联的监视器。

  • monitorenter指令：在编译后插入到同步代码块的开始位置，让锁计数器数值加1。
  • monitorexit指令：插入到同步代码块的结束处和异常处，让锁计数器数值减1。
  • JVM保证每个monitorenter都必须有一个monitorexit与之匹配。

注意：

  1. 锁计数器数值初始为0，表示可以获取锁；大于0表示已经被占用；能不断累加，这也是可重入的体现。思想和 JUC 的AQS 一样。
  2. wait 和notify 等方法也依赖于 monitor 对象，这也是为什么wait 需要获得锁的原因

同步方法实现原理：基于ACC_SYNCHRONIZED 标识。

JVM会调用指令读取运行时常量池中方法的 ACC_SYNCHRONIZED 标志。该标识指明了该方法是⼀个同步方法。这样对象监视器一次只会让一个线程执行同步方法。

二者本质都是对象监视器 monitor 的获取

对象头

Klass Word为指针，指向对应的Class对象；Mark Word 存储锁信息

Java对象头：synchorized的锁信息是存在Java对象头里，数组类型3字宽，其他2字宽。Mark Word存储锁信息，来标记是哪个线程获得了该锁，下面是Mark Word存的数据。

  • 偏向锁：存放线程ID
  • 轻量级锁：指向栈中锁记录的指针
  • 重量级锁：指向互斥量的指针

锁的优化：Java1.6对synchorized进行了优化

锁的升级：JavaSE1.6中，锁有四种状态，级别从低到高依次为 无锁—> 偏向锁 —> 轻量级锁 —> 重量级锁，锁可以升级但是不能降级。

偏向锁

缘由：HotSpot作者发现，大多数情况下，锁很少存在竞争情况，大多数情况下，锁都是一个线程多次释放和获得，所以为了让线程获得锁的代价更低，引入了偏向锁

方案：

  • 无锁情况获取：直接获得锁，并在对象头和栈帧的锁记录锁偏向的线程ID（CAS方式）
  • 有锁情况获取：任意线程获取锁时，会比较是否为偏向锁是否有本线程的ID，如果有表示已经占有了锁；如果没有，则看当前是否还是偏向锁，若是则会尝试让偏向锁指向自己的线程ID，若不是则属于竞争其他锁的情况。
  • 锁的释放：当有其他线程竞争锁时，系统会判断持有锁线程是否存活，
    • 若不存活，则把对象头设置为无锁状态，然后当前线程获得锁（还是偏向锁，CAS记录ID）；
    • 若存活，则JVM会在持有偏向锁的线程执行完成之后进行锁的升级，把对象头的 Mark Word 拷贝到当前线程栈中的锁记录中，然后把对象头的 Mark Word 中锁记录的指针执行 自己，若成功表面竞争到了轻量级锁。

关闭锁：

总结：偏向锁就是线程A持有锁，不正常释放，直到线程B来获取锁时，才让线程A释放锁

  • 加锁：让对象头记录本线程的ID
  • 释放锁：先清空ID，在记录成新的线程的ID

轻量级锁

  • 加锁：
    1. 执行同步代码块之前，栈中创建锁记录（Lock Record）对象，储存对象的Mark Word和对象引用reference。
    2. 把锁记录中的对象引用reference 指向 要锁住的那个对象，把对象头中的Mark Word复制到锁记录中（称为Diasplaced Mark Word）。
    3. 使用 CAS 的方式尝试把 要锁住的对象 的对象头的 Mark Word 替换，让对象的对象头储存的就是锁记录的地址和状态，成功则获得了锁。
    4. 失败，则会直接膨胀成为重量级锁（将Mark Word指向重量级锁Monitor的指针，然后挂起本线程）
  • 解锁：
    1. 如果获取的是取值为 null 的锁记录 ，表示有重入，这时重置锁记录，表示重入计数减一
    2. 获取的锁记录取值不为 null，那么使用cas将Mark Word的值恢复给对象。
      • 成功，表示成功释放锁
      • 失败，表示在当前线程持有锁的这段时间内，其他线程也竞争过锁，并且发生了锁升级为重量锁，这时需要去Monitor的等待队列中唤醒一个线程去重新竞争锁。

总结： 使用CAS获取锁，如果释放时竞争激烈，就把锁升级成重量级锁。

  • 加锁：先把对象头的信息备份，然后让对象头的信息更换为当前线程的信息
  • 解锁：把备份还原

重量级锁

自旋：重量级锁竞争的时候，还可以使用自旋来进行优化，如果当前线程自旋成功（即在自旋的时候持锁的线程释放了锁），那么当前线程就可以不用进行上下文切换就获得了锁

加锁和释放：即线程的阻塞和唤醒，这个操作是借助操作系统的系统调用来实现的，常见的 Linux 下就是利用 pthread 的 mutex 来实现的，所以消耗资源大。

注意：自旋会占用 CPU 时间，单核 CPU 自旋就是浪费，多核 CPU 自旋才能发挥优势。在 Java 6 之后自旋锁是自适应的，比如对象刚刚的一次自旋操作成功过，那么认为这次自旋成功的可能性会高，就多自旋几次；反之，就少自旋甚至不自旋，总之，比较智能。Java 7 之后不能控制是否开启自旋功能

总结

偏向锁和轻量级锁是为了针对特定环境下，没有多线程竞争时，没必要用重量锁消耗资源。

一旦竞争激烈一点，就会立即升级为重量级锁，重量级锁会进行轻度自旋来优化效率。

7、JMM三大特性总结

  • 原子性：保证指令不会受到线程上下文切换的影响
  • 可见性 ：保证指令不会受 cpu 缓存的影响
  • 有序性 ：保证指令不会受 cpu 指令并行优化的影响

synchronized ：保证原子性和可见性

volatile：保证了可见性，防止了指令重排

二者都能保证有序性：

  • synchronized ：通过同步机制，保证代码块之间的有序进行
  • volatile：通过禁止部分指令重排，保证代码执行顺序
