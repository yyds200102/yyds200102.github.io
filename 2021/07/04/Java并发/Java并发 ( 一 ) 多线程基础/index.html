1、线程简介

1.1、什么是线程

​ 操作系统运行一个程序时，会为其创建一个进程。操作系统调度的最小单元是线程，也叫轻量级进程，一个进程可以创建多个线程，每个线程有自己的计数器、栈、局部变量，能够访问共享的方法区、堆。处理器在线程上高速切换，让使用者感觉线程在同时执行。

​ 一个Java程序就有多个线程，至少有主线程、GC线程，异常处理线程。

程序、进程、线程

  • 程序：指一段静态的代码，静态对象。
  • 进程：程序的一次执行过程，或是正在运行的一个程序。
  • 线程：进程可进一步细化为线程，线程存在于进程内，是进程的一个子集

进程属于操作系统的范畴，主要是同⼀段时间内，可以同时执行⼀个以上的程序，而线程则是在同⼀程序内几乎同时执行⼀个以上的程序段

线程 是 进程 划分成的更小的运行单位。线程和进程最大的不同在于基本上各进程是独立的，而各线程则不⼀定，因为同⼀进程中的线程极有可能会相互影响。线程执行开销小，但不利于资源的管理和保护；而进程正相反

并行与并发

  • 并行：多个线程同时执行多个任务。
  • 并发：多个线程同时执行一个任务。

并发：操作系统中有一个组件叫做任务调度器，将 cpu 的时间片（windows下时间片最小约为 15 毫秒）分给不同的程序使用，只是由于 cpu 在线程间（时间片很短）的切换非常快，人类感觉是同时运行的 。一般会将这种线程轮流使用 CPU 的做法称为并发（concurrent）

并行：多核 cpu下，每个核（core） 都可以调度运行线程，这时候线程可以是并行的，不同的线程同时使用不同的cpu在执行。

CPU与多线程

单核CPU：如果开启多线程，不仅不能提高效率，（因为总共只有一核CPU，是通过线程上下文的切换让程序多线程执行），还会因为线程间上下文的切换导致效率降低。

多核CPU：有些任务没有必要拆分，这时候开启多线程没有意义。

多线程的问题

内存泄漏、上下文切换、死锁等

IO操作

IO操作不占用CPU，只是我们拷贝文件一般使用阻塞IO，没有充分利用线程。非阻塞IO 和 异步 IO 能够进行优化。

同步与异步

概念：以调用方的角度讲，如果需要等待结果返回才能继续运行的话就是同步，如果不需要等待就是异步

多线程异步：多线程可以使方法的执行变成异步的，比如说读取磁盘文件时，假设读取操作花费了5秒，如果没有线程的调度机制，这么cpu只能等5秒，啥都不能做。

总结：

  • 项目中，进行费时操作时，可以开启一个新线程处理视频转换，避免阻塞主线程
  • tomcat 的异步 servlet 也是类似的目的，让用户线程处理耗时较长的操作，避免阻塞 tomcat 的工作线程
  • ui 程序中，开线程进行其他操作，避免阻塞 ui 线程

1.2、多线程的好处

更多的处理核心

处理器上的核心越来越多，超线程技术运用广泛，现在大多数计算机更加擅长并行计算

更快的响应时间

把数据性不强的操作派发给其他线程处理（也可以使用消息队列）这样响应用户请求的线程能够尽可能地处理完成，缩短了响应时间，提示用户体验。

更好的编程模型

Java提供的多线程编程的良好环境

1.3、线程的运行原理

虚拟机栈

虚拟机栈是线程私有的，每个虚拟机栈只有一个活动栈帧，对应当前执行的方法，一个线程只能同时执行一个方法。

开启多个线程后，每个线程有自己的活动栈帧，就可以同时执行多个方法了。

线程上下文切换

概念：CPU在不同线程间的切换，触发的情况如下

  • 线程的 cpu 时间片用完(每个线程轮流执行，看前面并行的概念)
  • 垃圾回收
  • 有更高优先级的线程需要运行
  • 线程自己调用了 sleep、yield、wait、join、park、synchronized、lock 等方法

每次线程上下文切换的时候，程序计数器会记录下一条指令的地址，这样程序切换回来之后就可以正确的执行了。

3、线程的终止

3.1、中断

概念：中断可以理解为线程的一个标识位，标识一个运行中的线程是否被其他线程进行了中断操作。线程内部通过检查自己是否被中断来进行响应。

作用：让处于堵塞或者等待状态的线程立即停止等待状态。并且抛出一个异常，对于wait 这种需要捕获异常的会直接跳转到catch 执行，对于 synchorized 这样的处于阻塞的，会直接跳过同步代码块。

  • 等待堵塞（wait）、睡眠、IO堵塞、同步堵塞（synchorized）、sleep，wait，join

本质：比如Thread.sleep 需要捕获异常，如果设置中断，就会抛出异常并且被捕获，程序会重捕获的代码之后继续运行； synchorized等待的线程设置中断没有效果。

5.5、join的使用

线程A调用thread.join()：表示线程A开始等待，直到线程thread执行完成后，线程A才继续执行。

原理：假设线程A调用 thread.join 这个同步方法，那么线程A就持有了thread对象的锁，所以调用wait 方法让线程A进入阻塞状态，一直等待，等到线程thread 结束之后，JVM会调用 notify 让线程A苏醒。

总之：就是通过wait方法让线程等待，JVM负责唤醒

注意：因为底层是wait 方法，所以会释放锁

5.6、ThreadLocal的使用

ThreadLocal相当于一个map：所有线程的线程ID作为map的键，值作为value，不同线程设置的值和获得的值是不同的，互相没有冲突。

原理：内部维护了一个ThreadLocalMap（节点Entry），每个线程的ID作为键，存的数据作为value。同样是懒加载的

内存泄漏问题：内部map的key 是弱引用，value是强引用，所以可能 key 被回收了，但是value 永远无法回收，ThreadLocal 解决的思路是：调用3个方法的时候，会自动清除键为 null 的 Entry 结构。

6、线程问题简单探究

6.1、根本原因

线程出现问题的根本原因是因为线程上下文切换，导致线程里的指令没有执行完就切换执行其它线程了

直接原因：– 和 ++ 的操作不是原子性的

6.2、临界区

概念：一段代码内如果存在对某些共享资源的多线程读写操作，那么称这段代码为临界区

竞态条件

多个线程在临界区执行，那么由于代码指令的执行不确定而导致的结果问题，称为竞态条件

6.3、synchorized

synchorized：可以使用在方法或这代码块上，俗称对象锁，它采用互斥的方式让同一时刻至多只有一个线程持有对象锁，其他线程访问到如果获取对象锁失败，就会进入阻塞状态。

作用：这样同时只会有一个线程访问到共享变量，就不会出现线程线程上下文切换导致的问题。换句话说，保证了临界区代码的原子性。

6.4、变量的线程安全分析

1、成员变量和静态变量的线程安全分析

  • 如果没有变量没有在线程间共享，那么变量是安全的
  • 如果变量在线程间共享
    • 如果只有读操作，则线程安全
    • 如果有读写操作，则这段代码是临界区，需要考虑线程安全

2、局部变量线程安全分析

  • 局部变量【局部变量被初始化为基本数据类型】是安全的
  • 局部变量引用的对象未必是安全的
    • 如果局部变量引用的对象没有引用线程共享的对象，那么是线程安全的
    • 如果局部变量引用的对象引用了一个线程共享的对象，那么要考虑线程安全的

6.6、不可变类的线程安全

  • 类：String和Integer等类都是不可变的类，因为其类内部状态是不可改变的，因此它们的方法都是线程安全的。

  • 对象：如使用final定义的，部分比枚举类成员，都是线程安全的

  • 原因：对象的引用是原子性的，比如String的类的方法，基本都是创建新对象然后引用，不同线程创建的对象不同，而且引用操作时原子性，所以没有线程安全问题。

7、活跃性问题

7.1、死锁

产生自锁的条件

  1. 互斥条件：该资源任意⼀个时刻只由⼀个线程占⽤。

  2. 请求与保持条件：⼀个进程因请求资源⽽阻塞时，对已获得的资源保持不放。

  3. 不剥夺条件：线程已获得的资源在末使⽤完之前不能被其他线程强行剥夺，只有自己使用完毕后才释放资源。

  4. 循环等待条件：若干进程之间形成⼀种头尾相接的循环等待资源关系。

原因：两个线程互相占用对方需要的资源。

避免死锁

  • 破坏互斥条件 ：无法改变，违背了加锁的初衷（临界资源需要互斥访问）。
  • 破坏请求与保持条件 ：⼀次性申请所有的资源（锁粒度太大会影响性能）
  • 破坏不剥夺条件 ：某个线程获取资源失败，主动释放它占有的资源（比如 wait 释放锁）
  • 破坏循环等待条件 ：按照 资源 1 、资源 2 顺序申请 ，这样就不会出现 线程A 占用资源1 需要2 ，线程2 占用 2需要1 导致死锁的情况发生

7.2、活锁

两个线程互相改变对方的结束状态，导致谁也无法结束

避免活锁：让两个线程执行的实际错开，比如加上睡眠时间。

7.3、线程饥饿

普通锁：一个线程由于优先级太低等原因，始终得不到 CPU 调度执行，也不能够结束

读写锁：由于读、写互斥，所以在大量读的情况下，写操作往往很难得到CPU调度执行。

9、总结

  1. 分析多线程访问共享资源时，哪些代码片段属于临界区
  2. 使用 synchronized 互斥解决临界区的线程安全问题
    1. 掌握 synchronized 锁对象语法
    2. 掌握 synchronzied 加载成员方法和静态方法语法
    3. 掌握 wait/notify 同步方法
  3. 使用 lock 互斥解决临界区的线程安全问题
    掌握 lock 的使用细节：可打断、锁超时、公平锁、条件变量
  4. 学会分析变量的线程安全性、掌握常见线程安全类的使用
  5. 了解线程活跃性问题：死锁、活锁、饥饿
  6. 应用方面
    1. 互斥：使用 synchronized 或 Lock 达到共享资源互斥效果，实现原子性效果，保证线程安全。
    2. 同步：使用 wait/notify 或 Lock 的条件变量来达到线程间通信效果。
  7. 原理方面
    1. monitor、synchronized 、wait/notify 原理
    2. synchronized 进阶原理
    3. park & unpark 原理
  8. 模式方面
    1. 同步模式之保护性暂停
    2. 异步模式之生产者消费者
