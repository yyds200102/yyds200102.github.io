3、自动配置原理

3.1、引导加载自动配置类

@SpringBootApplication注解剖析：里面有3个主要的注解

@SpringBootConfiguration

作用：@Configuration 表示当前是一个配置类，表明启动类本身也是Spring中的一个组件而已.负责启动组件而已

@ComponentScan

作用：自动扫描并加载符合条件的组件或者bean ， 将这个bean定义加载到IOC容器中

@EnableAutoConfiguration 核心

1、@AutoConfigurationPackage：自动扫描包下的组件并注册名字

**2、@Import(AutoConfigurationImportSelector.class)**：加载全部的自动配置类

1、利用getAutoConfigurationEntry(annotationMetadata)；给容器中批量导入一些组件

2、调用List< String > configurations = getCandidateConfigurations(annotationMetadata, attributes)获取到所有需要导入到容器中的配置类

3、利用工厂加载 Map<String, List< String >> loadSpringFactories(@Nullable ClassLoader classLoader)；得到所有的组件

4、从META-INF/spring.factories位置来加载一个文件。 默认扫描我们当前系统里面所有META-INF/spring.factories位置的文件 spring-boot-autoconfigure-2.3.4.RELEASE.jar包里面也有META-INF/spring.factories

Spring.factories 里面有各种AutoConfiguration 类

xxxAutoConfiguration配置类：其实就是一个配置类，通过读取 xxxproperties里面的信息，实例化Bean交给容器，这样就完成了容器的自动装配。

3.2、按需开启自动配置项

虽然我们127个场景的所有自动配置启动的时候默认全部加载。xxxxAutoConfiguration 按照条件装配规则（@Conditionalxxx），最终会按需选择是否生效。

3.3、修改默认配置

SpringBoot默认会在底层配好所有的组件。但是如果用户自己配置了则以用户的优先

在 @Configuration 配置类中 使用 @Bean 给容器托管，会替代系统的Bean

3.4、总结

注解的作用：扫描包下的所有文件、表明把自己注入容器、表明要加载Spring.factories的所有的配置类

  1. SpringBoot在启动的时候从类路径下的META-INF/spring.factories中获取各种自动配置类即EnableAutoConfiguration指定的值
  2. 每个自动配置类按照条件（用户是否自定义了、@Condition的条件是否满足）来选择是否生效（即通过反射调用方法生成Bean），只要容器中有这些组件（Bean），相当于这些功能就有了。
  3. 默认都会绑定配置文件xxxxProperties指定的值。优先从用户自定义的配置文件拿，没有再从默认配置的xxxxProperties里面拿。

定制化配置

  1. 用户直接自己@Bean替换底层的组件
  2. 用户去看这个组件是获取的配置文件什么值就去修改。

xxxxxAutoConfiguration —> 组件 —> xxxxProperties里面拿值 —-> application.properties

返回值解析器

返回值处理器解析原理

  1. 返回值处理器判断是否支持这种类型返回值，根据 supportsReturnType 方法判断。
  2. 如果支持，返回值处理器调用 handleReturnValue 进行处理
  3. RequestResponseBodyMethodProcessor类进行具体的逻辑： 利用 MessageConverters 消息转换器处理返回值标了@ResponseBody 注解的， 将数据写为json
    • 内容协商（浏览器默认会以请求头的方式告诉服务器他能接受什么样的内容类型）
    • 服务器最终根据自己自身的能力，决定服务器能生产出什么样内容类型的数据，
    • SpringMVC会挨个遍历所有容器底层的 HttpMessageConverter ，看谁能处理？
    • 得到MappingJackson2HttpMessageConverter 消息转换器可以将对象写为json

支持的返回值类型

MessageConverters 消息转换器

HttpMessageConverter: 看是否支持将 此 Class类型的对象，转为MediaType类型的数据，最终 MappingJackson2HttpMessageConverter 把对象转为JSON（利用底层的jackson的objectMapper转换的）

2、内容协商

内容：根据客户端接收能力不同，返回不同媒体类型的数据。

做法：只需要改变请求头中Accept字段。Http协议中规定的，告诉服务器本客户端可以接收的数据类型。

案例：实现支持返回类型转换为xml

1、引入支持XML响应类型的依赖

2、开启基于请求参数的内容协商功能：

作用：系统会根据浏览器接收能力不同，返回不同的媒体类型的数据，如application/json;客户端能接收的内容类型一般会再响应头当中accept中

测试：开始之后，只要再请求后面携带参数，加 format = xml ，响应类型为xml，只支持XML和JSON

内容协商的具体流程

  1. 判断当前响应头中是否已经有确定的媒体类型。MediaType
  2. 获取客户端（PostMan、浏览器）支持接收的内容类型。（获取客户端Accept请求头字段）
    • contentNegotiationManager 内容协商管理器 默认使用基于请求头的策略
    • HeaderContentNegotiationStrategy 确定客户端可以接收的内容类型
  3. 遍历循环所有当前系统的 MessageConverter，看谁支持操作这个对象（Person）
  4. 找到支持操作Person的 MessageConverter，把converter支持的媒体类型统计出来。会发现如客户端需要【application/xml】。服务端能力【10种、json、xml】
  5. 进行内容协商选择最佳匹配媒体类型
  6. 用 支持 将对象转为 最佳匹配媒体类型 的 MessageConverter 。调用它进行转化 。
  7. 返回数据给客户端

3、自定义MessageConverters 消息转换器

5.8、拦截器

原理

1、根据当前请求，找到HandlerExecutionChain【可以处理请求的handler以及handler的所有 拦截器】

2、先来顺序执行 所有拦截器的 preHandle方法

  • 如果当前拦截器prehandler返回为true。则执行下一个拦截器的preHandle
  • 如果当前拦截器返回为false。直接 倒序执行所有已经执行了的拦截器的 afterCompletion；

3、如果任何一个拦截器返回false。直接跳出不执行目标方法

4、所有拦截器都返回True。执行目标方法

5、倒序执行所有拦截器的postHandle方法。

6、前面的步骤有任何异常都会直接倒序触发 afterCompletion

7、页面成功渲染完成以后，也会倒序触发 afterCompletion

2、原理

  • SpringBoot应用启动发现当前是Web应用（发现导入了web场景包-导入tomcat）

  • web应用就会创建一个web版的ioc容器 ServletWebServerApplicationContext

  • ServletWebServerApplicationContext 启动的时候寻找 ServletWebServerFactory （Servlet 的web服务器工厂---> Servlet 的web服务器） ；SpringBoot底层默认有很多的WebServer工厂；TomcatServletWebServerFactory, Jettyxxxx，,Undertowxxxx，只要导入了对应的包，就能够创建出来。

  • 底层直接会有一个自动配置类：ServletWebServerFactoryAutoConfiguration，该类会自动导入配置类ServletWebServerFactoryConfiguration，会动态判断系统导入的servlet容器包，并且创建具体的容器并且启动。默认是TomcatServletWebServerFactory 创建出Tomcat服务器并启动；TomcatWebServer 的构造器拥有初始化方法initialize---this.tomcat.start();

  • 内嵌服务器，就是手动把启动服务器的代码调用（tomcat核心jar包存在）

原理：

  1. WebMvcAutoConfiguration 默认的SpringMVC的自动配置功能类。静态资源、欢迎页…..
  2. 一旦使用 @EnableWebMvc 会 @Import(DelegatingWebMvcConfiguration.class)，只保证SpringMVC核心组件。
  3. DelegatingWebMvcConfiguration 作用：
    • 会把所有系统中的 WebMvcConfigurer 都拿过来一起生效。
    • 自动配置了一些非常底层的组件。比如RequestMappingHandlerMapping、这些组件依赖的组件都是从容器中获取
  4. WebMvcAutoConfiguration 里面的配置要能生效 ，必须 @ConditionalOnMissingBean ( WebMvcConfigurationSupport**.**class)
  5. @EnableWebMvc 导入的 DelegatingWebMvcConfiguration 是 WebMvcConfigurationSupport的 父类，所以会导致了 WebMvcAutoConfiguration 没有生效。
