JVM(一):内存结构

发表于2021-06-22|更新于2021-06-22|Java
|字数总计:3.3k|阅读时长:11分钟|阅读量:

1、程序计数器

  • 栈、本地方法栈、程序计数器100%不会有垃圾
  • JVM调优99%在调整堆(方法区属于特殊的堆)
  • 一般插件都是在执行引擎上动手脚

Program Counter Register 程序计数器（寄存器）

作用：

  • 每条JVM指令都会交给JVM解释器，解释器把字节码变成机器码，让CPU执行，然后此时下一条 jvm 指令的执行地址行号会被放入程序计数器中，然后解释器根据地址去取指令并执行。
  • 多线程的环境下，如果两个线程发生了上下文切换，那么程序计数器会记录线程下一条指令的地址行号，以便线程切换回来继续执行命令。
  • 总而言之，程序计数器就是存放JVM下一条指令的地址，特殊的：执行本地方法时，无法定位地址

特点：

  • 是线程私有的，每个程序有自己的程序计数器
  • JVM中唯一一个不会存在内存溢出

实现方式：

  • 通过寄存器来实现的

2、虚拟机栈（线程运行时需要的内存空间）

2.1、定义

概念：

  • 每个线程运行需要的内存空间，称为虚拟机栈，栈是**私有**的
  • 每个栈由多个栈帧（Frame）组成，对应着每次调用方法时所占用的内存（参数、变量等）
  • 每个线程只能有一个活动栈帧，对应着当前正在执行的方法，每⼀次函数调用都会有⼀个对应的栈帧被压⼊ Java 栈，每⼀个函数调⽤结束后（或抛出异常导致结束），都会有⼀个栈帧被弹出
  • 先进后出，main方法最先执行，最晚结束

栈帧的信息：局部变量表(编译期可知的各种基本数据类型、对象引用)、操作数栈、动态链接、⽅法出⼝信息。

2.2、几个问题

  1. 垃圾回收是否涉及栈内存？

    • 不会。栈内存是方法调用产生的，方法调用结束后会弹出栈。

  2. 栈内存分配越大越好吗？

    • 不是，栈内存越大，每个线程的栈就越大，但是可执行的线程数就会越少。

  3. 方法内的局部变量是否线程安全？

    • 如果方法内部的变量没有逃离方法的作用访问，它是线程安全的

    • 如果是局部变量引用了对象，并逃离了方法的访问（其他线程也能访问到），就要考虑线程安全

2.3、线程溢出

  • 栈帧太多（如无限递归，java.lang.StackOverflowError）
  • 栈帧太大（一般不太可能发生）

2.4、线程运行诊断

案例一：cpu 占用过多

描述：Linux 环境下运行某些程序的时候，可能导致 CPU 的占用过高，这时需要定位占用 CPU 过高的线程

  1. 输入 top 命令，查看是哪个进程占用 CPU 过高
  2. 输入 ps H -eo pid, tid, %cpu | grep 加进程号（刚刚的进程号）可以过滤进程
  3. 输入 jstack 进程 id ，即可显示java的线程信息
  4. 刚才通过 ps 命令看到的 tid 来对比定位，注意 jstack 查找出的线程 id 是 16 进制的，需要转换。

案例二：线程很长时间未响应

同上，排查异常情况

2.5、本地方法栈

调用本地方法时需要使用本地方法栈

本地方法：native修饰的方法，底层调用的是C/C++方法等

3、堆

3.1、定义：

Heap 堆

  • 通过new关键字创建的对象都会被放在堆中，在堆里面分配内存。

特点

  • 它是所有线程共享，堆内存中的对象都需要考虑线程安全问题
  • 有垃圾回收机制

注意：从jdk 1.7开始已经默认开启逃逸分析，如果某些⽅法中的对象引⽤没有被返回或者未被外⾯使⽤（也就是未逃逸出去），那么对象可以直接在栈上分配内存。

3.2、堆内存溢出

OOM错误：

  • java.lang.OutOfMemoryError: Java heap space 堆内存满了，堆内存不够
  • 即 新生区 + 老年区 大小等于总堆内存

默认情况：JVM分配总内存为电脑四分之一，初始化的内存为六十四分之一

修改堆空间大小：-Xmx1024m 指定大小

-Xms1024m -Xmx1024m -XX:+PrintGCDetails

3.3、堆内存诊断

  1. jps 工具：查看当前系统中有哪些 java 进程
  2. jmap 工具：查看堆内存占用情况 jmap - heap 进程id
  3. jconsole 工具：图形界面的，多功能的监测工具，可以连续监测
  4. jvisualvm 工具：

3.4、垃圾回收

垃圾回收主要发生在堆里，因为几乎所有的对象都在堆中分配。

4、方法区：

4.1、定义

  1. 方法区被所有 Java 虚拟机线程之间**共享**的方法区域。

  2. 它存储每个类的结构，常量、静态变量、即时编译器编译后的代码等数据

  3. 方法区域是在**虚拟机启动时**创建的。

  4. 尽管方法区域在逻辑上是堆的一部分，但是并不强制要求，而且名字也叫做非堆。

  5. HotSpot（其他不一定）使用永久代实现方法区，在JDK1.8时移除，采用**元空间**实现（不再占用堆的空间，在本地内存占用）

  6. 也存在垃圾回收，但是较少可以忽略

4.2、方法区内存溢出

  • 1.8 之前会导致**永久代**内存溢出
    • 使用 -XX:MaxPermSize=8m 指定永久代内存大小
  • 1.8 之后会导致**元空间**内存溢出
    • 使用 -XX:MaxMetaspaceSize=8m 指定元空间大小

4.3、各种常量池剖析

常量池：每个class文件都有类的基本信息，常量池、类方法定义、虚拟机指令等信息，常量池用于存放编译期间生成的各种字面量和符号引用。

运行时常量池：程序运行到某个类时，即类加载到方法区时，Class文件的信息就会被解析到内存的方法区中的运行时常量池，因此每个类都有一个运行时常量池。加载之后，运行时常量池自然获得了常量池的各种信息。

字符串常量池：JDK7之前在方法区的运行时常量池中，JDK7开始被单独拿到堆；当类加载的时候，常量池的字符串文本就会加载进入字符串常量池中

原因：方法区的永久代回收要等到FullGC，频率太低，所以容易导致内存不够用。

注意：

  • 运行时常量池自始至终在方法区，JDK7在方法区（堆），JDK8在元空间
  • 字符串常量池JDK7之前在方法区的运行时常量池，JDK7开始单独放到堆中

4.4、StringTable字符串常量池

存储

  • 字符串用**哈希表**存储
  • 常量池的信息加载到字符串常量池，不过开始仅是符号，**只有用到时才会转化为对象**

创建字符串：

  • 字面量创建字符串时，对象指向的是字符串常量池里的对象

  • new方法创建或者拼接字符串变量时，对象指向的是堆里的对象

字符串拼接：

  • 字符串变量拼接的原理是**StringBuilder**，所以结果在堆里
  • 字符串常量拼接的原理是**编译器优化**，所以结果在常量池里

intern方法（1.8）：假设 字符串str调用

  • 若字符串常量池中没有a，就会**添加到常量池**中，返回常量池的对象
  • 若字符串常量池中有a，就会**添加失败**，不过仍然返回常量池的对象

intern方法（1.6）：假设 字符串str调用

  • 添加成功，会创建新对象放入常量池

    注意：intern不会改变原对象，只是会返回常量池的对象

4.5、StringTable垃圾回收

-Xmx10m 指定堆内存大小
-XX:+PrintStringTableStatistics 打印字符串常量池信息
-XX:+PrintGCDetails
-verbose:gc 打印 gc 的次数，耗费时间等信息

4.6、StringTable性能调优

一：上面说过StringTable是由HashTable实现的，所以可以适当增加HashTable桶的个数，来减少字符串放入串池所需要的时间

二：可以通过 intern 方法减少重复入池

5、直接内存

5.1、定义

  • 常见于 NIO 操作时，用于数据缓冲区
  • 分配回收成本较高，但读写性能高
  • 由系统直接管理，不受 JVM 内存回收管理，但是仍然可能导致OOM

5.2、直接内存的优点

文件读写流程

因为 java 不能直接操作文件管理，需要切换到内核态，使用本地方法进行操作，然后读取磁盘文件，会在系统内存中创建一个缓冲区，将数据读到系统缓冲区， 然后在将系统缓冲区数据，复制到 java 堆内存中。缺点是数据存储了两份，在系统内存中有一份，java 堆中有一份，造成了不必要的复制

使用了 DirectBuffer 文件读取流程

直接内存是操作系统和 Java 代码**都可以访问**的一块区域，无需将代码从系统内存复制到 Java 堆内存，从而提高了效率

5.3、直接内存的内存回收

  • 使用了 Unsafe 类来完成直接内存的分配回收，回收需要主动调用freeMemory 方法
  • ByteBuffer 的实现内部使用了 Cleaner（虚引用）来检测 ByteBuffer 。一旦ByteBuffer 被垃圾回收，那么会由 ReferenceHandler（守护线程） 来调用 Cleaner 的 clean 方法调用 freeMemory 来释放内存
