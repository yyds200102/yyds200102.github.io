JVM(一):内存结构

发表于2021-06-22|更新于2021-06-22|Java
|字数总计:5.5k|阅读时长:18分钟|阅读量:

1、程序计数器

  • 栈、本地方法栈、程序计数器100%不会有垃圾
  • JVM调优99%在调整堆(方法区属于特殊的堆)
  • 一般插件都是在执行引擎上动手脚

Program Counter Register 程序计数器（寄存器）

作用：存放JVM当前执行指令的地址（本地方法无法定位地址为空）

  • 每条JVM指令都会交给JVM解释器，解释器把字节码变成机器码，让CPU执行，然后此时下一条 jvm 指令的执行地址行号会被放入程序计数器中，然后解释器根据地址去取指令并执行。
  • 多线程的环境下，如果两个线程发生了上下文切换，那么程序计数器会记录线程下一条指令的地址行号，以便线程切换回来继续执行命令。
  • 字节码解释器通过改变程序计数器来依次读取指令，从⽽实现代码的流程控制，如：顺序执行、选择、循环、异常处理。

特点：

  • 是线程私有的，每个程序有自己的程序计数器
  • JVM中唯一一个不会存在内存溢出，即没有 OOM 的区域

实现方式：

  • 通过寄存器来实现的

2、虚拟机栈（线程运行时需要的内存空间）

2.1、定义

栈的概念

  • 每个线程运行需要的内存空间，称为虚拟机栈，栈是线程私有的（保护局部变量）；每个栈由多个栈帧（Frame）组成，对应着每次调用方法时所占用的内存（参数、变量等）
  • 每个线程只能有一个活动栈帧，对应着当前正在执行的方法，每⼀次函数调用都会有⼀个对应的栈帧被压⼊ Java 栈，每⼀个函数调用结束后（或抛出异常导致结束），都会有⼀个栈帧被弹出
  • 先进后出，main方法最先执行，最晚结束
  • 内存大小是否能够扩展由具体的JVM决定

栈帧的信息

  • 局部变量表（编译期可知的各种基本数据类型、对象引用（不一定是对象起始地址）、下一条字节码指令地址)；除了Long Double 2个字节，其他都是1字节
  • 操作数栈、动态链接、方法出口信息。

2.2、几个问题

  1. 垃圾回收是否涉及栈内存？

    • 不会。栈内存是方法调用产生的，方法调用结束后会弹出栈。

  2. 栈内存分配越大越好吗？

    • 不是，栈内存越大，每个线程的栈就越大，但是可执行的线程数就会越少。线程初始会申请栈内存，内存不足会OOM。

  3. 方法内的局部变量是否线程安全？

    • 如果方法内部的变量没有逃离方法的作用访问，它是线程安全的

    • 如果是局部变量引用了对象，并逃离了方法的访问（其他线程也能访问到），就要考虑线程安全

2.3、内存溢出

  • SOF：栈帧太多超过限制；或者不允许动态扩展的JVM新栈帧没内存分配。
  • OOM：线程初次申请栈内存，内存不够分；或者允许动态扩展的JVM当新栈帧没内存分配时会申请扩展栈内存，如果栈内存不足，会OOM。

注意：HotSpot不允许栈扩展；堆一般都可以扩展，从参数就可以看出。

2.4、栈的内存回收

当程序执行关闭，栈帧会被弹出，所以不涉及垃圾回收。

2.5、线程运行诊断

案例一：cpu 占用过多

描述：Linux 环境下运行某些程序的时候，可能导致 CPU 的占用过高，这时需要定位占用 CPU 过高的线程

  1. 输入 top 命令，查看是哪个进程占用 CPU 过高
  2. 输入 ps H -eo pid, tid, %cpu | grep 加进程号（刚刚的进程号PI即D）可以过滤进程
  3. 输入 jstack 进程 id ，即可显示java的线程信息
  4. 刚才通过 ps 命令看到的 tid 来对比定位，注意 jstack 查找出的线程 id 是 16 进制的，需要转换。

案例二：线程很长时间未响应

同上，排查异常情况

2.6、本地方法栈

调用本地方法时需要使用本地方法栈

本地方法：native修饰的方法，底层调用的是C/C++方法等

错误：同样有OSF和OOM

3、堆

3.1、定义：

Heap 堆：new关键字创建的对象和数组；类的Class对象 都存储在这里

特点

  • 它是所有线程共享，堆内存中的对象都需要考虑线程安全问题
  • 逻辑上包括方法区，内存连续，但是实际不一定
  • 有垃圾回收机制
  • 可以动态扩展

注意：随着JIT编译期的发展与逃逸分析技术逐渐成熟，栈上分配、标量替换优化技术将会导致⼀些微妙的变化，所有的对象都分配到堆上也渐渐变得不那么绝对了。从jdk 1.7开始已经默认开启逃逸分析，如果某些方法中的对象引用没有被返回或者未被外面使用（也就是未逃逸出去），那么对象可以直接在栈上分配内存。

3.2、线程私有区域

概念：堆中的线程私有缓存区域 TLAB（Thread Local Allocation Buffer)

原因：并发线程下对象分配可能会产生问题，比如地址重复分配。

作用：每个线程有字节的 私有缓存区域TLAB，优先在此处分配内存，如果内存分配完了，JVM就会尝试着通过使用加锁机制确保数据操作的原子性，从而直接在Eden空间中分配内存

3.3、堆内存溢出

OOM错误：

  • java.lang.OutOfMemoryError: Java heap space 堆内存满了，堆内存不够
  • 即 新生区 + 老年区 大小等于总堆内存

默认情况：JVM分配总内存为电脑四分之一，初始化的内存为六十四分之一

修改堆空间大小：-Xmx1024m 指定最大大小 -Xms512m 指定初始大小

-Xms1024m -Xmx1024m -XX:+PrintGCDetails

3.4、堆内存诊断

  1. JPS + JMap
    • jps 工具：查看当前系统中有哪些 java 进程 直接jps
    • jmap 工具：查看堆内存占用情况 jmap - heap 进程id
  2. jconsole 工具：图形界面的，多功能的监测工具，可以连续监测 jconsole
  3. jvisualvm 工具：

案例：垃圾回收后，内存占用仍然很高

  • 使用jps + jmap 发现 占用内存很高
  • 使用 jconsole 进行 Gc 回收之后，发现老年代仍然占用很高
  • 使用 jvisualvm 命令进入，监视里面点击堆dump，点击右边的查找，找出占用内存大的应用，点击对象然后查看对象的情况

3.5、垃圾回收

第二节详细介绍

4、方法区：

4.1、定义

  1. 方法区被所有 Java 虚拟机线程之间共享的方法区域，在虚拟机启动时创建的。

  2. 尽管方法区域在逻辑上是堆的一部分，但是并不强制要求，而且名字也叫做非堆。

  3. 方法区的大小和堆一样可以动态扩展

  4. HotSpot（其他不一定）使用永久代实现方法区，在JDK1.8时移除，采用元空间实现（不再占用堆的空间，在本地内存占用）

存储的数据结构

  • 已经被虚拟机记载的类的信息
  • 常量
  • 静态变量：JDK1.7之后被存放到Class对象里面，在堆中
  • 即时编译器编译后的代码
  • 运行时常量池等

为什么要将永久代 (PermGen) 替换为元空间 (MetaSpace) 呢

  1. 永久代使用堆内存，元空间使用的是操作系统内存，OOM概率更更小。
  2. 元空间里面存放的是类的元数据，很少垃圾回收而又必须存在，所以放在直接内存更好。
  3. 永久代本来就是逻辑上的一个东西，只是换一个叫法。

4.2、垃圾回收

JVM规范不要求实现方法区的垃圾回收，但是也可以实现，以下是HotSpot的实现

回收时机

  • 永久代：内存大小到达-XX:PermSize 时触发 FullGC、Full GC 时触发垃圾回收。
  • 元空间：内存大小到达-XX:MetaspaceSize时触发 FullGC、Full GC 时触发垃圾回收。

注意：这些都是标准值，JVM会根据回收时的标准值大小动态跳转允许的最大值；为了减少 Full GC 不能把标准值调整的过低。

回收内容

  • 常量回收：对象没有被引用则会被回收。
  • 类的卸载：无用的类会被允许回收，注意不是一定。
    • 所有实例都被回收
    • 加载该类的 ClassLoader 已经被回收。
    • 该类的Class对象没有在任何地方被引用，⽆法在任何地方通过反射访问该类的方法。（注意Class对象被JVM引用，永远不会被回收）

方法区内存溢出

  • JDK1.6，可以不断的创建字符串，因为字符串常量池在方法区中，所以 永久代 会OOM
  • JDK1.7，字符串常量池被移动到堆中，想 永久代OOM 必须大量创建字节码类，比如使用 Cglib
  • JDK1.8，想要元空间OOM 可以调小元空间的参数值。

4.3、常量池

常量池：每个class文件都有类的版本、字段、方法、接口等描述信息，除了这些之外，还有常量池表。常量池用于存放编译期间生成的各种字面量和符号引用。

运行时常量池：当一个类被加载时，就会为其产生对应的运行时常量池。其中存放常量池的信息，以及程序执行期间动态生成的常量（intern()方法）。运行时常量池是方法区的一部分，存在OOM的可能。运行时常量池逻辑包含字符串常量池。

字符串常量池：JDK7之前在方法区的运行时常量池中，JDK7开始被单独拿到堆；当类加载的时候，常量池的字符串文本就会加载进入字符串常量池中。

注意

  • 运行时常量池自始至终在方法区，JDK7在方法区（堆），JDK8在元空间
  • 字符串常量池JDK7之前在方法区的运行时常量池，JDK7开始单独放到堆中。因为方法区的永久代回收要等到FullGC，频率太低，所以容易导致内存不够用。

4.4、StringTable字符串常量池

存储

  • 字符串用**哈希表**存储
  • 常量池的信息加载到字符串常量池，不过开始仅是符号，**只有用到时才会转化为对象**

创建字符串：

  • 字面量创建字符串时，对象指向的是字符串常量池里的对象

  • new方法创建或者拼接字符串变量时，对象指向的是堆里的对象

字符串拼接：

  • 字符串变量拼接的原理是**StringBuilder**，所以结果在堆里
  • 字符串常量拼接的原理是**编译器优化**，所以结果在常量池里

延迟初始化：

intern方法（1.7）：假设 字符串str调用

  • 若字符串常量池中没有a，就会直接添加到常量池中（因为常量池就在堆中），返回常量池的对象
  • 若字符串常量池中有a，就会**添加失败**，不过仍然返回常量池的对象

intern方法（1.6）：假设 字符串str调用

  • 若字符串常量池中没有a，就会把对象复制一份放入常量池，返回常量池的对象

  • 如果有该字符串对象，则放入失败，返回常量池的对象

    注意：无论是否成功，堆和常量池的都不会是同一个对象。

4.5、StringTable垃圾回收

-Xmx10m 指定堆内存大小
-XX:+PrintStringTableStatistics 打印字符串常量池信息
-XX:+PrintGCDetails
-verbose:gc 打印 gc 的次数，耗费时间等信息

4.6、StringTable性能调优

一：上面说过StringTable是由Hash表实现的，所以可以适当增加HashTable桶的个数，来减少字符串放入串池所需要的时间

二：通过 intern 方法

调用intern方法，获取到串池中的对象，这样堆内的被回收掉，可以减少内存。

5、直接内存

5.1、定义

  • 常见于 NIO 操作时，用于数据缓冲区
  • 分配回收成本较高，但读写性能高
  • 由系统直接管理，不受 JVM 内存回收管理，但是仍然可能导致OOM
  • 默认与Java堆内存一致，也可以999999999999999999999999999999999999999999999自己指定

5.2、直接内存的优点

文件读写流程

因为 java 不能直接操作文件管理，所以需要切换到内核态，使用本地方法进行操作，然后读取磁盘文件，会在系统内存中创建一个缓冲区，将数据读到系统缓冲区， 然后在将系统缓冲区数据，复制到 java 堆内存中。缺点是数据存储了两份，在系统内存中有一份，java 堆中有一份，造成了不必要的复制

使用了 DirectBuffer 文件读取流程

直接内存是操作系统和 Java 代码**都可以访问**的一块区域，无需将代码从系统内存复制到 Java 堆内存，从而提高了效率

5.3、直接内存的内存回收

释放原理总结

  • 使用了 Unsafe 类来完成直接内存的分配回收，回收需要主动调用freeMemory 方法（JDK帮忙做了）
  • ByteBuffer 的实现内部使用了 Cleaner（虚引用）来检测 ByteBuffer 。一旦ByteBuffer 被垃圾回收，那么会由 ReferenceHandler（守护线程） 来调用 Cleaner 的 clean 方法调用 freeMemory 来释放内存

OOM

直接内存并不是虚拟机运行时数据区的⼀部分，也不是虚拟机规范中定义的内存区域，但是这部分内存也被频繁地使用。而且也可能导致OutOfMemoryError 错误出现。

注意：JVM会在申请前计算够不够分配，不够抛出OOM，够了才会申请，

6、对象创建的过程

6.1、类加载检查

虚拟机遇到⼀条 new 指令时，⾸先将去检查这个指令的参数是否能在常量池中定位到这个类的符号引⽤，并且检查这个符号引⽤代表的类是否已被加载过、解析和初始化过。如果没有，那必须先执行相应的类加载过程。

注意：在类加载的初始化阶段，为一些成员变量初始化时可能会触发此过程。

6.2、分配内存

虚拟机将为新生对象分配内存。对象所需的内存大小类加载完成后便可确定，为对象分配空间的任务等同于把⼀块确定大小的内存从 Java 堆中划分出来。有两种分配方式，根据Java内存是否整齐来决定，取决于GC 收集器的算法是”标记-清除”，还是”标记-整理”（也称作”标记-压缩”），值得注意的是，复制算法内存也是规整的。

  • 指针碰撞：整理内存，把没有的内存整理到一起，再分配内存
  • 空闲列表：虚拟机维护一个列表，记录哪些位置可用，分配内存

内存分配并发问题

作为虚拟机来说，必须要保证线程是安全的，否则可能出现内存被多次分配的情况，通常来讲，虚拟机采用两种方式一起来保证线程安全

  • TLAB本地线程分配缓冲：为每⼀个线程预先在 Eden 区分配⼀块内存，JVM 在给线程中的对象分配内存时，首先先在 TLAB 分配，当对象大于 TLAB 中的剩余内存或 TLAB 的内存已用尽时，使用上述的 CAS 进行内存分配
  • CAS + 失败重试

6.3、初始化零值

对于基本类型需要初始化零值或者false等。

6.4、设计对象头

通过Class对象获取方法区中该类的元信息，如对象的哈希码、对象的 GC 分代年龄、偏向锁信息等，这些信息存放在对象头中。

6.5、执行init方法

执行对象的 < init > 方法 按照程序员的意愿进行初始化（和第三节的一致）。

扩展一：对象访问的定位方式

对象的访问方式有虚拟机实现而定，⽬前主流的访问方式有使用句柄、直接指针两种：

  • 句柄：Java堆中将会划分出⼀块内存来作为句柄池，reference 中存储的就是对象的句柄地址，而句柄中包含了对象实例数据与类型数据各自的具体地址信息。
  • 直接指针： Java 堆对象的布局中就必须考虑如何放置访问类型数据的相关信息，而reference 中存储的直接就是对象的地址。

总结：一般情况访问对象更频繁，而使用句柄需要多一次转发操作，所以HotSpot采用第二种方式。

  • 句柄 ———–> 找到句柄池 ———-> 找到对象实例、类信息（方法区）
  • 直接指针 ————-> 找到对象实例 ———————> 找到类信息（方法区）
