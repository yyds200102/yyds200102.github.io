1、判断对象可以回收

1.1、引用计数法

当一个对象被引用时，就当引用对象的值加一，当值为 0 时，就表示该对象不被引用，可以被垃圾收集器回收。

缺陷：循环引用时，两个对象的计数都为1，导致两个对象都无法被释放。

1.2、可达性分析算法

扫描堆中的对象，看能否沿着 GC Root 对象为起点的引用链找到该对象，如果找不到，则表示可以回收

GC Root 对象：

  • 虚拟机栈（栈帧中的本地变量表）中引用的对象。
  • 方法区中类静态属性引用的对象
  • 方法区中常量引用的对象
  • 本地方法栈中 JNI（即一般说的Native方法）引用的对象

1.3、如何判断⼀个常量是废弃常量

假如在常量池中存在字符串 “abc”，如果当前没有任何String对象引用该字符串常量的话，就说明常量 “abc” 就是废弃常量，如果这时发生内存回收的话而且有必要的话，”abc” 就会被系统清理出常量池。

1.4、 如何判断⼀个类是无用的类

满足下面3个条件，可能被回收，不是一定

  • 该类所有的实例都已经被回收，也就是 Java 堆中不存在该类的任何实例。
  • 加载该类的 ClassLoader 已经被回收。
  • 该类对应的 java.lang.Class 对象没有在任何地⽅被引⽤，⽆法在任何地⽅通过反射访问该类的方法。

1.5、四种引用

  1. 强引用：宁愿抛出OOM异常也不会被回收，除非不再被强引用
  2. 软引用（SoftReference）：在垃圾回收后，内存仍不足时会回收软引用引用的对象。
  3. 弱引用（WeakReference）：在垃圾回收时，无论内存是否充足，都会回收弱引用引用的对象。
  4. 虚引用（PhantomReference）：当虚引用对象所引用的对象被回收以后，虚引用对象就会被放入引用队列中，调用虚引用的方法，虚引用的一个体现是释放直接内存所分配的内存，当引用的对象ByteBuffer被垃圾回收以后，虚引用对象Cleaner就会被放入引用队列中，然后调用Cleaner的clean方法来释放直接内存

注意点：

  • 软引用、弱引用也可以和⼀个引用队列（ReferenceQueue）联合使用，在弱引用和虚引用所引用的对象被回收以后，会将这些引用放入引用队列中，方便一起回收这些软/弱引用对象。
  • 虚引用和终结器引用必须配合引用队列，虚引用和终结器引用在使用时会关联一个引用队列。
  • 在程序设计中⼀般很少使用弱引用与虚引用，使用软引用的情况多，这是因为软引用可以加速JVM对垃圾内存的回收速度，可以维护系统的运行安全，防止内存溢出OOM等问题的产生。

弱引用案例

弱引用的使用和软引用类似，只是将 SoftReference 换为了 WeakReference

扩展

终结器引用（FinalReference）：所有的类都继承自Object类，Object类有一个finalize方法。当某个对象不再被其他的对象所引用时，会先将终结器引用对象放入引用队列中，然后根据终结器引用对象找到它所引用的对象，然后调用该对象的finalize方法。调用以后，该对象就可以被垃圾回收了。

2、垃圾回收算法

2.1、标记清除算法

  1. 标记：把需要清除的垃圾标记起来
  2. 清除：定义一张表，记录垃圾占用空间的首地址和尾地址，这些空间下次就可以继续分配

优点：速度快

缺点：会产生内存碎片，不连续的内存导致无法给大对象进行内存分配，进而导致频繁进行GC，影响性能。

2.2、标记整理

  1. 标记：把需要清除的垃圾标记起来
  2. 整理：把垃圾清除后，再把剩余的空间整理，确保没有内存碎片

缺点：速度慢

优点：没有内存碎片

2.3、复制

  1. 标记：把需要清除的垃圾标记起来
  2. 复制：把from的区域的存活的对象全部复制到to区域
  3. 交换：复制完后，to区变成原来的from区，from区变为to区

优点：没有内存的碎片

缺点：浪费内存空间，一半空间永远为空

3、分代垃圾回收算法

3.1、堆内存介绍及算法

特点：新生代采用复制算法，老年代采用标记整理算法，提高效率

新生代：有伊甸园区、幸存区（From 、TO 区）

  1. 新创建的对象首先分配在 eden 区。
  2. 新生代空间不足时，触发 minor gc（轻GC、会触发 stop the world， 暂停其他用户线程，只让垃圾回收线程工作） ，eden 区 和 from 区存活的对象使用 复制算法 复制到 to 中，存活的对象年龄加一，然后交换 from to区。
  3. 当幸存区对象的寿命超过阈值时，会晋升到老年代，最大的寿命是 15（4bit）
  4. 当老年代空间不足时，会先触发 minor gc（轻GC），如果空间仍然不足，那么就触发 full fc （重GC），停止的时间更长！

老年代：当幸存区对象的寿命超过阈值时，会晋升到老年代

永久代：常驻内存，存java运行时的环境，不存在垃圾回收，虚拟机关闭时释放

  • JDK6之前，叫永久代，常量池在方法区
  • JDK7时， 叫永久代，但是慢慢退化，去永久代，常量池在堆中
  • JDK8之后，无永久代，常量池还在堆里，方法区在元空间里

分配担保机制：提前进入老年代

当新生代几乎全部被分配完的时候，会发生轻GC，如果此时需要分配内存空间给对象，但是空间有不足，就会提前把新生代的转移到老年代中

大对象直接进入老年代

字符串、数组等，为了避免为⼤对象分配内存时由于分配担保机制带来的复制⽽降低效率。

OOM错误

当一个线程抛出OOM异常后，它所占据的内存资源会全部被释放掉，从而不会影响其他线程的运行，进程依然正常

Hotsopt 更新阈值

Hotspot 遍历所有对象时，按照年龄从小到大对其所占用的大小进行累积，当累积的某个年龄大小超过了 survivor 区的⼀半时，取这个年龄和MaxTenuringThreshold 中更小的⼀个值，作为新的晋升年龄阈值

Hotsopt 遍历所有对象时，按照年龄从小到大对其所占用的大小进行累积，当累积的某个年龄大小超过了 survivor 区的⼀半时，取这个年龄和MaxTenuringThreshold 中更小的⼀个值，作为新的晋升年龄阈值

4、垃圾回收器

相关概念

并行收集：指多条垃圾收集线程并行工作，但此时用户线程仍处于等待状态。

并发收集：指用户线程与垃圾收集线程同时工作（不一定是并行的可能会交替执行）。用户程序在继续运行，而垃圾收集程序运行在另一个CPU上

吞吐量：即CPU用于运行用户代码的时间与CPU总消耗时间的比值（吞吐量 = 运行用户代码时间 / ( 运行用户代码时间 + 垃圾收集时间 )），也就是。例如：虚拟机共运行100分钟，垃圾收集器花掉1分钟，那么吞吐量就是99%

4.1、串行

  • 单线程的垃圾回收器。且在该线程执行回收工作时，其他线程进入阻塞状态，
  • 优点：避免垃圾回收时，因为移动对象地址使得其他线程找不到被移动的对象
  • 适合：堆内存较小的个人电脑。

Serial 收集器

Serial收集器是最基本的、发展历史最悠久的收集器，处理新生代

特点：单线程、简单高效（与其他收集器的单线程相比），采用复制算法。没有线程交互的开销。所以单个CPU下的效率最高，收集器进行垃圾回收时，必须暂停其他所有的工作线程，直到它结束（Stop The World）。

Serial Old 收集器

Serial Old是Serial收集器的老年代版本

特点：同样是单线程收集器，采用标记-整理算法

4.2、吞吐量优先

  • 多线程，适合堆内存较大，多核 cpu的程序
  • 进行GC时，仍然会暂停用户的线程，不同之处在于会开启多个线程进行GC操作。考虑的是单位时间内，让暂停的时间达到最短。
  • JDK1.8默认使用的垃圾回收器

Parallel Scavenge 收集器

与吞吐量关系密切，故也称为吞吐量优先收集器

特点：属于新生代收集器也是采用复制算法的收集器（用到了新生代的幸存区），又是并行的多线程收集器（与ParNew收集器类似），使用GC自适应调节策略（与ParNew收集器最重要的一个区别）。

GC自适应调节策略：虚拟机会根据系统的运行状况收集性能监控信息，动态的设置新生代的大小（-Xmn）、Eden与Survivor区的比例（-XX:SurvivorRation）、晋升老年代的对象年龄（-XX:PretenureSizeThreshold）等。可设置-XX:+UseAdptiveSizePolicy参数调整调用的CPU核数量。

Parallel Scavenge收集器使用两个参数控制吞吐量：

  • XX:MaxGCPauseMillis 控制最大的垃圾收集停顿时间
  • XX:GCRatio 直接设置吞吐量的大小

Parallel Old 收集器

是Parallel Scavenge收集器的老年代版本

特点：多线程，采用标记-整理算法

4.3、响应时间优先

  • 多线程，适合堆内存较大，多核 cpu的程序
  • 进行GC时，大部分不需要暂停用户的线程，会开启多个线程进行GC操作。考虑的是让每次暂停的时间达到最短（不影响用户的体验）。

CMS 收集器

Concurrent Mark Sweep，一种以获取最短回收停顿时间为目标的老年代收集器

特点：

  • 基于标记-清除算法实现。
  • 并发收集、低停顿，但是会产生内存碎片，所以只能使用选择空闲列表(Free List) 执行内存分配。
  • 是一款真正意义上的并发收集器。

应用场景：适用于注重服务的响应速度，希望系统停顿时间最短，给用户带来更好的体验等场景下。如web程序、b/s服务

  1. 初始标记：标记GC Roots能直接到的对象。速度很快但是仍存在Stop The World问题

  2. 并发标记：进行GC Roots Tracing 的过程，找出GC ROOT 能达到的存活对象（无STW）。

  3. 重新标记：用户操作使得并发标志需要更新，所以需要修正，仍存在Stop The World问题。会扫描整个堆空间。

  4. 并发清理：清理删除掉标记阶段判断的已经死亡的对象,释放内存空间，无STW，但是由于获得的CPU资源少一些（还需运行用户程序），所以效率会低一些。

优点：并发收集、低延迟

缺点：

  • 对CPU资源要求高一些
  • 产生内存碎片
  • 不处理浮动垃圾：一开始被引用，且标记为黑色，后来失去了引用（成为垃圾）。

ParNew 收集器

ParNew收集器其实就是Serial收集器的多线程版本

特点：多线程、ParNew收集器默认开启的收集线程数与CPU的数量相同，在CPU非常多的环境中，可以使用-XX:ParallelGCThreads参数来限制垃圾收集的线程数。和Serial收集器一样存在Stop The World问题

Serial Old 收集器

概念：Serial Old是Serial收集器的老年代版本

Serial Old 收集器

Serial Old是Serial收集器的老年代版本

新生代采用复制算法，老年代采用标记整理算法。可以和CMS配合使用

4.4、G1回收器

定义： Garbage First，JDK 9以后默认使用，而且替代了CMS 收集器。是一个分代的，增量的，并行与并发的标记-复制的垃圾回收器。

特点：

  • 并行与并发：G1 能充分利⽤ CPU、多核环境下的硬件优势，使⽤多个 CPU（CPU 或者CPU 核心）来缩短 Stop-The-World 停顿时间，可以避免其他垃圾回收器存在的STW现象。
  • 分代收集：G1把堆分为2048个区域,每一个region的大小是1 - 32M不等,必须是2的整数次幂，每个区域对应 E S O 的一种。内存的回收以 region为基本单位。
  • 空间整合：整体上是标记-整理算法，局部之间是复制算法。
  • 软实性：用户可以指定GC的最大停顿时间，回收器会根据时间找到最优的回收方案，这也是有时候为什么部分老年代没有被回收的原因。
  • 特殊的H区域：一个对象如果容量占每个region （小堆区）的 一半以上，则认为是巨型对象，默认分配在老年代，若内存不够，则会寻找连续的H区域存放，若还是不够，会不断使用GC甚至 Full GC 来获得区域。

适用场景：

  • 同时注重吞吐量和低延迟（响应时间）
  • 适合超大堆内存（内存大的），会将堆内存划分为多个大小相等的区域

相关参数：

  • JDK8 并不是默认开启的，所需要参数开启

1、G1 垃圾回收阶段

  • Young Collection：对新生代垃圾收集
  • Young Collection + Concurrent Mark：如果老年代内存到达一定的阈值了，新生代垃圾收集同时会执行一些并发的标记。
  • Mixed Collection：会对新生代 + 老年代 + 幸存区等进行混合收集，然后收集结束，会重新进入新生代收集。

第一阶段：年轻代GC

E：伊甸园 S：幸存区 O：老年代

  1. 在Eden区域快要耗尽的时候触发，若Eden不足，则进入Survivor区域，若还是不足，进入 Old 区域。
  2. 同时也会对S 区进行 复制算法的垃圾回收，Survivor区域进入新的Survivor区域，部分进入 Old 区域。
  3. 不会对O 区域进行垃圾回收。
  4. 最终Eden 为空，GC停止工作。

第二阶段：并发标记过程

Young + CM（并发标记） 阶段

  • 在 Young GC 时会对 GC Root 进行初始标记（有STW）
  • 在老年代占用堆内存的比例达到阈值时（45% 因为用户也需要使用内存），阈值可以根据用户来进行设定；会进行并发标记（不会STW），方便把存活的进行标记-复制算法，并发标记之后就会进入第三阶段。

第三阶段

会对E S O 进行全面的回收

  • 新生代：部分会存活，部分拷贝到老年代
  • 老年代：满足最大暂停时间的情况下，选择尽有价值老年代进行回收，主要为 复制算法。

-XX:MaxGCPauseMills:xxx 用于指定最长的停顿时间

注意：在G1 垃圾回收器关于Full GC的触发问题（CMS也是一样的）

  • 如果垃圾产生速度慢于垃圾回收速度，不会触发Full GC，还是并发地进行清理
  • 如果垃圾产生速度快于垃圾回收速度，便会触发Full GC，进行串行的收集，长时间 STW，会严重影响系统的响应时间。

2、Young Collection 跨代引用

只要父节点在老年代，则一律视为根节点，而遍历老年代找到GC ROOT 对象比较麻烦

办法：把老年代细分，每512KB的大小设置为1个卡，如果该区域引用了新生代对象 就标记为脏卡

Remembered Set（记忆集）：会把脏卡记录下来

3、Remark重标记（三色标记）

并发标记：处理完成之后，如果存在引用就会被保留，不存在就会被回收。

  • 黑色：节点被遍历完成,而且子节点都遍历完成
  • 灰色：当前正在遍历的节点,而且子节点还没有遍历
  • 白色：还没有遍历到的节点

原始快照SATB算法解决误回收（原本的垃圾被重新引用）

  • 并发标记之后拍摄快照，设定所有存在引用的对象，都是存活的
  • 重新标记，再次拍摄快照，将新引用的存活对象标记。
  • 最后把快照叠加，得到最终标记信息。

重新标记：

  • C的引用发生变化时，加一个写屏障，写屏障的指令会被执行，将C放入一个队列当中，并将C变为 处理中 状态，也就是灰色。
  • 在并发标记阶段结束以后，重新标记阶段会STW，然后将放在该队列中的对象重新处理，发现有强引用引用它，就会处理它

缺点：仍然存在浮动垃圾。

4、JDK 8u20 字符串去重

G1 回收时，会检查所有的字符串，把 char[] 值一样的指向同一个对象。

String.intern：关注的是String 对象，G1 关注的是 Char[] 数组

特点：节省大量内存，但是增加回收时间。

5、JDK 8u40 并发标记类卸载

在并发标记阶段结束以后，就能知道哪些类不再被使用。如果一个类加载器的所有类都不在使用，则卸载它所加载的所有类

6、JDK 8u60 回收巨型对象

  • 一个对象大于region的一半时，就称为巨型对象
  • G1不会对巨型对象进行拷贝
  • 回收时被优先考虑
  • G1会跟踪老年代所有incoming引用，如果老年代incoming引用为0的巨型对象就可以在新生代垃圾回收时处理掉。

4.5、ZGC回收器

概念：ZGC是JDK 11中推出的一款低延迟垃圾回收器，它的设计目标包括：

  • 停顿时间不超过10ms；
  • 停顿时间不会随着堆的大小，或者活跃对象的大小而增加；
  • 支持8MB~4TB级别的堆（未来支持16TB

使用范围：适用于大内存低延迟服务的内存管理和回收

使用ZGC回收器的原因：CMS和G1的停顿时间（STW）瓶颈

  • 标记阶段：初始时寻找 GCROOT时需要 STW、并发标记之后需要修正，重新标记的时候需要 STW
  • 清理阶段：清点出有存活对象的分区和没有存活对象的分区，该阶段不会清理垃圾对象，也不会执行存活对象的复制。该阶段是STW的。
  • 复制阶段：转移阶段需要分配新内存和复制对象的成员变量。转移阶段是STW的，该阶段耗时较长。G1未能解决转移过程中准确定位对象地址的问题。

ZGC

最大特点：没有进行分代，因为ZGC的底层算法比较复杂，暂时还没有写分代，后续会加入。

内存分布：以Region的方式进行内存布局的，暂时没有设置分代，使用读屏障、颜色指针等技术来实现可并发的标记–整理算法。

  • 小型Region(small Region)：容量为2mb，用于放置小于256kb的对象。
  • 中型Region(medium Region)：容量为3mb，用于放置容量大于等于256kb但小于4mb的对象。
  • 大型Region(large Region)：容量不固定，大小可变，但必须是2的整数倍，且大小大于等于4mb的对象。

注意：每个大型Region只存放一个大对象，所以大型对象不会被ZGC重分配，当成为垃圾时直接销毁即可。

优势：ZGC也采用标记-复制算法，不过ZGC对该算法做了重大改进：ZGC在标记、转移和重定位阶段几乎都是并发的，这是ZGC实现停顿时间小于10ms目标的最关键原因。

三次STW：只有 初始标记、再标记、初始转移阶段 是STW的

  • 初始标记：初始标记就扫描 GC Roots 直接可达的，耗时很短，
  • 再标记：再标记阶段STW时间很短，最多1ms，超过1ms则再次进入并发标记阶段。
  • 初始转移

关键技术：着色指针和读屏障技术

着色指针

描述：将信息存储在指针中的技术。

思想：使用“空间换时间”思想，去降低GC停顿时间，将对象存活信息存储在42~45位中，这与传统的垃圾回收并将对象存活信息放在对象头中完全不同。

读屏障

描述：JVM向应用代码插入一小段代码的技术。当应用线程从堆中读取对象引用时，就会执行这段代码。需要注意的是，仅“从堆中读取对象引用”才会触发这段代码。

作用：如果发现对象被移动了，那么“读屏障”会通过着色指针把读出来的指针更新到对象的新地址上

G1的优势

高并发，一秒数十万的请求，使用G1更好

原因：

  1. 由于消息量大，但是不需要长时间保存，所以进行minor gc是最好的
  2. G1可以进行最大停顿时间控制，让程序边回收程序边运行，

4.6、垃圾回收器的选择

  1. 优先调整堆的大小让服务器自己来选择。
  2. 如果内存小于100M，使用串行收集器。
  3. 如果是单核，并且没有停顿时间的要求，串行或JVM自己选择。
  4. 如果允许停顿时间超过1秒，选择并行或者JVM自己选。
  5. 如果响应时间最重要，并且不能超过1秒，使用并发收集器。
  6. 4G以下可以用parallel，4-8G可以用ParNew+CMS，8G以上可以用G1，几百G以上用ZGC
  7. 实时数据占用超过一半的堆空间、对象分配或者晋升的速度变化大使用G1

4.7、G1和CMS

适用范围

CMS收集器是老年代的收集器，可以配合新生代的Serial和ParNew收集器一起使用

G1收集器收集范围是老年代和新生代。不需要结合其他收集器使用

STW时间

CMS收集器以最小的停顿时间为目标的收集器。

G1收集器可预测垃圾回收的停顿时间（建立可预测的停顿时间模型）

垃圾碎片

CMS收集器是使用“标记-清除”算法进行的垃圾回收，容易产生内存碎片

G1收集器使用的是“标记-整理”算法，进行了空间整合，降低了内存空间碎片。

回收过程

CMS：初始标记 —》 并发标记 —-》 重新标记 —-》 并发清除

G1： 初始标记 —-》并发标记 —–》 最终标记 —–》 筛选回收

内存空间

CMS：是分代的

G1：内存拆分成多等份，多个域（Region），同时保留不同代的区别。

回收算法

CMS：标记清除

G1：标记整理 复制

大对象处理

G1在压缩空间方面有优势、避免内存碎片、Eden、Survivor、Old区不再固定，在内存使用效率上来说更灵活、设置预期停顿时间、适合大内存的情况。

5、GC调优

查看GC相关的参数

调优方向：内存、锁竞争、CPU占用、IO、GC

确定目标：低延迟/高吞吐量？ 选择合适的GC

  • CMS G1 ZGC
  • ParallelGC
  • Zing

最快的GC：

  1. 优化一下代码，避免大量频繁创建对象。
  2. 查看 Full GC 前后的内存占用，考虑以下几个问题
    • 数据是不是太多
    • 数据表示是否太臃肿
    • 是否存在内存泄漏

5.1、新生代调优

  • 所有的new操作分配内存都是非常廉价的
  • 死亡对象回收零代价
  • 大部分对象用过即死（朝生夕死）
  • MInor GC 所用时间远小于Full GC

新生代内存越大越好么？

  • 新生代内存太小：频繁触发Minor GC，会STW，会使得吞吐量下降
  • 新生代内存太大：老年代内存占比有所降低，会更频繁地触发Full GC。而且触发Minor GC时，清理新生代所花费的时间会更长

优化：新生代内存设置为内容纳[并发量*(请求-响应)]的数据为宜

5.2、幸存区调优

  • 幸存区需要能够保存 当前活跃对象+需要晋升的对象
  • 晋升阈值配置得当，让长时间存活的对象尽快晋升

5.3、老年代调优

  • CMS 的老年代内存越大越好
  • 先尝试不做调优，如果没有 Full GC 那么，可以先尝试调优新生代。
  • 观察发现 Full GC 时老年代内存占用高，将老年代内存预设调大 1/4 ~ 1/3

5.4、案例

案例1：Full GC 和 Minor GC 频繁

  • 尝试增大新生代内存，

案例2：请求高峰期发生 Full GC，单次暂停时间特别长（CMS）

  • 重新标记阶段前进行一次 轻 GC

案例3：老年代充裕情况下，发生 Full GC（CMS jdk1.7）

  • 永久代的空间不足，增大永久代的大小
