JVM(二):堆内存回收

1、判断对象可以回收

1.1、引用计数法

当一个对象被引用时，就当引用对象的值加一，当值为 0 时，就表示该对象不被引用，可以被回收。

缺陷：循环引用时，两个对象的计数都为1，导致两个对象都无法被释放。

1.2、可达性分析算法

扫描堆中的对象，看能否沿着 GC Root 对象为起点的引用链找到该对象，如果找不到，则表示可以回收

GC Root 对象：

  • 虚拟机栈中引用的对象。
  • 方法区中类静态属性引用的对象
  • 方法区中常量引用的对象
  • 本地方法栈中 JNI（即一般说的Native方法）引用的对象
  • JVM内部的引用，比如Class对象，同步（synchronized）对象

GC ROOT查找过程

  1. 第一次可达性后，如果没有被GC RooT引用，那么做一个标记
  2. JVM选择性调用目标的 finalize 方法（重写了该方法，并且全局该对象未调用过该方）
  3. 第二次可达性后，如果没有被GC RooT引用，那么第二次标记，随后回收。如果可达，就不回收。

1.3、四种引用

  1. 强引用：宁愿抛出OOM异常也不会被回收，除非不再被强引用

  2. 软引用（SoftReference）：在垃圾回收后，内存仍不足时会回收。

  3. 弱引用（WeakReference）：在垃圾回收时，无论内存是否充足，都会回收。

  4. 虚引用（PhantomReference）：无法通过虚引用获取实例。虚引用的唯一目的就是对象回收时会收到收集器的一个通知。这样便于虚引用对象被回收。当虚引用对象所引用的对象被回收以后，虚引用对象就会被放入引用队列中，调用虚引用的方法，虚引用的一个体现是释放直接内存所分配的内存，当引用的对象ByteBuffer被垃圾回收以后，虚引用对象Cleaner就会被放入引用队列中，然后调用Cleaner的clean方法来释放直接内存

注意点：

  • 软引用、弱引用也可以和⼀个引用队列（ReferenceQueue）联合使用，在弱引用和虚引用所引用的对象被回收以后，会将这些引用放入引用队列中，方便一起回收这些软/弱引用对象。
  • 虚引用和终结器引用必须配合引用队列，虚引用和终结器引用在使用时会关联一个引用队列。
  • 在程序设计中⼀般很少使用弱引用与虚引用，使用软引用的情况多，这是因为软引用可以加速JVM对垃圾内存的回收速度，可以维护系统的运行安全，防止内存溢出OOM等问题的产生。

软引用案例

弱引用的使用和软引用类似，只是将 SoftReference 换为了 WeakReference

扩展

终结器引用（FinalReference）：所有的类都继承自Object类，Object类有一个finalize方法。当某个对象不再被其他的对象所引用时，会先将终结器引用对象放入引用队列中，然后根据终结器引用对象找到它所引用的对象，然后调用该对象的finalize方法。调用以后，该对象就可以被垃圾回收了。

2.1、概述

分代收集理论

  • 弱分代假说：绝大部分对象都是朝生夕灭的
  • 强分代假说：熬过多次回收的对象难以消亡
  • 跨代引用假说：假设跨代的对象之间几乎不产生引用。

以上提供了一个思路：

  1. 按照对象的年龄分代存储，分代回收。
  2. 跨代引用只需要在更年轻的一代维护一个全局的数据结果（记忆集），记录存在跨代引用的老年代区域，这样每次回收只需要遍历新生代和部分老年代，分代的作用得到体现。

分代回收名词

  1. 部分收集（Partial GC）：
    • 新生代收集（Minor GC、Young GC）：只回收新生代
    • 老年代收集（Major GC、Old GC）：注意只有 CMS 收集器会单独回收老年代
    • 混合收集（Mixed GC）：收集整个新生代以及部分老年代的垃圾收集。只有G1会涉及。
  2. 整堆收集（Full GC）：全部堆、方法区

2.2、几种垃圾回收算法

1、标记清除算法

  1. 标记：把需要清除的垃圾标记起来
  2. 清除：定义一张表，记录垃圾占用空间的首地址和尾地址，这些空间下次就可以继续分配

特点：速度快，但是会产生内存碎片，不连续的内存导致无法给大对象进行内存分配，进而导致频繁进行GC，影响性能。

2、标记整理

  1. 标记：把需要清除的垃圾标记起来
  2. 整理：把垃圾清除后，再把剩余的空间整理，确保没有内存碎片

特点：速度慢，但是没有内存碎片

3、标记复制

  1. 标记：把需要清除的垃圾标记起来
  2. 复制：把from的区域的存活的对象全部复制到to区域
  3. 交换：复制完后，to区变成原来的from区，from区变为to区

特点：没有内存的碎片，但是浪费内存空间，一半空间永远为空

复制算法的优化

现象：针对大量的对象存在朝生夕灭的情况，比如98%的对象会在第一轮被回收。

策略：即把新生代的区域设置的大一些，比如HotSpot默认情况下是 Eden 比 Survivor 等于 8 ：1。这样只会浪费很少的空间。但是也存在问题，就是不一定每次都会淘汰掉大部分对象，可能存在空间不足的情况。

特殊策略：分配担保，老年代可以担保，当幸存区的空间不足，对象可以直接进入老年代。

4、分代垃圾回收

1、概述

  1. 新生代使用标记复制算法
  2. 老年代使用标记清除（关注延迟）或者 标记整理（关注吞吐量）

2、步骤分析

  1. 新创建的对象首先分配在 eden 区 的 from 区域。
  2. eden 的 from 空间不足时，触发 minor gc（ 会触发 stop the world， 暂停线程），存活的标记加1，进入 to 区 然后交换 from to区。如果年龄大就进入 old 区域。
  3. old 不足时触发 Full GC。

注意：stop the world触发是因为复制对象时必须保证用户不会干扰到对象复制，所以必须暂停。这样引出了下文。

3、延迟和吞吐的选择

概述：复制对象时必须停止用户线程防止出现问题，这时候如果还要进行内存整理，那么耗时更长，但是不操作，就会对象不连续，以后获取对象耗时更长，导致吞吐率下降。

  1. 选择吞吐量：标记整理
  2. 选择延迟：标记清除（注意标记清除次数多了内存碎片会极大影响效率，所以当碎片化超过阈值的时候，一般也会选择进标记整理）

3、HotSpot分析

3.1、根节点枚举

OopMap数据结构

作用：每当类加载完成或者即时编译后，就会同时把哪些地方存在对象引用记录到OopMap中，这样JVM寻找GC Root的时候就不需要遍历虚拟机栈，只需要根据OopMap记录的位置区寻找即可。

安全点

问题：导致OopMap发送变化的代码行非常多，不可能为所有的地方都生成OopMap。消耗空间太大。

安全点：JVM把一些特殊的地方设置为安全点，安全点出保存了OopMap数据结构，现在需要考虑如果让线程能够停止到安全点。

中断策略

  • 抢先式中断：中断后再唤醒线程，直到线程到达安全点。现在几乎不使用。
  • 主动式中断：需要中断的线程设置标志位，线程会不断轮询标记位，等线程执行到安全点时就会主动中断。

安全区

问题：中断需要线程主动的执行，如果线程处于休眠状态，显然无法主动移动到安全点。

安全区：安全区是不会改变引用关系的区域，处于安全区的线程不会收到垃圾回收造成的线程停止的制约，但是如果想要退出安全区就必须等待。直到 Stop The Word 结束。

3.2、记忆集和卡带

概述：只要涉及到部分回收，那么就离不开记忆集和卡带。

记忆集

概念：记录从非收集区域到收集区域（新生代到老年代）和指针集合的数据结构。

  • 字长精度：精确到一个字长（处理器的寻址位数，如32位，64位）
  • 对象精度：精确到一个对象
  • 卡精度：精确到一块区域

卡带

概念：是对记忆集的一种实现，规定了记忆集的精度为一块内存。

体现：HotSpot表现为一个字节数组。该字节数组的长度即代表内存被划分为多少块，只要该区域存在对象引用，就标志为1，代表变脏了，如果不存在，则为0。

写屏障

问题：卡带如何变脏？原则上引用类型字段赋值的时候触发。如果赋值的时候是执行字节码时候，比较容易实现。但是如果已经解释&编译成机器码的时候，比较难处理。

解决：HotSpot加入写屏障，可以将该操作看做一个操作系统层面的AOP操作。赋值前称为前屏障、复制后称为后屏障。只需要在后屏障更新卡表即可。

缺点：如果开了写屏障，那么所有的引用更新都会增加一步操作。

伪共享

缓存行：缓存系统中是以缓存行（cache line）为单位存储的，当多线程修改互相独立的变量时，如果这些变量共享同一个缓存行，就会无意中影响彼此的性能，这就是伪共享。

卡表：1个卡表占1个字节，64个卡表元素共享一个缓存行，64个卡表对应32KB大小的内存。如果不同的线程操作，可能导致更新卡表的时候正好写入一个缓存，影响性能。

解决：增加条件，之后未脏的才进行更新，减少更新的频率，减少冲突

3.3、三色标记法

可达性分析回顾

要求：可达性分析要求必须 Stop The World，要求全过程保持一致性。

  • GC ROOT 对象：现在已经优化的时间短暂固定（不随堆容量增长而增长）。
  • GC ROOT 引用对象：随堆容量增长而增长，如果堆很大，消耗时间会很长，这部分需要优化。

总结：当堆很大的情况下，如果Stop The World 等待GC ROOT遍历根完成，无疑会造成很大的问题，所以需要该部分主线程能够不 Stop The World

并发情况下的一致性保证

GC ROOT 遍历过程中，按照颜色标记对象的访问状态：访问完成后一条引用链上的应该全部为黑色。

  • 黑色：节点被遍历完成,而且子节点都遍历完成。
  • 灰色：当前正在遍历的节点,而且子节点还没有遍历，需要继续遍历。
  • 白色：还没有遍历到的节点，一开始默认全部是白色。

可以想象，GC ROOT 线程和用户线程并行，会出现两个问题

  1. 未访问的白色节点被黑色的引用；并且原本白色节点相连的灰色节点不存在或者被删除，导致白色的节点被误认为已访问，不再判断是否被引用。最后被当成没有被引用的垃圾被回收。不可容忍。
  2. 黑色的节点，失去了引用，但是不能即时更新导致仍然认为是GC ROOT 可达。会导致浮动垃圾。但是可以容忍。

1、增量更新（未访问的白色节点被黑色的引用）

思想：只要引用了新的对象，就需要把自己变为灰色。但是需要重新遍历，会导致消耗时间过长

  1. 重新扫描（变灰）新生代。
  2. 重新扫描并发标记结束时老年代的Card Table记录下的对象。
  3. 重新扫描并发标记结束时老年代的Mod-Union Table记录下的对象。

技术：在原始快照中运用到了写后屏障

实例：CMS回收器使用。

2、原始快照（原本白色节点相连的灰色节点不存在或者被删除）

思想：只要灰色删除了白色的引用，就记录下来放进当前用户线程私有的SATB Queue中，最终汇集到一个全局的SATB Queue Set中，在最终标记阶段再扫描这些记录下来的引用记录。

技术：在原始快照中运用到了写前屏障，因为要在引用被删除前把灰色对象到白色对象的关系记录。

实例：G1使用

注意

  • 以上两种方式都是通过加入写屏障实现的（操作系统层面的AOP）。
  • GC ROOT 往下寻找的时候是 并发的，但是最后修正标记的时候也必须 Stop the world。但是时间会很短。

3.4、JVM的一些优化

  1. 优先分配Eden区域：
  2. 大对象直接进入老年代：
  3. 长期存活的进入老年代：
  4. 动态更新阈值：Hotspot 遍历所有对象时，按照年龄从小到大对其所占用的大小进行累积，当累积的某个年龄大小超过了 survivor 区的⼀半时，取这个年龄和MaxTenuringThreshold 中更小的⼀个值，作为新的晋升年龄阈值。
  5. 线程OOM抛弃：当一个线程抛出OOM异常后，它所占据的内存资源会全部被释放掉，从而不会影响其他线程的运行，进程依然正常。
  6. 分配担保机制——提前进入老年代：当发生Minor GC 时，会判断老年代剩余内存和新生代占用内存
    • 老年代大：直接进行 MinorGC
    • 新生代大：会先看是否允许分配担保，若允许，则会根据以往晋升的对象数量判断是否进行Minor GC。若无法进行 Minor GC，则会进行 Full GC。允许Minor GC 的情况下，如果新生代内存不足，那么就会直接进入老年代。
    • 担保失败： Full GC
    • JDK 6 UPdate 24 之后，默认开启。

4、垃圾回收器

概述

相关概念

并行收集：指多条垃圾收集线程并行工作，但此时用户线程仍处于等待状态。

并发收集：指用户线程与垃圾收集线程同时工作（不一定是并行的可能会交替执行）。用户程序在继续运行，而垃圾收集程序运行在另一个CPU上

吞吐量：即CPU用于运行用户代码的时间与CPU总消耗时间的比值（吞吐量 = 运行用户代码时间 / ( 运行用户代码时间 + 垃圾收集时间 )），也就是。例如：虚拟机共运行100分钟，垃圾收集器花掉1分钟，那么吞吐量就是99%

4.1、Serial 收集器

Serial 收集器

  1. Serial收集器是最基本的、发展历史最悠久的收集器，负责处理新生代
  2. 单线程执行的垃圾回收器，进行垃圾回收时 会全程 Stop The World ，暂停用户线程。
  3. 采用标记复制算法实现新生代的垃圾回收。

优缺点：虽然会 Stop The World ，但是也避免了并发的安全问题，线程交互的开销。适合堆内存较小的电脑，这样停顿时间也会很小，不影响体验。

Serial Old 收集器

  1. Serial Old是Serial收集器的老年代版本
  2. 同样是单线程收集器，采用标记-整理算法

注意：该收集器属于串行收集器。

4.2、ParNew 收集器

概念：Serial 收集器的多线程版本，处理新生代。目前只能和CMS收集器配合使用。为了推广G1 收集器，把ParNew 和Serial Old 、Serial 和 CMS 的取消了，可以说是第一款退出历史舞台的收集器。

4.3、Parallel Scavenge 收集器

Parallel Scavenge 收集器

  1. 该收集器关注的是系统的吞吐量，不关心停顿时间，故也称为吞吐量优先收集器。
  2. 多线程执行的垃圾回收器，进行垃圾回收时 会全程 Stop The World ，暂停用户线程。
  3. 采用标记复制算法进行新生代垃圾回收
  4. 使用GC自适应调节策略（与ParNew收集器最重要的一个区别）

GC自适应调节策略：虚拟机会根据系统的运行状况收集性能监控信息，动态的设置新生代的大小（-Xmn）、Eden与Survivor区的比例（-XX:SurvivorRation）、晋升老年代的对象年龄（-XX:PretenureSizeThreshold）等。可设置-XX:+UseAdptiveSizePolicy参数调整调用的CPU核数量。

控制吞吐量的两个参数：

  • XX:MaxGCPauseMillis：控制最大的垃圾收集停顿时间，收集器会尽力执行。是通过控制新生代堆内存的大小来实现的，所以需要合理设置。
  • XX:GCRatio：设置垃圾回收时间占总时间的比例，默认99%，即1%时间用于垃圾回收。

Parallel Old 收集器

  1. 是Parallel Scavenge收集器的老年代版本
  2. 多线程，采用标记-整理算法，

4.4、CMS 收集器

CMS 收集器

  1. Concurrent Mark Sweep，一种以获取最短回收停顿时间为目标的老年代收集器
  2. 基于标记-清除算法实现，是一款真正意义上的并发（和用户线程共同工作）收集器。
  3. 唯一一款单独对老年代进行垃圾回收的收集器。
  4. 多线程收集、低停顿，会产生内存碎片，所以只能使用选择空闲列表(Free List) 执行内存分配。
  5. 对CPU资源要求高（程序变慢）、无法处理浮动垃圾。

垃圾回收过程

  1. 初始标记：标记GC Roots能直接到的对象。速度很快但是仍存在**Stop The World **问题

  2. 并发标记：进行GC Roots Tracing 的过程，找出GC ROOT 能达到的存活对象（无STW）。

  3. 重新标记：用户操作使得并发标志需要更新，所以需要修正，仍存在Stop The World问题。会扫描整个堆空间。（增量更新解决误回收对象）

  4. 并发清理：清理删除掉标记阶段判断的已经死亡的对象,释放内存空间，无STW（不需复制对象），但是由于获得的CPU资源少一些（还需运行用户程序），所以效率会低一些。

为什么不用标记整理算法？

  1. 标记整理算法需要在把空间进行整理， STW 时间变长，延迟增大。
  2. 实际上也会使用，当碎片化太严重时，会进行整理。JDK9之前Full GC时整理，但是会提高时延，于是把整理提到 Full GC 前。注意 整理的时候不能并发。

Concurrent Mode Failure 并发失败

原因：第一点是因为无法处理浮动垃圾，导致占用一定的内存；第二点是因为和用户线程并发执行，所以需要预留足够的内存空间给用户，所以 JDK 5 时老年代达到 68 % 的时候，就会就会进行回收。为了效率，JDK 6 提升到 92 %。这也导致可能无法拿出足够的内存给用户使用，当内存分配失败的时候，CMS就会退化成为 Serial Old 收集器，进行 STW 暂停 用户线程。。

4.4、G1回收器

G1的设计思路

  1. 注重延时：根据优先级部分回收垃圾，这样程序 STW 时间就不会过长。
  2. 注重吞吐量：追求吞吐量，而不是追求延时，这也是为什么回收阶段不选择和用户线程并发的原因（注意G1线程是可以并发的）
  3. 没有内存碎片：整体上可以看做标记-整理算法，局部之间可以看做是标记复制算法。

4.1、特点概述

定义： Garbage First，JDK 9以后默认使用，而且替代了CMS 收集器。是一个分代的，增量的，并行与并发的标记-复制的垃圾回收器。

1、区域粒度更小

  • 细粒度：G1把堆分为2048个区域,每一个region的大小是1 - 32M不等,必须是2的整数次幂。
  • 动态分代：每个区域根据内部垃圾的种类 对应 E（新生代） S （幸存区）O（老年代） 的一种。

2、Humongous区域（大对象优先回收）

  • 特殊的H区域：一个对象如果容量占每个region （小堆区）的 一半以上，则认为是巨型对象，默认分配在老年代，若内存不够，则会寻找连续的H区域存放，若还是不够，会不断使用GC甚至 Full GC 来获得区域。
  • 优先回收：大内存对象往往独占整个 Region ，所以一旦大对象变为垃圾该区域都是垃圾，比较容易处理，会优先被回收
  • 不支持拷贝：JVM不会对巨型对象进行拷贝。

3、可预测停顿（缩短STW时间）

  • 优先级划分：G1收集器会跟踪各个 Region 里面的垃圾堆积的价值大小，综合考虑回收的内存和耗费时间，把结果维护在一张优先级列表里面，
  • 指定停顿时间：用户可以指定GC的最大停顿时间，回收器会根据优先级列表找到最佳的垃圾回收方案，这也是有时候为什么部分新生代没有被回收的原因。所以适合超大内存，可以边产生边回收。
  • 混合回收：会回收全部的新生代，根据指定停顿时间回收选择性的回收老年代。

4.2、一些问题

1、跨带引用问题

记忆集：G1的每个 Region 区域都有自己的记忆集，记录的是双向指针：别的Region指向自己的指针、指针所在的卡页范围。

本质：G1使用哈希表实现，key为其他 Region 的起始地址，value为一个集合记录所有卡表的索引号。每个value集合相当于普通分代策略的记忆集。

缺点：由于G1的每个 Region 区域都有自己的记忆集，所以消耗空间也比较大，一般需要消耗堆的 10% ~20% 来维持运行。

2、并发的安全性问题

  • 标记错误问题：G1 采用原始快照，CMS采用增量更新。
  • 内存分配问题：（注意CMS是老年代的，对象分配在新生代所以CMS无需考虑）由于并发标记期间和用户是并行的，所以用户会产生新的对象，对于这部分对象，G1 收集器设置两个名为 TAMS 的指针，垃圾回收期间新对象内存在此处分配；该指针范围内的对象不会参与本次垃圾回收。

注意：当 TAMS 因为内存不足而无法分配时，G1收集器也会被迫 Stop Thre World，进行 Full GC 串行的垃圾收集。

4.3、回收过程

第一阶段：Young Collection 年轻代

概述：所有的Eden快满时触发，会 根据卡带的信息 对E、S区域进行如下操作，并且最后更新卡带。

  1. 处理 E 区：存活后进入S区域，标记复制算法。
  2. 处理 S 区：存活后进入新S区域，标记复制算法，部分进入O区域。
  3. 暂停线程： 存在STW ，目的是对象复制；同时对GC ROOT对象进行初始标记。

第二阶段：Young Collection + Concurrent Mark 并发标记

触发：在老年代占用堆内存的比例达到阈值时（45% 因为用户也需要使用内存），阈值可以根据用户来进行设定；会进行并发标记（不会STW），具体过程如下。

  1. 初始标记：存在STW，会找到GC ROOT直接关联的对象，并且设置 TAMS 指针便于对象分配。具体实现是通过进行一次新生代的 GC 。
  2. 并发标记：不存在STW，会使用三色标记 进行可达性分析，同时拍摄 SATB（原始快照，记录被取消引用的对象）。

扩展：并发标记会进行类卸载，并且会检查所有字符串，把 char[] 值一样的指向同一个对象。注意与String类的 intern() 方法区分开。

第三阶段：混合回收 Mixed Collection（很长的一个过程，涉及多次GC）

概述：采用和新生代一样的标记复制算法。优先回收新生代，在满足最大暂停时间的情况下，选择有价值老年代进行回收。

  1. 重新标记：存在短暂的STW，会对原始快照进行处理。
  2. 筛选回收：存在STW，此阶段由第三阶段混合回收完成。会结合用户设置的最大停顿时间 和 维护的优先级列表进行回收，并且更新优先级列表。回收方式是清空原本的 Region 区域，存活的迁移到另一个区域。

4、Full GC（Mixed GC 失败）

  • 如果垃圾产生速度慢于垃圾回收速度，不会触发Full GC，还是并发地进行清理
  • 如果垃圾产生速度快于垃圾回收速度， TAMS 指针分配内存失败，便会触发Full GC，进行串行的收集，长时间 STW，会严重影响系统的响应时间。

4.4、CMS和G1的对比

1、回收算法

  • CMS：标记清除，内存碎片。只有老年代的回收
  • G1：整体标记整理，局部标记复制，没有内存碎片。分为Mix Gc 和 Young GC

2、卡表

  • CMS：只有一份，在新生代记录被老年代所引用的区域
  • G1：每个 Region 区域都有一份，占用资源，所以如果 堆总内存较小，反而占用资源更多。

3、标记修正

  • CMS：增量更新，对新增引用的对象处理（增量更新remark阶段需要重新扫描新生代）
  • G1：原始快照，对失去引用的对象处理（原始快照可以减少并发标记和重新标记阶段的资源消耗）

4、内存屏障

  • CMS：写后屏障维护卡表，直接同步操作。
  • G1：写后屏障维护卡表，写前屏障维护原始快照。由于比较复杂，所以以消息队列的形式存储，过一会儿再实现。

5、回收时间

  • CMS：放弃整理内存，换取延时的降低。
  • G1：放弃对全部内存的回收，换取延时的降低。

5、实战

  • G1：适合大内存，一般 6 ~ 8 G 内存以上比G1更好
  • CMS

4.5、低延迟收集器

1、概述

1、垃圾收集器的三个指标（通常只能实现两项）

  • 内存占用
  • 吞吐量
  • 延迟

2、低延迟收集器

描述：Shenandoah 和 ZGC 几乎全程并发，只有初始标记、最终标记阶段由短暂的停顿；并且停顿的时间与堆内存的大小几乎无关。

分析：低延迟收集器主要是减少了标记确认后垃圾回收执行过程的 Stop The World

2、Shenandoah收集器

1、概念

  • 第一款不由 Oracle 开发的收集器，所以收到排挤，知名度也不如其他收集器。
  • 相比 G1 收集器，支持标记确认后的回收阶段 与用户线程并发执行。
  • 相比 G1 收集器，同样使用 Region 进行分区，但是完全抛弃分代的概念。
  • 相比 G1 收集器，使用名为 连接矩阵的数据结构来存储解决跨 Region 引用的问题。

2、回收阶段概述

  1. 并发标记
  2. 并发回收
  3. 并发引用更新：回收完成之后需要复制存活的对象，这时就需要能够即使的更新引用。

3、转发指针

概念：转发指针就是让指针进行一次转发，指向真实的地址，每次更新先把旧对象的转发指针设置为新地址，这样就不会导致获取对象失败。

问题： 复制旧对象 —–> 用户修改字段 ——–> 修改转发指针 导致用户的修改操作失效。

解决：CAS方式。

3、ZGC回收器

1、概念

概念：ZGC是JDK 11中推出的一款低延迟垃圾回收器，

  • 停顿时间不超过10ms；
  • 停顿时间不会随着堆的大小，或者活跃对象的大小而增加；
  • 支持8MB~4TB级别的堆（未来支持16TB）

使用范围：适用于大内存低延迟服务的内存管理和回收

2、G1的缺陷

使用ZGC回收器的原因：G1的停顿时间（STW）瓶颈

G1复制阶段：对象复制阶段需要分配新内存和复制对象的成员变量。转移阶段是STW的，该阶段耗时较长。G1未能解决转移过程中准确定位对象地址的问题。

3、ZGC的优势

最大特点：没有进行分代，因为ZGC的底层算法比较复杂，暂时还没有写分代，后续会加入。

内存分布：以Region（动态创建、销毁、扩容）的方式进行内存布局的，暂时没有设置分代，使用读屏障、颜色指针等技术来实现可并发的标记–整理算法。

  • 小型Region(small Region)：容量为2mb，用于放置小于256kb的对象。
  • 中型Region(medium Region)：容量为3mb，用于放置容量大于等于256kb但小于4mb的对象。
  • 大型Region(large Region)：容量不固定，大小可变，但必须是2的整数倍，且大小大于等于4mb的对象。每个大型Region只存放一个大对象，所以大型对象不会被ZGC重分配，当成为垃圾时直接销毁即可。

优势：ZGC也采用标记-复制算法，不过ZGC对该算法做了重大改进：ZGC在标记、转移和重定位阶段几乎都是并发的，这是ZGC实现停顿时间小于10ms目标的最关键原因。

着色指针

描述：将信息存储在指针中的技术。

思想：使用“空间换时间”思想，去降低GC停顿时间，将对象存活信息（三色标记状态、是否进入重分配集等）存储在指针的第42~45位中，说这与传统的垃圾回收并将对象存活信息放在对象头中完全不同。不过由于占用4位，所以最大支持内存只有4T，比其他的少。

其他的收集器：Serial把标记记录在对象头上，G1、Shenandoah使用一个叫 BItMap的数据结构。

着色指针技术的基础：内存多重映射

原理：将不同的虚拟内存地址映射到同一个物理内存地址上的技术，这样即使第42~45位改变了，映射的地址还是一致的

读屏障

描述：JVM向应用代码插入一小段代码的技术。当应用线程从堆中读取对象引用时，就会执行这段代码。需要注意的是，仅“从堆中读取对象引用”才会触发这段代码。

作用：如果发现对象被移动了，那么“读屏障”会通过着色指针把读出来的指针更新到对象的新地址上

4、回收流程

  1. 并发标记：会STW，并且在此阶段会更新着色指针。
  2. 并发预备重分配：不会STW，会扫描整个 Reigon 区域 ，根据条件计算回收哪些区域，并且回收只保证 Region 被释放，不保证对象被回收。
  3. 并发重分配：内部维护一个转发表，记录旧地址到新地址的映射关系，由于染色指针，所以仅仅通过引用就可以知道处于重分配过程，用户线程获取对象时就会根据转发记录表修正指针，实现地址重分配。至于没有立即访问但是又存在引用的将来可能访问的，只需要保留转发表，将来访问时，根据染色指针判断处于重分配状态，就会查找转发表修正。
  4. 并发重映射：修正堆中指向重分配集中就对象的所有引用。ZGC把这部分操作交给下一次的并发标记阶段去处理，而再次期间，只需要保留转发表即可。

4.6、ZGC和G1

是否分代

  • G1：每一个 Region 都维护一个卡表，需要依靠写屏障实现；内存占用大，影响性能。
  • ZGC：不分代，不需要跨带引用，但是需要全堆扫描，所以每次并发收集的时间都会很长。回收期间产生的垃圾收集器都默认存活，所以会造成大量浮动垃圾。增大对内存可以很好的缓解这个问题，所以ZGC可以在一些比较大内存的设备上使用。

使用场景

  • 高并发，一秒数十万的请求，比如 Kafka 使用G1更好。因为分代有 Young GC更快，而且可以最大停顿时间控制，让程序边回收程序边运行。
  • 时延要求高，可以使用ZGC。

4.6、垃圾回收器的选择

  1. 优先调整堆的大小让服务器自己来选择。
  2. 如果内存小于100M，使用串行收集器。
  3. 如果是单核，并且没有停顿时间的要求，串行或JVM自己选择。
  4. 如果允许停顿时间超过1秒，选择并行或者JVM自己选。
  5. 如果响应时间最重要，并且不能超过1秒，使用并发收集器。
  6. 4G以下可以用parallel，4-8G可以用ParNew+CMS，8G以上可以用G1，几百G以上用ZGC
  7. 实时数据占用超过一半的堆空间、对象分配或者晋升的速度变化大使用G1

5、GC调优

查看GC相关的参数

调优方向：内存、锁竞争、CPU占用、IO、GC

确定目标：低延迟/高吞吐量？ 选择合适的GC

  • CMS G1 ZGC
  • ParallelGC
  • Zing

最快的GC：

  1. 优化一下代码，避免大量频繁创建对象。
  2. 查看 Full GC 前后的内存占用，考虑以下几个问题
    • 数据是不是太多
    • 数据表示是否太臃肿
    • 是否存在内存泄漏

5.1、新生代调优

  • 所有的new操作分配内存都是非常廉价的
  • 死亡对象回收零代价
  • 大部分对象用过即死（朝生夕死）
  • MInor GC 所用时间远小于Full GC

新生代内存越大越好么？

  • 新生代内存太小：频繁触发Minor GC，会STW，会使得吞吐量下降
  • 新生代内存太大：老年代内存占比有所降低，会更频繁地触发Full GC。而且触发Minor GC时，清理新生代所花费的时间会更长

优化：新生代内存设置为内容纳[并发量*(请求-响应)]的数据为宜

5.2、幸存区调优

  • 幸存区需要能够保存 当前活跃对象+需要晋升的对象
  • 晋升阈值配置得当，让长时间存活的对象尽快晋升

5.3、老年代调优

  • CMS 的老年代内存越大越好
  • 先尝试不做调优，如果没有 Full GC 那么，可以先尝试调优新生代。
  • 观察发现 Full GC 时老年代内存占用高，将老年代内存预设调大 1/4 ~ 1/3

5.4、案例

案例1：Full GC 和 Minor GC 频繁

  • 尝试增大新生代内存，

案例2：请求高峰期发生 Full GC，单次暂停时间特别长（CMS）

  • 重新标记阶段前进行一次 轻 GC

案例3：老年代充裕情况下，发生 Full GC（CMS jdk1.7）

  • 永久代的空间不足，增大永久代的大小
