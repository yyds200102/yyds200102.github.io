1、JVM简介

Java 虚拟机（JVM）是运行 Java 字节码的虚拟机。JVM 有针对不同系统的特定实现
（Windows，Linux，macOS），目的是使用相同的字节码，它们都会给出相同的结果。

字节码

概念：JVM 可以理解的代码就叫做字节码（即扩展名为 .class 的文件），只面向虚拟机。

Java 语⾔通过字节码的方式，在⼀定程度上解决了传统解释型语言执行效率低的问题，同时保留了解释型语言可移植的特点。所以 Java 程序运行时比较高效，同时，由于字节码并不针对⼀种特定的机器，因此，Java 程序无须重新编译便可在多种不同操作系统的计算机上运行。

4、类加载过程

概述

  1. 类加载机制：虚拟机把类的结构从Class文件（也可以是二进制字节流如Cglib动态生成的类）加载到内存，经过校验、解析、初始化后，最终形成能够直接被虚拟机使用的Java类型。
  2. 运行期类加载：类型的加载、 连接、初始化都是在运行期间动态完成的。这样虽然让类加载增加了性能开销，但是提供了极高的扩展性和灵活性。
  3. 动态扩展：Java的动态扩展就是依赖运行期动态加载、动态 连接完成的。比如运行期间再指定实现类；可以运行时从网络等地方加载一个二进制流作为程序代码的一部分。基本的Applet、JSP、OGSI都依赖Java运行期类加载。

类加载过程

加载 —–> 连接（验证、准备、解析）—–> 初始化 ——> 使用 ——> 卸载

4.1、加载

  1. 通过一个类的全限定名来获取定义此类的二进制字节流 （不一定读取.class文件、比如从jar包、网络、动态代理生成、JSP生成、数据库、加密Class等获取）
  2. 将这个字节流所代表的结构转化为方法区运行时的数据结构，如何实现JVM无约束。
  3. 在堆内存中创建一个代表这个类的java.lang.Class对象，作为程序访问方法区中的类型数据的外部接口，通过Class对象可以获取方法区类的结构。

注意：加载字节码到方法区，注意运行时常量池、字符串常量池等就是此过程生产的。还有一点就加载和连接只有开始时间保持先后顺序。

数组与非数组

  • 非数组：是开发人员可控的阶段，非数组可以由用户自定义的类加载器完成加载，实现根据自己的想法来赋予应用程序运行代码的动态性。
  • 数组：数组由Java虚拟机直接在内存中构建出来的，所以不涉及加载。
    • 数组类型为引用类型：数组的类型还是需要依靠加载器加载的，会使用正常的加载过程区加载。数组会标识在该组件类型的类加载器的类名称空间上。
    • 数组类型为非引用类型：那么数组直接与引导类加载器相关联。

概述：关于类的加载，JVM规范没有规定，但是类的初始化时间，JVM规定如下。

会导致类初始化的情况（类的主动引用）

  • 创建对象：虚拟机启动时初始化主类、new 创建对象、反射创建对象
  • 调用到类：对类进行反射、调用类的静态成员和方法(final 常量除外)
  • 父类相关：优先初始化父类；若子类访问父类的静态变量，只会触发父类的初始化

不会导致类初始化的情况（类的被动使用）

  • 子类：子类引用父类的静态变量，子类不会初始化

  • 常量：引用常量(常量在链接阶段就存入调用类的常量池中了)

  • 数组：数组类型 的那个类，在数组创建时不会被初始化。

  • 加载：执行类加载器的 loadClass 方法不会初始化

注意

  1. 验证类是否被初始化，可以看该类的静态代码块是否被执行
  2. Class.forName() 参数二为 true，会初始化，为false不会

4.5、类的卸载

类卸载的条件

  • 该类所有的实例都已经被回收
  • 加载该类的ClassLoader对象被回收（所以Java自带的加载器加载的永远不会回收）
  • Class对象没有任何地方被引用

Class对象对象卸载后，方法区的运行时数据结构也会消亡。

5、类加载器

Java虚拟机设计团队有意把类加载阶段中的“通过一个类的全限定名来获取描述该类的二进制字节流”这个动作放到Java虚拟机外部去实现，以便让应用程序自己决定如何去获取所需的类。实现这个动作的代码被称为“类加器” （ ClassLoader）

5.1、Java的类加载器

1、类的唯一性

概念：类由本身和加载该类的加载器确定。数组也是同理，数组与数组类型的加载器和数组类型有关。

2、Java自带的类加载器

作用：把JVM编译的class二进制流加载，把类的结构转换成运行时的数据结构存到方法区，并且在堆中生成一个class对象作为访问方法区的接口。

名称 加载的类 说明
Bootstrap ClassLoader（启动类加载器） JAVA_HOME/jre/lib（核心Jar包） 无法直接访问（C++实现的）
Extension ClassLoader(拓展类加载器) JAVA_HOME/jre/lib/ext（扩展Jar包） 上级为Bootstrap
Application ClassLoader(应用程序类加载器) classpath（我们写的Jar包） classpath 上级为Extension
自定义类加载器 自定义 自定义 上级为Application

类加载器是谁加载的？

启动类加载器：当jvm启动时，一块特殊的C++机器码会执行，这块特殊的机器码叫做启动类加载器。启动类加载器不是java类，而其它的类加载器是java类。

注意：所以也可以把类加载器看成两种，一种是C++编写根加载器，程序启动时启动。另一种是实现抽象类 ClassLoader 实现的类加载器，由根加载器加载，只不过Java预先实现了2个。

什么时候进行加载

类加载器并不需要等到某个类被“首次主动使用”时再去加载它，jvm规范允许类加载器在预料某个类将要被使用时就预先加载它，如果在预先加载的过程中遇到了.class文件缺失或者存在错误，那么类加载器将在程序首次主动使用该类时才报告错误（LingkageError错误），如果一直都没有被程序主动使用，那么类加载器就不会报告错误。

5.2、双亲委派机制

概述：

  1. 类加载器**需要加载类时**，会不断向上委托父类加载器，直到（启动类加载器）根加载器。
  2. 根加载器寻找有没有需要加载的类，能加载结束，没有则抛出异常通知子类去加载，
  3. 重复步骤②，直到加载成功或者全部失败。

优点1：自定义String类会报错，可以保护系统核心类不被覆盖

优点2：避免全限定命名的类重复加载

优点3：类由 全限定名 和 类加载器共同确定，使用不同的类加载器就可以为相同名称的类创建额外的命名空间

破坏双亲委派模式

  1. 双亲委派模型的第一次“被破坏”其实发生在双亲委派模型出现之前——即JDK1.2面世以前的“远古”时代。当时ClassLoader接口已经存在，可以自定义加载器通过loadClass()方法 实现自己定义加载策略。当双亲委派机制出现后，为了兼容之前的代码，只好定义个protected 的 findClass 方法，避免父类的双亲委派被覆盖。
  2. 双亲委派模型的第二次“被破坏”是由这个模型自身的缺陷导致的，当系统基础类需要调用用户代码时，就由高级的类加载器需要加载用户的类，此时也会破坏双亲委派模式。比如JDBC的规范需要加载第三方提供的 mysql -connection 包，此时就破坏了双亲委派机制。
  3. 双亲委派模型的第三次“被破坏”是由于用户对程序动态性的追求而导致的，这里所说的“动态性”指的是一些非常“热”门的名词：代码热替换（Hot Swap）、模块热部署（Hot Deployment）等。每个模块都有自定义的类加载器，通过替换加载器和代码实现 热部署。

5.3、线程上下文加载器

线程上下文类加载器：当前加载当前线程的类加载器。

  1. Java存在第三方接口（SPI），属于核心库，在rt.jar包中，需要根类加载去加载
  2. 而这些接口的实现，常见的比如JDBC一般存放在ClassPath路径下，
  3. SPI需要调用第三方包，但是SPI是通过根加载器去加载的，于是就只能根加载器委托子类加载器去加载违背了双亲委派机制，而根加载器无法直接获取子类加载器，只能通过线程上下文获取类加载器。
  4. Thread.setContextClassLoaser()方法设置子类的加载器，这样系统就可以通过线程上下文获取类加载器。

7、运行期优化

该过程经过Cglib等技术动态生成的字节码也能享受到。

7.1、即时编译

JVM 将执行状态分成了 5 个层次：当一段字节码多次被解释时就会调用即时编译器进行优化

  • 0层：解释执行，用解释器将字节码翻译为机器码
  • 1层：使用 C1 即时编译器编译执行（不带 proﬁling）
  • 2层：使用 C1 即时编译器编译执行（带基本的profiling）
  • 3层：使用 C1 即时编译器编译执行（带完全的profiling）
  • 4层：使用 C2 即时编译器编译执行

即时编译器（JIT）与解释器的区别

  • 解释器
    • 将字节码解释为机器码，下次即使遇到相同的字节码，仍会执行重复的解释
    • 是将字节码解释为针对所有平台都通用的机器码
  • 即时编译器
    • 将一些被多次调用的字节码编译为机器码，并存入 Code Cache，下次遇到相同的代码，直接执行，无需再编译
    • 根据平台类型，生成平台特定的机器码

对于大部分的不常用的代码，我们无需耗费时间将其编译成机器码，而是采取解释执行的方式运行；另一方面，对于仅占据小部分的热点代码，我们则可以将其编译成机器码，以达到理想的运行速度。 执行效率上简单比较一下 Interpreter < C1 < C2，总的目标是发现热点代码（hotspot名称的由来），并优化这些热点代码

7.2、逃逸分析

逃逸分析（Escape Analysis）简单来讲就是，Java Hotspot 虚拟机可以分析新创建对象的使用范围，并决定是否在 Java 堆上分配内存的一项技术

逃逸分析的 JVM 参数如下：

  • 开启逃逸分析：-XX:+DoEscapeAnalysis
  • 关闭逃逸分析：-XX:-DoEscapeAnalysis
  • 显示分析结果：-XX:+PrintEscapeAnalysis

逃逸分析技术在 Java SE 6u23+ 开始支持，并默认设置为启用状态，可以不用额外加这个参数

对象逃逸状态

全局逃逸（GlobalEscape）：即一个对象的作用范围逃出了当前方法或者当前线程

  • 对象是一个静态变量
  • 对象是一个已经发生逃逸的对象
  • 对象作为当前方法的返回值

参数逃逸（ArgEscape）：即一个对象被作为方法参数传递或者被参数引用，但在调用过程中不会发生全局逃逸，这个状态是通过被调方法的字节码确定的

没有逃逸：即方法中的对象没有发生逃逸

逃逸分析优化

  • 锁消除：既然没有逃逸，就不会有线程安全问题，没必要上锁了
  • 标量替换：对象是聚合量，它又可以被进一步分解成标量，将其成员变量分解为分散的变量，这就叫做标量替换。
  • 栈上分配：当对象没有发生逃逸时，就表明其使用的范围只在该作用域，所以只需要直接使用该对象分解成的标量即可。换句话话说，就相当于该对象是在栈上分配的。

7.3、方法内联

内联函数

内联函数就是在程序编译时，编译器将程序中出现的内联函数的调用表达式用内联函数的函数体来直接进行替换

触发的条件：拥有一下条件则容易触发内联

  1. 热点代码
  2. 方法体不能太大
  3. 如果希望方法被内联，尽量用private、static、final修饰，这样jvm可以直接内联。如果是public、protected修饰方法jvm则需要进行类型判断，因为这些方法可以被子类继承和覆盖，jvm需要判断内联究竟内联是父类还是其中某个子类的方法。
