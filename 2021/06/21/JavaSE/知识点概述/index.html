1、char和string

  • 操作少量的数据: 适⽤ String
  • 单线程操作字符串缓冲区下操作⼤量数据: 适⽤ StringBuilder

2、方法问题

重载与重写

  1. 静态方法可以重载但是不可以重写
  2. 方法重写约束:
    • 方法必须具有相同的名称和参数
    • 返回值类型、异常类型要和父类一致，或为父类返回类型的子类
    • 访问权限要比父类的高或者相等
    • 无法重写static、final、private方法，因为static方法是类方法，不是成员方法
  3. 子类对象实例化时不会实例化父类对象
  4. 方法的签名：方法的名字和参数构成方法的签名，与返回值无关，重载需要签名不同

值传递：Java里方法的参数传递方式只有一种：值传递。 将实际参数值的副本（复制品）传入方法内，而参数本身不受影响。

3、构造器

特点：不能重写可以重载、没有修饰符、子类构造器必须调用父类构造器（默认自动调用无参）

构造器调用构造器：最多调用一个；必须放在最上面

初始化顺序：

  1. 声明成员变量的默认初始化
  2. 显式初始化、多个初始化块依次被执行（同级别下按先后顺序执行）
  3. 构造器再对成员进行初始化操作
  4. 通过”对象.属性”或”对象.方法”的方式，可多次给属性赋值

注意：构造器被调用后，会先判断是否调用了其他构造器

  • 若调用了，则转到另一个构造器执行
  • 若没调用，先调用初始化块，在调用构造器剩下的部分

静态代码块：优先于类的构造，只能对静态成员变量初始化，从上而下执行

4、static和final

final：

  • 变量：不能修改，直接定义或者构造函数处赋值
  • 方法：不能被重写，可以重载
  • 类：不能被继承

static

  • 变量：不再属于某个对象，而是属于类被共享
  • 方法：独立于类，静态方法不能调用非静态字段，类和对象都可以调用静态方法。

5、多态问题

多态：多态是运行时实现的，一个引用变量在运行时才能确定指向哪个对象。

数组向上转型的问题：数组对象转型问题，数组向上转型之后，只能添加子类对象而不能添加父类对象，因为子类和父类引用都是指向该地址，添加父类对象，就相当于子类指向父类对象属于向下转型不安全。

6、枚举类

  • 枚举是一个特殊的类，一般表示一组常量，参数默认用public static final 修饰，
  • 枚举类构造器只能是private，
  • 枚举类初始化对象，里面成员变量全部初始化

7、接口和抽象

接口：

  • 接口只含 静态常量值，抽象方法，jdk8新增默认方法，静态方法，jdk9允许私有
  • 接口继承规则
    • 超子类冲突超类优先：如果超类提供了一个具体方法，子类方法签名一致的默认方法将被忽略
    • 超类之间冲突： 子类需要重写

抽象：抽象只是可能含有抽象方法，其他和普通类完全一致。

注意点：

  • 抽象类可以实现多个接口，只能继承实现一个抽象类

  • 一个接口可以继承多个接口。

  • 从设计层⾯来说，抽象是对类的抽象，是⼀种模板设计，⽽接口是对行为的抽象，是⼀种行

    为的规范。

8、内部类

  • 局部内部类的this指向外部，只能获得外部不改变的值
  • 普通成员内部类不能含静态方法属性，可以重写外部方法。
  • 静态成员内部类，静态方法只能调静态属性和静态方法，和外部资源。

14、泛型原理

原理：编译器把数据存为Object，再通过泛型知道了数据的类型，自动帮我们把数据转换回来

泛型多态性问题：父类被重写的方法的参数类型被擦除，不构成重写，如何保持多态？

描述：子类继承泛型父类（泛型参数string），并且重写方法create（string str），由于泛型擦除，子类父类方法参数不一致，被重载了，如何恢复原来的重写多态？

桥方法解决：实际调用父类的擦除的方法，但是此方法内部调用子类的方法，也就是说实际调用子类的方法。

协变问题：非继承关系下自动继承Object，泛型擦除导致的方法重复

解决：JVM会采用重写的方法

  3. 不能创建参数化类型的数组：数组不知道里面具体要加入的类型，但是由于Obejct[] 是父类，

  4. 不能 new 泛型变量：new不知道要创建的具体类型，无法直到调用哪个构造器。

  6. 不能捕获抛出泛型类实例：

  7. 泛型类的泛型变量不能使用static：静态变量初始化时无法获得变量类型

22、反射

反射：类加载器加载完类之后，堆里面就会生成一个Class类型的对象（一个类只有一个Class对象，JVM创建，每个类的实例都会记得自己是由哪个 Class 实例所生成），这个对象包含类的完整信息，通过这个对象来访问类的过程就是反射

  • 正常方式： 引入包名—->new实例化——>取得实例化对象
  • 反射方式： 实例化对象—–>getClass()方法——->得到完整的”包类”名称

反射用途：

  • 运行时取得类的内部信息，
  • 运行时检查对象，直接操作对象的内部属性和方法。
  • 实现泛型数组操作代码

29、同步、异步，阻塞和非阻塞

异步同步关心的是消息通信机制

  • 同步：发出一个请求时，在没有得到结果之前，该请求就不返回直到得到结果
  • 异步：请求后，这个调用就直接返回了，等结果出来后通过回调函数、通知等方法 返回结果。

阻塞非阻塞关心的是程序的状态

  • 阻塞：发出一个请求，得到结果之前，会一直被阻塞。
  • 非阻塞：发出一个请求，得到结果之前，自己可以做其他事情。

30、BIO、NIO、和AIO

BIO：同步阻塞式IO，服务器实现模式为一个连接一个线程（多线程），即客户端有连接请求时服务器端就需要启动一个线程进行 IO 操作。虽然此时服务器具备了高并发能力，但是随着开启的线程数目增多，将会消耗过多的内存资源，导致服务器变慢甚至崩溃，NIO可以一定程度解决这个问题。

NIO：同步非阻塞式IO，服务器实现模式为 IO 多路复用，只需要开启一个线程就可以处理来自多个客户端的IO事件，客户端发送的连接请求都会注册到多路复用器上，多路复用器轮询到连接有I/O请求时才启动一个线程进行处理。

  • 若服务端监听到客户端连接请求，便为其建立通信套接字(java中就是通道)，然后返回继续监听，若同时有多个客户端连接请求到来也可以全部收到，依次为它们都建立通信套接字。
  • 若服务端监听到来自已经创建了通信套接字的客户端发送来的数据，就会调用对应接口处理接收到的数据，若同时有多个客户端发来数据也可以依次进行处理。
  • 监听多个客户端的连接请求和接收数据请求同时还能监听自己时候有数据要发送。

AIO(NIO.2)：异步非阻塞式IO，服务器实现模式为一个有效请求一个线程，客户端的I/O请求都是由OS先完成了再通知服务器应用去启动线程进行处理。

总结

  • NIO适合处理连接数目特别多，但是连接比较短（轻操作）的场景，Jetty，Mina，ZooKeeper等都是基于java nio实现。
  • BIO方式适用于连接数目比较小且固定的场景，这种方式对服务器资源要求比较高，并发局限于应用中。
