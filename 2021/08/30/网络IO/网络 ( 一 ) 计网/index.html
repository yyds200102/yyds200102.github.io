1.6、实际应用

浏览器输入地址之后：

  1. 应用层：发出一个 HTTP 请求
  2. 运输层：给该HTTP请求 添加上TCP首部（可靠传输、区分应用进程），把请求包装为 TCP报文段
  3. 网络层：把 TCP报文段 添加上 IP 首部（能被路由器转发，在互联网传输），包装为 IP数据报
  4. 数据链路层：把 IP报文段 添加上 首部（传输）和尾部 （是否误码），包装为帧
  5. 物理层：把帧看做比特流，添加前导码；然后变换为信号发送。

信号发送到路由器：

  1. 物理层：把信号变化为比特流，去掉前导码，剩下的是帧，然后交付。
  2. 数据链路层：去掉帧的首部和尾部，剩下IP 数据报，交付网络层。
  3. 网络层：解析IP数据报的首部，提取目的网络地址，查找路由表确定转发路径。
  4. 接着 经过 数据链路层 和 物理层 重新变为信号，继续往下传递。

一直循环直到到达Web服务器

  1. 进过不断解析，变为原本的 HTTP 请求
  2. 根据请求进行处理，把结果封装为 HTTP 响应报文
  3. HTTP 响应报文在Web 服务器不断封装，直到成为信号，然后传输。

3.4、可靠传输

注意：由于数据链路层相隔距离短，而且发送接收速度一般都是已经由硬件决定了快慢，所以接收窗口和发送窗口的大小不会改变。

3.5、可靠传输：停止等待协议

概述：接收窗口大小为1，发送方只有收到确认才能继续发送下一个帧。请求编号防止重复接收，响应编号防止重复发送。

注意：为了防止第一次ACK丢失导致接收方一直发送数据，每次收到请求，即使已经确认过了，也还要继续发送ACK

  • 确认与否认：接收后需要发送确认收到，或者收到误码的回复。
  • 超时重传：防止丢失消息
  • 确认丢失：给请求编号，防止重传后 接收方 重复接收消息。
  • 确认迟到：给应答编号，防止重传后接收方多次应答导致发送方确认后发的消息。点对点的不需要。但是广播信道需要。

3.6、可靠传输：回退N帧

概念：是一种ARQ（滑动窗口）协议，接收窗口还是1。通过ACK错误的帧序号实现回退。

  • 停止等待：发送后就必须等待，直到超时重传或者接收到ACK
  • 回退N帧：发送方可以利用滑动窗口不断发送数据（不考虑ACK），发送方通过累计确认（ACK）来移动窗口，接收方通过重复发送错误的帧序号实现窗口的回退。

回退N帧的逻辑：根据接收方的ACK确定，接收方收到第5个错误，就会发送ACK=4 表明4之前的正确，注意只要收到一个错误就发送一个ACK=4。接受方收到重复的ACK=4，就会知道第5个出现问题，于是回退到第5个重发。

保留超时等待：防止所有数据全部失败，这样一个ACK都没有，如果没有超时重传，会一直等待。

回退N帧协议窗口的上限

注意：窗口的大小一定要在编号范围以内，否则无法区分数据是否重复

3.7、可靠传输：选择重传SR

概念：没有累积确认，接收窗口大于1。依赖自动重传机制实现错误消息重传。

  • 回退N帧：接收窗口WR只能为1，也就是说只能按照顺序接收，导致效率低下。
  • 选择重传：接收窗口WR大于1。发送方可以不考虑ACK一直发送直到滑动窗口满了，每个帧都必须ACK才会移动窗口。接收方也有一个滑动窗口，用来维护接收的帧，只有接收到正确的帧，窗口才能移动。对于中途出错的帧，不发生ACK，让发送方按时进行重传。

特点：接收窗口向前滑动必须等最后面的确认了才可以。接收窗口如果太大，浪费资源，因为标志位限制发送数量，所以用于接收不到那么多，即窗口永远满不了。

1、CSMA/CD 解决碰撞

强化碰撞：一旦检测出碰撞，除了立即停止发送帧外，还需要发送32或者48比特的认为干扰信号，让足够多的站点检测出发生了碰撞。

争用期：争用期的大小为信道返回的时间和，只有经过争用期才知道是否发生了碰撞。

最小帧长：发送完就不再检测碰撞，所以必须保证发送时间大于争用期，所以长度不能太短。

最大帧长：防止长期占用信道，防止接收方缓冲区接收不了。

退避时间：发送失败，需要退避，避免多个主机争抢信道，导致始终无法使用。

2、CSMA/CA 解决碰撞（无限网络）

CA：碰撞避免 而非碰撞检测CD

退避算法：保证单个连接主机不会长时间占用信道。内部维护退避计时器，退避计时器只有信道为空闲且经过短帧间隔时间DIFS才会计时。

3、ARP协议

概述：根据目的IP地址找出下一次应该发往的目的地的MAC地址。

  1. 先从主机的ARP高速缓存表中（存有IP和MAC的对应关系）查找
  2. 未找到就发送ARP请求报文（广播的形式）请求MAC地址
  3. 符合条件的主机会发送响应高速自己的MAC地址
  4. 主机接收到请求，更新ARP高速缓存表

注意：ARP请求只能在一段链路发出，不能超过范围跨越。

已经有了IP地址，为什么还需要MAC地址？

在数据链路层传输，无法识别IP地址。

5.5、TCP的拥塞控制

注意：拥塞窗口是几，就能发送几个数据报文段。

1、慢开始和拥塞避免

  1. 初始 拥塞窗口为 1，ssthresh 为16 ，一开始只能发送 1个数据报文段
  2. 每次 发送成功时， 拥塞窗口 变为原来2倍
  3. 当 拥塞窗口内发送的报文数量 ＞ ssthresh 时，拥塞窗口 每次 加1
  4. 当某次报文发送时，出现超时重传，系统判断拥塞。于是 拥塞窗口为 1，ssthresh 变为 当前 拥塞窗口的 一半。

2、快开始和快恢复

上面有一个缺陷：有时候超时重传并不是拥塞引起的，这样导致效果降低。

5.8、ARQ协议

自动重传请求（Automatic Repeat-reQuest，ARQ）是OSI模型中数据链路层和传输层的错误纠正协议之⼀。它通过使用确认和超时这两个机制，在不可靠服务的基础上实现可靠的信息传输。

概念：发送方发送一段时间之后如果没有收到确认，就需要重发。

停止等待ARQ协议

原理：每发完⼀个分组就停止发送，等待对方确认（回复ACK）。

特点：简单，但是信道利用效率很低

  1. 服务端重复收到消息：服务端会直接丢弃，并且发送确认ACK防止 客户端 阻塞。
  2. 客户端重复收到确认：直接丢弃

连续ARQ协议

原理：使用下面的滑动窗口实现，根据接收方窗口大小分为回退N帧、超时重传

5.9、三次握手

报文分类

  • SYN报文段（SYN=1的报文段）：不能携带数据，但需要消耗掉一个序号，初始化seq。
  • ACK报文段可以携带数据，但是如果不携带数据则不消耗序号。

seq 和 ack

  • 发送时 的 seq 为x，接收时就会收到 ack = x + 1
  • 第一次发送 seq = 1，第二次发送就是 seq = x + 1

状态

  • SYN-SENT（同步已发送状态）状态。
  • LISTEN (监听)
  • SYN-RCVD（同步收到）状态。
  • ESTABLISHED（已建立连接）状态。

三次握手：最开始的时候客户端和服务器都是处于CLOSED状态。主动打开连接的为客户端，被动打开连接的是服务器

  1. TCP服务器创建传输控制块TCB，进入监听状态准备接收连接，
  2. 第一次握手：TCP客户端创建传输块TCB，向服务器发送连接请求报文。
  3. 第二次握手：TCP服务器收到请求报文同意连接，然后发送确认报文。
  4. 第三次握手：TCP客户进程收到确认后，还要向服务器给出确认报文。此时，TCP连接建立。
  5. 当服务器收到客户端的确认后也进入ESTABLISHED状态，此后双方就可以开始通信了。

传输失败的情况分析

  1. 第一次握手失败：客户端会不断重试直到服务端收到。
  2. 第二次握手失败：客户端一直得不到ACK，重新第一次握手直到服务端第二次握手成功
  3. 第三次握手失败：服务端一直得不到ACK，重新第二次握手直到客户端第三次握手成功

为什么TCP客户端最后还要发送一次确认呢？

两次： 第一次握手成功服务端建立连接，第二次握手成功客户端建立连接。

三次：第二次握手成功客户端建立连接，第三握手成功服务端建立连接。

注意：仔细比对发送，只有两次握手的情况下，如果第二次握手失败，客户端会一致请求，如果只有二次握手，那么服务端将会不停的建立连接。如果是三次握手，那么就不会触发。

5.10、四次挥手

报文

  • FIN报文（FIN = 1）段即使不携带数据，也要消耗一个序号。

状态

  • ESTABLISHED（已建立连接）状态。
  • FIN-WAIT-1（终止等待1） FIN-WAIT-2（终止等待2） TIME-WAIT（时间等待）状态。
  • CLOSE-WAIT（关闭等待）状态。 LAST-ACK（最后确认）状态，

数据传输完毕后，双方都可释放连接。最开始的时候，客户端和服务器都是处于ESTABLISHED状态，然后客户端主动关闭，服务器被动关闭。

  1. 第一次挥手：客户端进程发出连接释放报文，并且停止发送数据。客户端进入FIN-WAIT-1（终止等待1）状态。
  2. 第二次挥手：服务器收到连接释放报文，发出确认报文（ACK不携带数据，不消耗序号）；服务端就进入了CLOSE-WAIT（关闭等待）状态。
  3. 客户端收到服务器的确认请求后，此时，客户端就进入FIN-WAIT-2（终止等待2）状态，等待服务器发送连接释放报文（在这之前还需要接受服务器发送的最后的数据）。
  4. 第三次挥手：服务器将最后的数据发送完毕后，就向客户端发送连接释放报文。此时，服务器就进入了LAST-ACK（最后确认）状态，
  5. 第四次挥手：客户端收到服务器的连接释放报文后，必须发出确认，此时，客户端就进入了TIME-WAIT（时间等待）状态。注意此时TCP连接还没有释放，必须经过2 * MSL（最长报文段寿命）的时间后，当客户端撤销相应的TCB后，才进入CLOSED状态。
  6. 服务器只要收到了客户端发出的确认，立即进入CLOSED状态。同样，撤销TCB后，就结束了这次的TCP连接。可以看到，服务器结束TCP连接的时间要比客户端早一些。

总结：客户端释放 ——> 服务端确认(仍发送收尾数据) ——-> 服务端释放报文 ——-> 客户端确认 ———> 服务端收到后撤销TCB ——> 客户端跟着撤销TCB

注意：服务器先关闭的，然后才是客户端

为什么客户端最后还要等待2MSL？

  1. 保证客户端的确认报文能够到达服务器。否则服务器会重复发送释放连接的报文；这样就会导致服务端，收到RST（表示复位、用来异常的关闭连接）而不是ACK。 Server就会以为是连接错误把问题报告给高层。这样的情况虽然不会造成数据丢失，但是却导致TCP不符合可靠连接的要求。
  2. 让本次连接其他因为速度慢而滞留的报文段失效。因为客户端可能关闭后重连，这样导致接收到错误上次连接的报文段。

为什么建立连接是三次握手，关闭连接确是四次挥手

挥手：因为服务器收到请求之后，就需要立即发送确认报文，否则客户端会认为没送到，会不断发送。但是此时服务器还有一些数据发送，所以还需要继续传输，等传输结束才是释放报文，所以确认要2个阶段，就多了一次请求。

握手：没有数据需要发送，只需要建立连接即可。

如果已经建立了连接，但是客户端突然出现故障了怎么办？

TCP还设有一个保活计时器，显然，客户端如果出现故障，服务器不能一直等下去，白白浪费资源。服务器每收到一次客户端的请求后都会重新复位这个计时器，时间通常是设置为2小时，若两小时还没有收到客户端的任何数据，服务器就会发送一个探测报文段，以后每隔75秒发送一次。若一连发送10个探测报文仍然没反应，服务器就认为客户端出了故障，接着就关闭连接。

7.1、长连接和短连接

短连接

HTTP/1.0中默认使用短连接：客户端和服务器每进行一次HTTP操作，就建立⼀次连接，任务结束就中断连接。当客户端浏览器访问的某个HTML或其他类型的Web页中包含有其他的Web资源（如JavaScript⽂件、图像⽂件、CSS⽂件等），每遇到这样⼀个Web资源，浏览器就会重新建⽴⼀个HTTP会话

长链接

⽽从HTTP/1.1起，默认使用长连接：可以保持连接特性，网页打开后，连接不会关闭，这样便于下一次请求的速度。

总结：HTTP协议的长连接和短连接，实质上是TCP协议的长连接和短连接。

7.2、HTTP是不保存状态的协议如何保存用户状态

  • Session存储的是用户的信息，在客户端存储。

Cookie 被禁⽤怎么办

最常⽤的就是利用 URL 重写把 Session ID 直接附加在URL路径的后面

7.3、HTTP 1.0 和 1.1

  1. 长连接与短连接
  2. 错误状态码：在HTTP1.1中新增了24个错误状态响应码，如409（Conflict）表示请求的资源与资源的当前状态发生冲突；410（Gone）表示服务器上的某个资源被永久性的删除。
  3. 缓存处理：在HTTP1.0中主要使⽤header⾥的If-Modified-Since,Expires来做为缓存判断的标准，HTTP1.1则引⼊了更多的缓存控制策略例如Entity tag，If-Unmodified-Since, If-Match，If-None-Match等更多可供选择的缓存头来控制缓存策略。
  4. 带宽优化及网络连接的使⽤ :HTTP1.0中，存在⼀些浪费带宽的现象，例如客户端只是需要某个对象的⼀部分，⽽服务器却将整个对象送过来了，并且不支持断点续传功能，HTTP1.1则在请求头引⼊了range头域，它允许只请求资源的某个部分，即返回码是206（PartialContent），这样就⽅便了开发者⾃由的选择以便于充分利用带宽和连接。

7.4、URL 和 URI

  • URI(Uniform Resource Identifier) 是统⼀资源标志符，可以唯⼀标识⼀个资源。

  • URL(Uniform Resource Location) 是统⼀资源定位符，可以提供该资源的路径。它是⼀种具体的 URI，即 URL 可以用来标识⼀个资源，而且还指明了如何 locate 这个资源。

URI的作用像身份证号⼀样，URL的作用更像家庭住址⼀样。URL是⼀种具体的URI，它不仅唯⼀标识资源，⽽且还提供了定位该资源的信息。

7.5、HTTP 和 HTTPS

端口

HTTP 默认 80 HTTPS 默认 443

安全

  • HTTP协议运行在TCP之上，所有传输的内容都是明⽂，客户端和服务器端都无法验证对方的身份。
  • HTTPS是运行在SSL/TLS之上的HTTP协议，SSL/TLS 运行在TCP之上。所有传输的内容都经过加密，加密采用对称加密，但对称加密的密钥用服务器方的证书进行了非对称加密。

7.6、浏览器网址加载过程

网站是如何进行访问的

浏览器输入地址之后：检查本机的C:\Windows\System32\drivers\etc\hosts配置文件下有没有这个域名映射

  • 有，则直接返回对应的IP地址
  • 没有，去DNS服务器（管理域名的地方）

浏览器向 Web 服务器发送一个 HTTP 请求（携带该域名生效的cookie）（本质TCP连接）

  1. 应用层：发出一个 HTTP 请求
  2. 运输层：给该HTTP请求 添加上TCP首部（可靠传输、区分应用进程），把请求包装为 TCP报文段
  3. 网络层：把 TCP报文段 添加上 IP 首部（能被路由器转发，在互联网传输），包装为 IP数据报
  4. 数据链路层：把 TCP报文段 添加上 首部（传输）和尾部 （是否误码），包装为帧
  5. 物理层：把帧看做比特流，添加前导码；然后变换为信号发送。

信号发送到路由器：

  1. 物理层：把信号变化为比特流，去掉前导码，剩下的是帧，然后交付。
  2. 数据链路层：去掉帧的首部和尾部，剩下IP 数据报，交付网络层。
  3. 网络层：解析IP数据报的首部，提取目的网络地址，查找路由表确定转发端口。
  4. 接着 经过 数据链路层 和 物理层 重新变为信号，继续往下传递。

一直循环直到到达Web服务器：

  1. 进过不断解析，变为原本的 HTTP 请求
  2. 根据请求进行处理，得到 HTTP 响应报文
  3. HTTP 响应报文在Web 服务器不断封装，直到成为信号，然后传输。

7.7、Socket

  • TCP把连接作为最基本的对象，每一条TCP连接都有两个端点，这种断点我们叫作套接字（socket）
  • 它的定义为端口号拼接到IP地址即构成了套接字，例如，若IP地址为192.3.4.16 而端口号为80，那么得到的套接字为192.3.4.16:80。

7.8、OSI概述

三种交换方式

  • 电路交换： 建立 —-> 通话 —-> 释放。时延小，无冲突，但是连接时间长，独占线路。
  • 分组交换：发送 —-> 路由转发 —-> 接收。转发时延，有冲突，但是信道复用，线路利用率高。
  • 报文交换：和分组交换类似，但是不限制数据大小，所以需要缓存空间足够大，现在基本不使用。

1、物理层

  • 解决的问题：如何在计算机的传输媒体上传输比特流。
  • 对数据的封装：把帧看做比特流，添加前导码；然后变换为信号发送。

2、数据链路层

解决的问题： 实现数据的可靠传输、差错检测。

对数据的封装：把 网络层 的 IP报文段 添加上 首部（传输）和尾部 （是否误码）， 封装装为帧 ，传给物理层。

如何实现数据的传输

  1. 透明传输：帧定界符用于区分每个帧（数据），但是数据可以与帧定界符重复。
    • 字节：只有有转移字符或者定界符，就在后面加上转义字符。
    • 比特：每5个1后面添加一个0
  2. 差错检测：保证数据是否完整的接收到
    • 奇偶校验：通过一位来记录奇偶的个数；存在漏检问题，
    • 循环冗余检测CRC：根据规则计算出冗余码，每次收到计算后与携带的比对。
  3. 不可靠传输：丢失错误的帧，对于其他问题一概不管，
  4. 可靠传输：按序接收到客户端传输的完整数据。
    • 停止等待：确认应答和否认应答；超时重传；确认丢失（发送方编号，丢失重复的）；确认迟到（应答编号，防止迟到的确认把后发的数据确认了，注意广播信道才存在）。
    • 回退N帧：一种ARQ协议，接收窗口大小只能为1，超时等待，连续发送，累积确认机制。
    • 选择重传：回退N帧的改进，接收窗口WR大于1，超时等待，连续发送，没有累积确认，只能手动确认。

注意：

  1. 可靠传输不局限于数据链路层，其他层都可以选择实现。
  2. 停止等待发送后就必须等待ACK，回退N帧和选择重传都可以连续发送

MAC、IP、ARP

  • MAC地址：以太网的MAC子层使用（数据链路层），也称作物理地址。
  • IP：TCP/IP 网际层的使用地址（网际层）
  • ARP：属于 TCP/IP 体系结构的网际层，作用是根据IP地址获取到MAC地址（网际层）。发送ARP请求报文（广播的形式）请求MAC地址，得到后更新ARP高速缓存表，注意不能跨域。

注意：数据传输过程中 目的和源IP 地址不变，目的和源MAC地址一直变化。在数据链路层传输，无法识别IP地址，所以设置MAC地址和ARP协议。

集线器和交换机

  • 集线器：仍然相当于一个总线网，多个主机使用一条信道，有碰撞问题。所以每个集线器和连接的主机之间是一个碰撞域。
  • 交换机：都有多个网络接口（MAC地址），每个接口都可以与一个主机或以太网连接，全双工模式。所以各个连接的主机之间没有碰撞，因为每个主机和一个交换机的MAC地址交换数据，所以不存在冲突

3、网络层

  • 解决的问题：数据在网络上的传输地址问题。
  • 对数据的封装：把 TCP报文段 添加上 IP 首部（能被路由器转发，在互联网传输），包装为 IP数据报

两种服务

  • 虚电路：需要连接，按序传输；传输顺序固定，一处故障全部故障。
  • 数据报：不需要连接，不按序传输；传输顺序不固定，一处故障就换一处。

IP数据报的发送与转发

  • 发送：先判断是否在同一个网络‘
    • 在同一网络：直接交付
    • 不在同一网络：间接交付，传输给默认网关（路由器），由默认网关帮忙转发。
  • 转发：转发只转发到其他路由
    • 检查IP数据报首部是否错误
    • 检查IP数据报目的地址是否在路由表中有匹配的。有就转发下一跳，没有就丢弃。
    • NAT是在你的网关处做的标记，网关会记录你的内网IP和外网IP的映射关系，以便数据包返回后通过这个映射找到你的主机。

路由选择协议

静态路由由人工配置；动态路由是通过路由选择协议自动获取的，分为内部协议与外部协议、

  • 内部协议
    • RIP：使用跳数作为距离，16就认为不可达。存在坏消息传的慢的情况
    • OSPF协议：基于迪杰斯特拉的算法计算距离，通过每10秒发送问候分组，维持关系。一旦40s未收到问候，则认为不可达。
  • 外部协议
    • 边界网关协议BGP：让每个自治系统内选出一个代言人，代言人之间建立TCP连接，互相交流，最后根据自己的策略选择网络。

VPN和NAT

  • VPN：本机通过公网和一个内网的公网路由建立连接（数据会加密），这样就可以通过该路由访问内网。
  • NAT：结合端口号 + 私有地址，对每个内网的每个端口号的请求都通过NAT转换分配不同的的端口号，这样就可以实现一个公网地址供多个内网地址上网，节约IPv4地址。

4、运输层

  • 解决的问题：运输层是端到端的服务，为应用进程之间提供服务。
  • 对数据的封装：给该HTTP请求 添加上TCP首部（可靠传输、区分应用进程），把请求包装为 TCP报文段
  • 复用和分用：多个应用进程同时发出的数据会封装成一个大的数据报文，在网络传输实现复用。当传输到目的主机后，就会分开，实现分用。

UDP和TCP

TCP的可靠性保证

  • 流量控制：滑动窗口，累积确认，超时重传。接收方调整接收窗口，传输方要跟着调整。
    • 零窗口导致的死锁：窗口大小为0，此时接收方调整大小，但是报文丢失，导致二者互相等待。
    • 解决：每次发送都会启动计时器，到时间没有报文响应就发送零窗口探测报文，接收方收到就会发送数据。
  • 拥塞控制：初始 拥塞窗口为 1，ssthresh 为16 ，一开始只能发送 1个数据报文段
    • 慢开始和拥塞避免：每次发送翻倍，数据达到ssthresh 就每次加1，发送超时重传就把窗口设为1， ssthresh 设置为当前窗口大小，重新慢开始。
    • 快重传和快恢复：要求出现故障时，就对按序收到的最大序号进行重复确认。使用慢开始和拥塞避免，如果有3次重复确认，就立即重传未收到的，同时执行快恢复算法： ssthresh 和窗口设置为当前 ssthresh 的 一半，然后继续拥塞避免。
  • 可靠传输：通过滑动窗口实现， 已发送且确认 p1 已发送未确认 p2 未发送 p3 不允许发送
    • 发送窗口不一定与接收窗口大小一致：若拥塞，可以适当减小窗口
    • 必须有累积确认机制
    • 不按序到达：可以丢弃。也可以缓存，发送重复确认，这样发送端就会发送缺失的，等补齐了，可以累积确认，这样就可以节约资源。
    • 全双工通信
  • 超时等待时间
  • ARQ协议：自动重传请求 是OSI模型中数据链路层和传输层的错误纠正协议之⼀。
    • 停止等待ARQ协议：发送一个就必须停止等待。
    • 连续ARQ协议：滑动窗口

  • 解决的问题：解决通过应用进程的交互来实现特定网络应用的问题。
  • 对数据的封装：发出一个 HTTP 请求

动态主机配置DHCP

两台连接到互联网上的电脑相互之间通信，必须有各自的IP地址，由于IP地址资源有限，宽带接入运营商不能做到给每个报装宽带的用户都能分配一个固定的IP地址，所以要采用DHCP方式对上网的用户进行临时的地址分配。也就是你的电脑连上网，DHCP服务器才从地址池里临时分配一个IP地址给你，每次上网分配的IP地址可能会不一样，这跟当时IP地址资源有关。当下线的时候，DHCP服务器可能就会把这个地址分配给之后上线的其他电脑。这样就可以有效节约IP地址，既保证了网络通信，又提高IP地址的使用率。
